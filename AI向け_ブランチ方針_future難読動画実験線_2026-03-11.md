# AI向け ブランチ方針: future 難読動画検証の実験線 2026-03-11

最終更新日: 2026-03-11

変更概要:
- `future` を難読動画検証の実験線として再定義
- `workthree` を UI を含む高速化本線として整理
- 実験結果の返却判定を、ユーザーが感じるテンポ感基準へ固定
- ファイル名は履歴都合で従来名を維持

## 1. この文書の目的

この文書は、AI エージェントが `future` 系の検証作業を進める時に、何を試し、何を本線へ戻す材料にするかを固定するための資料である。

`future` は UI 安定化の本線ではない。
難読動画を実動画で再現し、成功条件を比較し、`workthree` 本線へ返せる形に圧縮するための実験線として扱う。
ただし、実験結果を `workthree` へ返す時の最上位判断基準は、ユーザーが感じるテンポ感を改善するか、少なくとも壊さないかである。`future` での成功は、この基準を満たして初めて価値を持つ。

## 2. このブランチの立場

- このブランチは、難読動画のサムネイル失敗を再現し、成功条件を探る実験線として扱う。
- 通常動画の高速感改善は参考にはなるが、このブランチの主目標ではない。
- `workthree` が UI を含む高速化本線であり、この枝の変更採否を最終判断する。
- 本線へ返す時は、動画名ではなく一般条件、導入位置、回帰観点で返す。
- 本線返却の可否は、`workthree` での UI 一覧、キュー投入、サムネ表示のテンポ感で最終判定する。

## 3. 当面の目標

優先順位は以下で固定する。

1. 難読動画のサムネイルを全作成できるようにする
2. `workthree` のユーザー体感テンポを悪化させない形で、サムネ生成の初動時間と完了時間を短縮する
3. 成功条件を一般化し、`workthree` へ返せる形にする
4. FailureDb やログで比較可能な観測情報を残す

## 4. 採用してよい変更

- 実動画の再現性を上げるテスト
- FailureDb / ExtraJson / ログの観測強化
- retry policy / repair / preflight の仮説検証
- playground や Explicit テストの整備
- 難読動画専用の比較ハーネス
- 本線へ返すための条件整理ドキュメント

## 5. 慎重に扱う変更

- UI 一覧の再描画最適化
- Watcher の走査戦略変更
- 通常動画まで巻き込む並列制御変更
- 本線の既定動作を広く変える大規模リファクタ
- 実験と無関係な設計整理

これらは必要でも、`future` に閉じた観測や比較へ効かないなら後回しにする。

## 6. 原則禁止

- `workthree` の高速化本線ロジックを、比較目的なしで崩す
- ローカル実動画パスを既定値としてコードへ直書きする
- 個別動画名だけで本線条件を決め打ちする
- playground 成功をそのまま本番条件とみなす
- 比較ログを残さずに救済条件だけを増やす

## 7. AI が変更前に確認すること

- この変更は難読動画の再現、比較、一般化に効くか
- この変更を `workthree` に返した時、ユーザーが感じるテンポ感を改善するか、少なくとも壊さないか
- FailureDb / log / result_signature で後追い比較できるか
- `workthree` へ返す時に最小条件へ短文化できるか
- 通常動画や UI テンポへの副作用を必要以上に広げていないか

## 8. `workthree` との関係

- `workthree` は UI を含む高速化本線
- `future` は難読動画検証の実験線
- `future` で価値があるのは、成功条件、観測性、比較テスト、導入位置の仮説である
- `workthree` へ返す時は、成功条件、導入位置、回帰観点、テンポ感への影響の 4 点を必ず揃える

## 9. AI への明示指示

- このブランチでは、個別成功より比較可能性を優先すること
- 通常運用へ混ぜる提案をする時は、必ず `workthree` 本線への副作用を説明すること
- UI 高速化の話題を持ち込む時は、実験の観測性にどう効くかを先に説明すること
- 実験結果は動画名ではなく条件に圧縮して残すこと

## 10. 更新ルール

- このブランチの役割が変わったら、まずこの文書を更新する
- 新しい難読動画群が主戦場になったら、優先順位を追記する
- `workthree` へ返す候補が固まったら、判断理由を別紙へ残す
