# AI向け 本線方針: ユーザー体感テンポ最優先 2026-03-11

最終更新日: 2026-03-20

変更概要:
- 本プロジェクトを UI を含む高速化本線として再定義
- 過去の実験線（future 等）の成果取り込み基準を整理
- 採用基準、慎重項目、禁止事項を新しい役割に合わせて更新
- ファイル名は履歴都合で従来名を維持
- `ThumbnailCreationService` 系を `Factory + Interface + Args` 本流へ整理した前提を追記

## 1. この文書の目的

この文書は、AI エージェントが開発本線で改修する時に、何を最優先にし、何を別枝へ逃がすかを固定するための資料である。

メイン開発ラインは、一覧表示、キュー投入、サムネイル作成まで含めて、ユーザーが感じるテンポ感を最優先に守りながら安定化する本線として扱う。
難読動画の検証は価値があるが、この枝の主目標ではない。

## 2. このブランチの立場

- このブランチは、UI を含む高速化と安定化の**唯一の本線**として扱う。
- 改善対象は UI 一覧、Watcher、Queue、サムネイル生成テンポを含む。
- 過去の検証成果のうち、通常動画のテンポを壊さず一般化できた条件だけを選別して採用する。

## 3. 当面の目標

優先順位は以下で固定する。

1. ユーザーが感じるテンポ感を最優先で守り、改善する
2. UI の詰まりを解消し、一覧表示、キュー投入、サムネイル作成のテンポを上げる
3. 難読動画対応（過去の成果含む）は、通常動画の高速感を壊さない範囲で維持する
4. 観測可能なログと安全なフォールバックを維持する
5. 検索、並び替え、ページ移動、タブ切り替えなどの明示的なユーザー要求は、watch / rescue / thumbnail / poll などの背後処理より優先して実行する

## 3.1 2026-03-20 時点のサムネイル基盤前提

- `ThumbnailCreationService` 実装クラスは non-public とし、公開面は `ThumbnailCreationServiceFactory`、`IThumbnailCreationService`、`ThumbnailCreateArgs`、`ThumbnailBookmarkArgs` に固定した。
- `MainWindow` と `RescueWorker` の生成口は host 別 factory に分離した。
- `create` / `bookmark` の引数検証は service 本体ではなく coordinator 側へ寄せ、hot path の責務を分けた。
- service 本体は concrete coordinator を保持せず、delegate を受け取る facade に寄せた。
- 今後この周辺を触る時は、公開契約を広げず、UI 体感テンポを落とさない境界整理だけを採る。

## 4. 採用してよい変更

- UI スレッドを塞がない局所改善
- 検索、並び替え、ページ移動、タブ切り替えなど、ユーザー要求を背後処理より先に通すための抑止・延期・優先制御
- 一覧更新、絞り込み、並び替えの無駄コピー削減
- Queue の初動改善、先頭数件の見え方改善
- ログ追加、原因切り分け強化
- 設定追加。ただし既定値でテンポを悪化させないもの
- DTO、契約、型整理
- hot path を増やさずに境界を締める内部整理
- 難読動画対応のうち、通常運用へ副作用を広げないもの

## 5. 慎重に扱う変更

- 並列数制御の思想変更
- Watcher の走査戦略変更
- 自動救済や再試行条件の拡大
- 難読動画専用の特殊分岐を通常経路へ混ぜる変更
- 常駐サービス化や IPC 実装の本格導入
- 大規模リファクタ

これらは、ユーザーが感じるテンポ感がどう良くなるか、または少なくとも悪化しないかを示せないなら入れない。

## 6. 原則禁止

- 過去の実験実装を検証なしでそのまま持ち込む
- 個別動画名ベースの条件分岐を本線へ入れる
- 正しさだけを理由に UI テンポ悪化を受け入れる
- 高速化を名目にログや観測性を削る
- 巨大コミットを責務分割せずに取り込む

## 7. AI が変更前に確認すること

- ユーザーが最初に触る一覧、選択、最初の数件表示が軽いか
- 検索などのユーザー要求が、watch / rescue / thumbnail / poll に割り込まれて完了しない構造になっていないか
- 一覧更新で全件コピーや再代入を増やしていないか
- UI スレッドへ重い処理を戻していないか
- 自動救済が通常キューのテンポを壊していないか
- 難読動画対応が通常動画の既定経路を重くしていないか
- ログだけで遅くなった理由を追えるか

## 8. 過去の検証との関係

- 開発本線が唯一の高速化本線であり、すべての成果は本線で管理される
- 過去の実験的アプローチから持ち込めるのは、一般化できた成功条件、観測性改善、安全なテスト追加のみ

## 9. AI への明示指示

- このブランチでは、ユーザーが感じるテンポ感を最優先にし、その上で速さと安定運用を両立させること
- 検索等のユーザー要求は最優先で実行し、背後処理は必要に応じて遅延・抑止・再開すること
- 難読動画対応を提案する時は、通常動画の初動へ与える影響を必ず説明すること
- 他ブランチ由来の変更を提案する時は、何を捨てて何を採るかを明記すること
- 一覧、Watcher、Queue、サムネイル生成のどこを速くする変更かを先に示すこと
- `ThumbnailCreationService` 周辺では `Factory + Interface + Args` を正規入口として守り、互換入口や重複検証を service 本体へ戻さないこと

## 10. 更新ルール

- このブランチの役割が変わったら、まずこの文書を更新する
- 高速感の判断軸が増えたら、禁止事項より先にこの文書へ反映する
