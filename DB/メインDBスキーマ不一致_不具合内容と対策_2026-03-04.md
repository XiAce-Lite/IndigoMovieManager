# メインDBスキーマ不一致 不具合内容と対策（2026-03-04）

## 1. 不具合内容
- 症状:
  - スキーマ不一致の `*.wb` を開こうとしたとき、DBオープン処理の途中でエラーが発生する。
  - 場合によっては旧DBを先に閉じた後に失敗し、ユーザーが作業中だった状態を失う。
  - 失敗したDBパスが `LastDoc` に保存され、次回起動でも同じ失敗を繰り返す。

## 2. 影響
- DB切り替え時の操作継続性が下がる（開けないDBを選ぶと作業中コンテキストを失う）。
- 起動時の自動オープンで失敗ループが発生しやすくなる。
- 監視処理が `watchData` 取得失敗を前提にしておらず、連鎖的な不安定化リスクがある。

## 3. 原因
- `OpenDatafile` が「旧DB停止 -> 新DB起動」の順で進み、新DB検証が後段だった。
- メインDBの必須テーブル/列を開く前に検証していなかった。
- メニュー操作で、DBオープン成功前に `LastDoc` / Recent を更新していた。
- 監視側で `watchData == null` の防御が不足していた。

## 4. 対策
- 開く前のスキーマ検証を追加:
  - `TryValidateMainDatabaseSchema(...)` を追加し、`system/movie/history/watch/bookmark` の必須テーブルと必須列を確認する。
- DBオープン制御を変更:
  - `OpenDatafile` を `bool` 戻り値にし、検証NG時は旧DBを閉じずに中断する。
  - 検証NG時はエラーメッセージを表示し、ユーザー操作を継続可能にする。
- 設定保存タイミングを修正:
  - `OpenDatafile(...) == true` のときだけ `LastDoc` と Recent を更新する。
- 監視処理の防御を追加:
  - `CreateWatcher` / `CheckFolderAsync` で `watchData == null` を検知したら処理を中断し、ログだけ残す。

## 5. 変更ファイル
- `DB/SQLite.cs`
- `MainWindow.xaml.cs`
- `MainWindow.MenuActions.cs`
- `Watcher/MainWindow.Watcher.cs`

## 6. 想定される改善結果
- スキーマ不一致DBを選択しても、現在開いているDB状態を保持できる。
- 失敗したDBが自動オープン対象として保存されにくくなる。
- 監視処理の例外連鎖を抑止し、UI操作継続性を維持できる。

## 7. 確認手順
1. 正常な `*.wb` を開き、一覧表示・監視・検索が従来通り動くことを確認する。
2. 必須列を欠いた `*.wb` を開き、以下を確認する。
   - 「開く処理を中止しました」系のエラー表示が出る。
   - 旧DBの一覧/状態が維持され、操作を継続できる。
   - `LastDoc` が不正DBへ更新されない。
3. アプリ再起動後、起動時に不正DBで失敗ループしないことを確認する。

## 8. 備考
- 本対応は「開く前に検証して安全に中断する」ことを目的とし、`*.wb` スキーマ自体の変更は行っていない。
