# 勇者GPT-5.4の戦いの記録 ～autogen native crash との果てなき闘争～ (2026-03-14)

## 戦いの経緯🔥
「ユーザー体感テンポ最優先」を掲げる `workthree` ブランチにおいて、我々は一つの決断を下した。
**「どんなにじゃじゃ馬でも、最高速を誇る autogen (FFmpeg.AutoGen) を主力のまま飼いならす！」**

しかし、autogen は度重なる牙を剥いた。起動時、タブ切り替え時、突如としてアプリは虚空へと消え去る。
残されたのは `swscale-8.dll` のネイティブクラッシュ (0xc0000005) のログだけだった。

### 第一形態：swscale の同時実行競合を防ぐ盾🛡️
最初に考えたのは、`sws_scale` 周辺の並行実行競合だ。FFmpeg の一部機能は並列に弱いため、`sws_getContext` / `sws_scale` / `sws_freeContext` を `EngineExecutionGate` （セマフォ）で囲い、「1台ずつ通す高速車線（直列化）」の安全弁を入れた。
そしてさらに、UI側からの大量リクエストの雪崩を防ぐため、タブ切り替え時の自動救済を64件までに絞った。

**【勇者コメント】**
「逃げるが勝ちじゃない！主力のまま、暴れる所だけ首輪を付ける！これが俺たちの『最短ルート』だぜ！✨」

### 第二形態：ピクセルフォーマットのすれ違いを直す剣⚔️
しかし、それでもヤツは落ちた。
ログから「`sws_getContext` の初期化時に推測したフォーマットと、実際にデコードしたフレームのフォーマットが食い違っている」可能性を疑い、`AVFrame` の結果から直接コンテキストを再生成する実装（`TryRecreateSwsContextForFrame`）を投下！

**【勇者コメント】**
「思い込みの型ハメは危険だ！現実に返ってきたフレーム情報から毎回剣を打ち直す！これでネイティブ側の不整合は消し飛ぶはずだ！💥」

### 第三形態：まさかの coreclr.dll 崩壊… 真の正体は GC ヒープの破壊！💀
だが、試練は終わらなかった。
直後の短時間起動チェックではなんと、落ちる先が `swscale-8.dll` から `.NET の心臓部である coreclr.dll` へと変化したのだ！
これは「同一プロセス内でネイティブ側がマネージドメモリを派手にぶち壊し、後になってガベージ・コレクション（GC）が巻き込まれて爆死する」という最悪のパターン。

GDI の `BitmapData.Scan0` へ直接書き込んでいたのが原因と睨み、一旦独立したネイティブバッファ (`Marshal.AllocHGlobal`) へ書き出してから Bitmap へコピーする「分離作戦」を実行。
しかし… 今度はタブ切り替えで即落ち！何故だ！？

### 最終奥義：P/Invoke 配列長の罠を見破る鷹の目🦅
ログとコードを凝視した勇者GPT-5.4は、ついに真の犯人を見つけ出した。

```csharp
// 【犯行現場】
byte*[] dstData = [(byte*)nativeBuffer];
int[] dstLinesize = [sourceStride];

// 【FFmpeg 側の仕様】
// dstData は「4つのポインタを持つ配列（各プレーン用）」を求めている！
```

C# 側で「長さ1の配列」を渡していたため、FFmpeg 側は平然と「2番目、3番目のポインタ」を読みに行き、**配列の境界を越えてメモリ（CLRヒープ）を破壊していた**のだ！
これが `coreclr.dll` クラッシュの真の恐怖の正体だった。

```csharp
// 【最終奥義のコード修正】
byte*[] dstData = new byte*[4] { (byte*)nativeBuffer, null, null, null };
int[] dstLinesize = new int[4] { sourceStride, 0, 0, 0 };
```

**【勇者コメント】**
「見えたぜ、お前の急所が！ FFmpeg のヤロー、4プレーン分の配列長がないと機嫌を損ねるなんて聞いてないぞ！😂 長さ4の配列を与えてやる！これでメモリ境界突破の悪夢は完全消滅だ！最高じゃん！！💖」

## 運用メモ: debug-runtime.log の退避と軽量化（2026-03-14）

- 調査継続中に `%LOCALAPPDATA%\IndigoMovieManager_fork_workthree\logs\debug-runtime.log` が `1,046,320,663 bytes` まで肥大化した。
- Debug タブや手動確認で扱いやすくするため、コード変更なしでログ運用だけ整理した。
- 元ログ全体は次へ退避した。
  - `C:\Users\na6ce\AppData\Local\IndigoMovieManager_fork_workthree\logs\backup\20260314_054135\debug-runtime.zip`
- 現行の `debug-runtime.log` は直近約 `16 MB` の末尾だけを残す形に軽量化した。
- 軽量化後も先頭・末尾の読み取りができることを確認済み。

### 判断理由

- 今回必要なのは「直近クラッシュ前後の挙動」を追うログであり、1GB 超の全量を常時ぶら下げる必要は薄い。
- 既存コードは `debug-runtime.log` 固定で参照しているため、ファイル名は変えずに中身だけ軽くする方が安全。
- 全量調査が再度必要になった場合でも、zip 退避済みのため復元や再確認は可能。

### 次回の扱い

- 直近挙動の確認は現行 `debug-runtime.log` を使う。
- 長期比較が必要になったら、まず `logs\backup` 配下の退避物を参照する。
- 再び肥大化した場合も、同じ方針で「全量退避 -> 現行は末尾だけ保持」を優先する。
