# DB切り替え整理: 最近開いたファイル / 新規作成 / UI / DB / サムネ常駐処理 (2026-03-15)

## 1. 目的

- メニューの `最近開いたファイル` と `新規作成` を起点にした MainDB 切り替え処理を、UI / DB / 稼働中サムネ処理の3層で整理する。
- 現状の切り替えが「どこで止めるか」「何を生かしたまま次DBへ渡すか」を明文化する。
- 次の再設計で、入口の統一と責務分離を進めるための共通前提にする。

## 3. メニュー起点の現状

### 3.1 UI入口

- `新規作成`
  - `BtnNew_Click`
  - `SaveFileDialog` で `*.wb` を選ばせ、未存在なら `CreateDatabase(...)` してから `TrySwitchMainDb(...)` を呼ぶ。
- `ファイルを開く`
  - `BtnOpen_Click`
  - `OpenFileDialog` で選んだ `*.wb` を `TrySwitchMainDb(...)` へ渡す。
- `最近開いたファイル`
  - `MenuRecentTree_Click`
  - TreeView の `Tag` に入っているパスを `TrySwitchMainDb(...)` へ渡す。

### 3.2 成功後だけ保存する設定

- `TrySwitchMainDb(...)` の中で、`OpenDatafile(...) == true` の時だけ以下を更新する。
  - Recent ツリー再構築
  - `Properties.Settings.Default.LastDoc`
  - `Properties.Settings.Default.Save()`
- これは、スキーマ不一致や open 失敗時に `LastDoc` / Recent を壊さないための防御になっている。
- ただし `startup-auto-open` だけは、Recent / LastDoc を更新しない。

### 3.3 入口ごとの差

- `新規作成 / ファイルを開く / 最近開いたファイル` は、いまは同じ `TrySwitchMainDb(...)` を通る。
- UI起点の切り替えでは、旧DBと別DBへ移る時だけ `skin / sort` を切り替え前に保存する。
- `startup-auto-open` は例外で、保存系や Recent 更新を通さず「静かに開くだけ」の扱いにしている。

## 4. DB切り替え本体の現状

### 4.1 `OpenDatafile(...)` の責務

- `TrySwitchMainDb(...)` が入口になり、その内側で `OpenDatafile(...)` が次の順で動く。
  1. `TryValidateMainDatabaseSchema(...)`
  2. `ShutdownCurrentDb()`
  3. `BootNewDb(dbFullPath)`
- 失敗時は `false` を返し、旧DB状態を維持する。

### 4.2 開く前の安全弁

- `DB/SQLite.cs` の `TryValidateMainDatabaseSchema(...)` で、`system / movie / history / watch / bookmark` の必須テーブル・列を検証する。
- ここで弾けるため、「旧DBを閉じた後に新DB open 失敗で空振りになる」事故を避けている。

### 4.3 `ShutdownCurrentDb()` が止めるもの

- タブ選択を一旦 `-1` に戻す。
- `FileSystemWatcher` を全停止して `Dispose()` する。
- サムネキューのデバウンス状態と進捗UIキャッシュをクリアする。
- `watchData` をクリアする。
- Everything 関連の通知フラグをクリアする。
- 検索キーワードを空に戻す。

### 4.4 `ShutdownCurrentDb()` が止めていないもの

- アプリ寿命で動く常駐タスク自体は止めない。
  - サムネ consumer
  - Queue persister
  - Everything poll
  - rescued sync
- つまり現状は、`DBごとに常駐タスクを再生成する設計` ではなく、`常駐タスクは生かしたまま参照先DBを切り替える設計` である。

### 4.5 `BootNewDb()` が起動するもの

- `MainVM.DbInfo.DBName / DBFullPath` を更新する。
- `GetSystemTable(...)` で system を読み直す。
- `MainVM.MovieRecs.Clear()` 後に履歴・一覧を再構築する。
- `SwitchTab(...)` で skin を反映する。
- `GetBookmarkTable()` を読み直す。
- `QueueCheckFolderAsync(CheckMode.Auto, "OpenDatafile")` を流す。
- `CreateWatcher()` で新DBの監視フォルダに対する `FileSystemWatcher` を作る。

## 5. 常駐Watcher / サムネ処理の現状

### 5.1 起動単位

- 次の常駐処理は `MainWindow_ContentRendered` で1回だけ起動される。
  - `CheckThumbAsync(...)`
  - `RunThumbnailQueuePersisterSupervisorAsync(...)`
  - `RunEverythingWatchPollLoopAsync(...)`
  - 起動直後の `TryStartInitialThumbnailFailureSync()`
- DB切り替えでは、これらを再起動していない。

### 5.2 Watcher 側

- `CheckFolderAsync(...)` は開始時に以下をスナップショットする。
  - `snapshotDbFullPath`
  - `snapshotThumbFolder`
  - `snapshotDbName`
  - `snapshotTabIndex`
- 実行中に `MainVM.DbInfo.DBFullPath` が変わったら、`abort scan: db switched` で打ち切る。
- ただし `CancellationToken` で即キャンセルする構成ではない。
- そのため、切り替え応答は「ループ内の切替検知地点まで進んだ後」に効く。

### 5.3 通常サムネキュー

- Producer 側は `TryWriteQueueRequest(...)` で、投入時点の `mainDbFullPath` を `QueueRequest` に焼いて Channel へ流す。
- さらに `MainDbSessionStamp` も焼き、DB切り替え成功時に前セッションの要求を stale 扱いにできるようにしている。
- Persister は常駐で動き続け、Channel の要求を QueueDB へ永続化する。
- Persister は「現在のセッション印と一致する要求だけ保存する」ため、切り替え成功後に残っていた旧DB向け `QueueRequest` はここで破棄される。
- さらに切り替え成功後は、旧 `.queue.imm` にすでに保存済みだった `Pending` 行も削除する。
- ただし `Processing` 行は consumer が握ったまま完走する前提で、ここでは消していない。
- `Processing` 行については、2026-03-15 の切り替え体感テストでは許容だったため当面は未対応とする。
- ただし将来、DB切替の違和感が増えた場合に備えて、「切替またぎで既に `Processing` に入っている旧ジョブを切る」余地が残る論点として扱う。
- Consumer は `ResolveCurrentQueueDbService()` で、現在の `MainVM.DbInfo.DBFullPath` に対応する `QueueDbService` を取り直す。
- `QueueDbService` は MainDB ごとに `%LOCALAPPDATA%` 配下の別 `.queue.imm` へ切り替わる。
- つまり通常キューは、`常駐consumerは固定、QueueDBはMainDB追従` の構造である。

### 5.4 サムネ生成そのもの

- `CheckThumbAsync(...)` は QueueDB からジョブを取り、`CreateThumbAsync(...)` で作る。
- `CreateThumbAsync(...)` は `MainVM.DbInfo.DBName` / `ThumbFolder` をその都度参照して実行する。
- 通常レーン timeout や placeholder 化の制御はアプリ寿命タスクの中で継続し、DB切り替え時に再初期化していない。

### 5.5 FailureDb / rescued sync

- `ResolveCurrentThumbnailFailureDbService()` は現在の `MainDB` ごとに `ThumbnailFailureDbService` を差し替える。
- `FailureDb` も `%LOCALAPPDATA%` 配下の別 `.failure.imm` へ分かれる。
- rescued sync は `queue drained` または periodic で走り、現在MainDBの `rescued` 行だけを UI に反映する。

### 5.6 外部 rescue worker

- `TryStartExternalThumbnailRescueWorkerAsync(...)` は、通常キューが空の時だけ起動判定する。
- `ThumbnailRescueWorkerLauncher` は `mainDbFullPath` ごとに `ThumbnailFailureDbService` を差し替える。
- worker 自体は別プロセスだが、起動条件の判定は「今見ている MainDB」の `pending_rescue` に対して行う。

## 6. 現状設計の読み方

現状の MainDB 切り替えは、次の思想で作られている。

- UI入口は複数あるが、実切り替え本体は `OpenDatafile(...)` に寄せる
- そのさらに外側で `TrySwitchMainDb(...)` が入口差分を吸収する
- DBは「開く前に検証」してから旧DBを止める
- 常駐 worker はアプリ寿命で持ち、DB切り替えでは参照先だけ差し替える
- Watcher はスナップショットと切替検知で旧DB混入を防ぐ
- QueueDB / FailureDb は MainDB パス由来の別ファイルへ分離している

これは `DBセッション再構築型` ではなく、`常駐基盤は維持しつつ MainDB セッションだけ切り替える型` である。

## 7. 再考ポイント

### 7.1 起動時自動オープンだけ特例

- UI起点3入口の非対称は `TrySwitchMainDb(...)` でほぼ解消した。
- いま残っている明示的な特例は `startup-auto-open` だけである。
- これは意図した差だが、今後 `session restore` を強める時は再検討が必要。

### 7.2 `LastDoc` の役割は整理されたが、設定責務はまだ UI 近傍にある

- `UpdateSkin()` / `UpdateSort()` は現在DBパスへ直接書くようになった。
- `LastDoc` は「次回開く候補」を覚える用途へ寄った。
- ただし `Properties.Settings.Default.Save()` の責務自体はまだ UI イベント近傍にあり、将来は session 管理側へ寄せる余地がある。

### 7.3 常駐タスクが「DB切り替えをまたぐ」前提

- これはテンポ面では有利だが、責務境界は分かりにくい。
- 例えば `ShutdownCurrentDb()` を見ても、何が止まり、何が生きたままかは直感では追いにくい。
- 次の設計では、少なくとも `App lifetime task` と `DB session state` を名前として分けるべきである。

### 7.4 旧処理の止め方が二系統

- Watcher はスナップショット + 切替検知で止める。
- サムネ consumer / persister / failure sync は service 差し替えで追従する。
- つまり「止めて切る」処理と「生かしたまま向き先だけ変える」処理が混在している。
- 再設計時は、この違いを仕様として明文化しないと、将来の改修で破綻しやすい。

### 7.5 切替中のチャネル残件

- QueueRequest は投入時点の `mainDbFullPath` を持って Channel に積まれる。
- さらに `MainDbSessionStamp` も持つ。
- 切替成功時にセッション印を進めるため、切替直前に積まれてまだ永続化されていない旧要求は persister で破棄される。
- 旧 `.queue.imm` に保存済みだった `Pending` も、切替成功後に削除する。
- つまり現状方針は「切替成功後に残っていた旧DB向け未着手要求は捨てる」である。

## 8. 次の設計で先に決めるべきこと

### 8.1 入口統一の次段

- `TrySwitchMainDb(...)` と `switch source` は導入済み
- `RunMainDbPreSwitch / TryActivateMainDbSession / RunMainDbPostSwitch` で、入口のフェーズ分割も入っている
- 次はこのフェーズ境界を、常駐タスク側の session 境界とどう対応させるかを詰める

### 8.2 フェーズの明文化

- `PreSwitch`
  - 旧DBの表示状態保存
  - 新DBの事前検証
- `DeactivateCurrentSession`
  - Watcher停止
  - UI一時解除
  - 旧DBセッション印の更新
- `ActivateNewSession`
  - MainVM反映
  - system/history/bookmark/watch 読み直し
  - 新DB監視開始
- `PostSwitch`
  - Recent / LastDoc 更新
  - 進捗UI再同期

### 8.3 常駐タスクの立場決め

- 案A: 今のまま app lifetime task を維持し、DB参照先だけ切り替える
- 案B: DB session ごとに consumer / failure sync / watcher 制御を張り直す

`workthree` の体感テンポ優先なら、第一候補は案Aのまま責務だけ整理する形が現実的である。

## 9. 現時点の結論

- 現状の切り替えは、最低限の安全弁は入っている。
  - 開く前のスキーマ検証
  - 旧Watcherの Dispose
  - Watch走査のDBスナップショット
  - QueueDB / FailureDb の MainDB 分離
- 入口統一と旧表示状態保存の非対称は、`TrySwitchMainDb(...)` 導入でひとまず揃った。
- 入口統一に加えて、`PreSwitch / Activate / PostSwitch` のフェーズ分割もコードへ反映済みである。
- 切替直前に積まれた旧DB向け要求は、Channel 残件も QueueDB の `Pending` も捨てる方針まで入った。
- 切替またぎで走行中の旧 `Processing` は、2026-03-15 の体感テストでは一旦許容と判断した。
- ただしこれは「不要」ではなく、「後で必要になれば切れる余地を残す保留論点」である。
- いま残る本丸は、常駐タスクの寿命設計と、この保留論点をどの方式で扱うかである。
- 次にやるべきは全面改修ではなく、`切替フェーズ` と `app lifetime / db session` の境界をさらに明文化する設計整理である。
