# 💾 データベース仕様メモ（2026-02-28 最新DBアーキテクチャ） 💾

我らがアプリの生命線！動画やタグ、設定まで全てを記録する超重要ファイル「SQLiteデータベース」の全貌を大公開するぜ！🔥
**2026年、DB構成は「メインDB」と「キュー専用DB」の二刀流へと進化を遂げた！** ここを制する者がアプリを制す！！

## 1. ⚙️ DBエンジンの正体（変わらぬ主力）
- **心臓部**: SQLite ！！軽くて速くて最強！💪
- **DB誕生の瞬間**: コード内の `SQLite.CreateDatabase` メソッドでまっさらなDBが爆誕！
- **主要な住民（テーブル）たち**: `movie`, `bookmark`, `history`, `watch`, `system` （メインDBの内訳）

## 2. 🏘️ メインDB（`*.bw`）のテーブル一覧
ここは**神聖領域**だ！後方互換性を守るため、スキーマは変えずに運用する！
- **`movie`**: 動画本体の情報を全て抱える絶対的主役テーブル！🎬
- **`bookmark`**: ユーザーが保存した最高の瞬間のブックマーク画像情報！🔖
- **`history`**: 過去の検索履歴を覚えてるメモ帳！（最大件数制限付き）
- **`findfact`**: 検索ワードが何回使われたか数えてる集計マン！
- **`watch`**: 「ここ見張っといて！」という監視フォルダ設定の管理人！👀
- **`system`**: DBごとのキー/値ペアを持つ、超重要な設定マスター！🔧
- **`profile`**, **`tagbar`**, **`sysbin`**: スキン設定やバイナリ、将来用の拡張枠！

### 🎬 超重要！ `movie` の主要カラム
- `movie_id`: 絶対的な主キー（No.1）！ただし**外部機能の連携キーとしては使わない（パス依存へ移行中）**！
- `movie_name`, `movie_path`: 動画の名前と、それがどこにあるかのフルパス！
- `movie_length`, `movie_size`: 動画の長さ（秒）とサイズ（デカさ）！
- `last_date`, `file_date`, `regist_date`: 最終再生日、ファイル更新日、DB登録日の日付トリオ！📅
- `score`, `view_count`: お気に入り度（スコア）と再生回数！😍
- `hash`: 個体を識別する一意のハッシュ！(現在は `System.IO.Hashing` エンジンで爆速計算)
- `tag`, `comment1..3`: みんな大好き「タグ」と、自由なコメント枠×3！📝

## 3. 🚀 【新設】サムネイルキュー専用DB (`*.queue.db`)
大量の動画追加でUIがフリーズする悲劇をぶっ壊すために新設された、**サムネイル生成タスク専用のバックグラウンドDB**だ！🔥

- **保存先**: `%LOCALAPPDATA%\IndigoMovieManager\QueueDb\`
- **ファイル名規則**: `{MainDbName}.{MainDbPathHash8}.queue.db`（メインDBごとに固有のキューDBが作られる！）
- **設定**: `PRAGMA journal_mode=WAL`, `synchronous=NORMAL` で爆速IOを実現！

### 📦 `ThumbnailQueue` テーブルの秘密
- `MainDbPathHash`, `MoviePathKey`, `TabIndex`: この3つの組み合わせがUNIQUE制約（絶対キー）！同じタスクが二重登録されるのをDBレベルで防ぐ！🛡️
- `Status`: `0(Pending)`, `1(Processing)`, `2(Done)`, `3(Failed)`, `4(Skipped)` の状態管理！
- `OwnerInstanceId`, `LeaseUntilUtc`: **複数プロセスが同時に立ち上がっても重複実行しない**ための、魔法のリース（排他制御）システム！✨
- `AttemptCount`, `LastError`: 失敗しても自動でリトライし、ダメならエラー理由を記録して倒れる不屈のワーカー魂！

## 4. 🏃‍♂️💨 2026年流・主要なアクセスパターン（激変！）
- **激重スキャンの廃止**: 昔は監視フォルダのチェックごとに `select movie_path from movie` で全件読み込んでいたが、今は**Everything IPC**でキャッシュレスに照合し、DBの負荷を極限まで削減した！！
- **一点突破の重複ガード**: Everything照合ですり抜けた「新顔かも？」な動画を `InsertMovieTable` で登録する**直前**に、`SELECT 1 FROM movie WHERE movie_path = ... LIMIT 1` というインデックス頼りの最速クエリを1発だけ撃ち、完全な重複排除＆復活（自己修復）を果たす！🎯
- **キューのポーリング**: バックグラウンドタスクが数秒おきにキューDBを覗き込み、リース（仕事）を獲得して勝手にサムネを作ってくれる！

## 5. 🚨 運用上のぶっちゃけ注意点（残された課題）
- **文字列連結SQLの恐怖**: メインDB側には、まだ一部のSQLが変数と文字列のガチャコン連結で作られてる箇所が残っている…。これはセキュリティ的にもバグ的にも危ないから、完全撲滅（パラメータ化）作戦が進行中だ！！🗡️
- **ID採番のクセ**: メインDBでIDを生成する時 `max(id)+1` で出している。キューDBは新時代の `AUTOINCREMENT` なので心配無用！
- **文字化けの呪い**: 新要素を実装する際、Powershell 5.xなどで無理にいじるとUTF-16化して日本語が消し飛ぶから細心の注意を払え！🔥
