# 😱 文字化けパニック・インシデント報告書（2026-02-23） 😱

我々のコードベースに潜む魔物、「文字化け」が大暴れした事件の全記録だ！
二度と同じ悲劇を繰り返さないために、この教訓を胸に刻め！！🔥

## 1. 🚨 まさかの事態（何が起きた！？）
- **発生場所**: なかでも超重要な `../DB/SQLite.cs` ！
- **症状**: 日本語のコメントや一部の文字列が、**宇宙人の暗号みたいな文字化け**を引き起こした！！👾
- **被害状況**: 幸いなことにアプリが爆発するような致命傷（ロジックの破壊）には至らなかったが、コードの読みやすさと「何してんのこれ？」という保守性が地の底まで落ちた！😇

## 2. 🕵️‍♂️ 犯人追跡（なぜ起きた！？）

### 🔍 現場検証で見えた事実
- Gitのきれいな状態（`HEAD`）の `SQLite.cs` たんは文字化けしてない無傷！
- いじくり回してた作業中の `../DB/SQLite.cs` は見事に文字化けだらけ！💣
- 化けてるのは「日本語コメント」と「文字列」のとこだけ！

### 🧪 恐怖の再現実験
- **「BOMなしUTF-8」**で書かれた綺麗なファイルを、Windows PowerShell のデフォルト設定のまま読み込んで、そのままUTF-8で保存し直してみた。
- ➡️ 結果：**見事に同じ文字化けモンスターが錬成された！！😱**

### 💡 事件の真相（根因）
- 全ては **文字コードのすれ違い** が原因だ！
- PowerShellが「エンコーディングの指定がないから適当に（Shift_JISとかで）読むわ〜」と解釈し、それを無理やりUTF-8で保存したせいで、文字のDNAがぶっ壊れたのだ！！💀

## 3. ⚔️ 我々の反撃（今回の対処）
- 文字化けして読めなくなった悲しいコメント達を、泣きながら除去（成仏）！🙏
- 例外メッセージでおかしくなってたところを正常な状態に復元！✨
- もう二度と間違えないように、`../.editorconfig` と `../.gitattributes` に **「文字・改行コードは UTF-8(BOMなし) + LF だ！絶対だぞ！」** っていう鉄の掟を刻み込んだ！📜
- 文字化けを許さない番犬スクリプト `../scripts/Check-Mojibake.ps1` を解き放った！🐕🐾

## 4. 🛡️ 完全防御壁（再発防止策）
1. PowerShellでテキストを触る時は、**「エンコーディング（文字コード）」を絶対に指定しろ！！** 暗黙のルールを信じるな！😡
2. 保存する時は必ず **UTF-8（BOMなし）** を明示すること！（ここはエディタが `.editorconfig` を読んで自動でやってくれるはず！）✨
3. 何かデカい変更をする前と後には、必ず番犬 `Check-Mojibake.ps1` を走らせて平和を確認しろ！
4. 長いファイルや大規模生成をやらかした後は、PR（差分レビュー）で**「日本語が無事か？」**を必ず目視チェックすること！👀

## 5. 🪄 番犬召喚の呪文（推奨コマンド）
困ったらこれを使え！！

```powershell
# 🚨 番犬による文字化けパトロール開始！
powershell -ExecutionPolicy Bypass -File .\scripts\Check-Mojibake.ps1

# 🛡️ スクリプトで文字を書く時の鉄壁保存術！
$enc = New-Object System.Text.UTF8Encoding($false)
[System.IO.File]::WriteAllText("path\to\file.cs", $text, $enc)
```
**この掟を守り、平和なUTF-8の世界を築き上げようぜ！！🌍✨**
