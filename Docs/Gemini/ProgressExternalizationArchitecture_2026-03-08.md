# サムネイル進捗外部化アーキテクチャ（IPC回避）✨

**「サムネイルを大量に作るとUIがもたついてイラつく！！💢」**
この最大のストレスをこの世から消し飛ばすために生まれた、最高にスマートなアーキテクチャ設計だね！🔥

これまでの「UI内に `ThumbnailProgressRuntime`（進捗状態） を持ち、UIとWorkerが密結合している」状態から脱却し、完全な非同期・独立動作を達成しているよ！

## 🚀 要点：ハイテクを捨ててローテクで勝つ！逆転の発想💡

### 1. 【大目的】UIのもたつき・イラつきを完全に消し去る！
大量のサムネイルを作る時、進捗の度にUIと通信していると、通知の嵐でUI側がカクついたり、ひどい時はフリーズしてユーザーをイラつかせていたよね。
これを解決するために、あえてUIとWorker（裏の処理）の「直接の会話（通信）」を完全に断ち切った！これが一番の狙いだよ！😎

### 2. 「高度なIPC（プロセス間通信）」というハイテクをあえて捨てる
UIプロセスとWorkerプロセスで直接やりとり（IPC）しようとすると、名前付きパイプや共有メモリ、gRPCといった「ハイテクなリアルタイム通信技術」を使いたくなるのがエンジニアの性。
でも、ハイテクになればなるほど「UIがフリーズした時にWorkerも巻き添えで止まる」「通信エラー時の同期や再送処理がめちゃくちゃ複雑になる」といった罠が山盛りになるんだ。
そこで、この複雑な通信そのものを**「ただのJSONファイル書き出しと読み込み」という超ローテクな手法**にダウングレードさせた！この逆転の発想こそが、このアーキテクチャの真髄だよ！✨

### 3. 既存のコールバックをそのままローテクに横流し
サムネイル処理の心臓部 `ThumbnailQueueProcessor` は、元から以下の情報を報告してくれるようになっているよ。
- `progressSnapshot` (全体の完了件数や並列数の定期報告)
- `onJobStarted` (動画の処理開始)
- `onJobCompleted` (動画の処理完了)

これをわざわざ複雑な通信に乗せるのではなく、**Worker側が自ら自分専用の .json ファイルへバカ正直に書き出す（Publish）** だけに繋ぎ変えたんだ。リアルタイムなハイテク通信を諦めることで、無駄な新機能を作らずに既存の仕組みをフル活用できているね！💡

### 4. UIは「ただファイルを読むだけ（ポーリング）」
UI側の仕事は、Workerが吐き出した進捗のJSONファイルを定期的に読み取って画面に表示するだけになる！👀
この**「ハイテクなプッシュ通知を捨てて、ローテクな定期ファイル読み込み（ポーリング）にする」**究極の割り切りによって、**「UIがどんなに重くなってもWorkerの爆速処理には一切影響しない」**し、逆に**「Workerがどれだけ全力で回っていてもUIはサクサク動く」**という完璧な切り離しが完成したんだ。どや！🥰

## 🤔 コラム：なぜSQLite（QueueDB）を使わずJSONにしたの？

「進捗渡すだけなら、今使ってるSQLite（QueueDB）にそのまま書けば速くない？」
普通なら絶対そう考えるよね！でも、あえてSQLiteを使わずJSONファイルの単一上書きにしたのには、**超深〜い「割り切り」と「DBを守るための戦略」**があるんだ！🔥

### 1. ⚔️ DBのロック競争（渋滞）を避けるため！
QueueDB（SQLite）は、ワーカー達が「次どれやる？」「これ終わった！」と血眼でアクセスし続ける**絶対的な戦場**なんだ。
そこに「今50%だよ！」「次の動画に入ったよ！」みたいな、**1秒間に何十回も発生する「ぶっちゃけ消えてもいい進捗報告」**までDBに書き込もうとすると、本当に大事な「ジョブの更新」の邪魔になって、DBがロック渋滞を起こしちゃうんだよね😱
だから、進捗報告という「軽いおしゃべり」はDBという本丸から完全に外に追い出したんだ！

### 2. 🗑️ 過去の履歴はいらない、「最新の1枚」だけ見れればいい！
DBは「データを蓄積する」のには最強だけど、「1つの状態をものすごいスピードで上書きし続ける」のは、裏でログ（WAL）が膨らんだりして意外とオーバーヘッドがある。
進捗ってぶっちゃけ**「UIが今画面を描画するための、最新の1枚（スナップショット）」**さえあれば良くて、1秒前の進捗なんて誰も興味ないよね？
ファイルなら `Tempファイルに書く -> 既存のJSONにMoveで上書き(ドーン！)` ってやれば、一瞬で最新状態にすり替わって、ゴミも残らず超絶軽いんだ！🚄💨

### 3. 🔓 片方は「書くだけ」、片方は「読むだけ」だからロックしない！（真のロックフリー）
SQLiteだと「読み手（UI）」と「書き手（Worker）」の両方が同じファイル（DB）を触るから、どうしても「誰かが書いてる時は待つ」っていうロックの奪い合いが発生しちゃう。
でもJSONファイル方式なら、**「Workerはただ上書きするだけ」「UIはただそれを読むだけ」**と役割が完全に一方通行になるんだ！
さらに、コードレベルで**「真のロックフリー（完全な競合回避）」**を実現するための超クールな工夫が2つ入っているよ！✨

1. **【Worker側】Tempからのアトミックなすり替え ＆ 破棄**
   ワーカーはいきなり目的のJSONに書き込まず、まずTempファイルに完全なJSONを書き込んでから、最後に `File.Move` で一瞬（アトミック）で本番のファイルにすり替える。だからUIが「書きかけで壊れたJSON」を読むことは絶対にない！
   さらに、この `Move` を `try-catch` で囲むことで、万が一OSレベルで「今UIがガチガチに掴んでるから上書きダメ！」と弾かれても、**「じゃあ今回の0.1秒前の進捗は捨てて、次をパブリッシュしよ！」とワーカーは一切止まらない**という最強の割り切りをしている！💨

2. **【UI側】「上書き・削除OK」の許可を出してファイルを開く**
   UI側は標準のファイル読み込みメソッドを使わず、あえて `FileShare.ReadWrite | FileShare.Delete` という権限を指定してファイルを開く！
   これは**「今私が読んでる最中だけど、Workerくん、後ろからこのファイルを消して新しいのにすり替えて（Moveして）いいよ！」**という特別許可証なんだ。
   これにより、UIが読んでいる瞬間でもワーカーの妨げにならず、お互いが完全にフルスピードで独立して動ける無敵の状態を作っているんだよ！😎✨

**「絶対守るべき真実（ジョブ結果）」は SQLite に刻み、「今この瞬間のノリ（進捗表示）」は JSONの使い捨て上書き にする！**
この**情報の重要度による、記憶媒体の完全分離**こそが、ワーカーの爆速処理を一切邪魔しないための極意なんだ！天才的な役割分担でしょ？どや！！🥰💖

## まとめ
**「Workerはただファイルに書いて、UIはただファイルを読む」**
サムネ大量作成時の「もたつき」という最大の敵を倒すために、あえてハイテクな通信技術を捨てて「ファイル経由」というローテクに原点回帰し、さらに「DB」という最強の武器さえも「用途が違う」として退けた。
この逆転の発想と徹底した役割分担こそが大正義！バグが入り込む隙を無くしつつ、ノーストレスで爆速のサムネイル処理を実現する最高のアプローチだよ！💖
