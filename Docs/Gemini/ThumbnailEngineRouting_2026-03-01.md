# 🚀 爆速サムネ職人・エンジンの歴史と切り替え条件 (2026-03-01版) 🚀

やっほー！このドキュメントは、IndigoMovieManagerの要である「サムネイル生成エンジン」たちがどうやって進化してきたのか、そして**「今、裏側でどうやって最適なエンジンが選ばれているのか（ルーティング条件）」**を大公開するぜ！🔥

何千という動画ファイルから最速でサムネをぶっこ抜くため、我々は長きにわたり血みどろの試行錯誤を繰り返してきた…！その歴史と現在の生存戦略（アルゴリズム）を目に焼き付けろ！👀

---

## 1. ⚔️ 歴代エンジンの戦いの歴史（時系列順）

サムネ生成エンジンの実体は、`IThumbnailGenerationEngine` インターフェースを継承したクラスたちだ。
以下が我々が辿ってきた「進化の系譜」である！

### 🦖 第一世代: OpenCV (`OpenCvThumbnailGenerationEngine.cs`)
- **[ID: `opencv`]** 太古の昔から存在する安定のエンジン。
- `OpenCvSharp4` を用いて動画フレームを取得していたが、とにかく**遅い**！シークが使い物にならない！🐌
- 現在はフォールバックのさらに奥底、最終防衛ラインとしてひっそり生き残っている。

### 🛡️ 第二世代: FFmpeg CLI (`FfmpegOnePassThumbnailGenerationEngine.cs`)
- **[ID: `ffmpeg1pass`]** 外部プロセス(`ffmpeg.exe`)を呼び出して1フレームずつ画像を吐き出させる正統派強硬手段。
- **圧倒的な安定感**を誇るが、プロセス起動（`Process.Start`）のコストと、複数フレーム取得時に毎回エンコードが走るため、やはり「速度」の面で壁にぶつかった。
- しかし、巨大ファイルや超絶長時間ファイルに対しては**絶対にクラッシュしない**という絶対的な信頼があるため、今でも「高負荷時の救世主」として活躍中！

### 🏎️ 第三世代: FFMediaToolkit (`FfMediaToolkitThumbnailGenerationEngine.cs`)
- **[ID: `ffmediatoolkit`]** C#から ffmpeg の dll (.so) を直接叩き歩く「禁断の果実」。
- これにより、サムネイルの生成速度が **数倍〜十倍** に跳ね上がるという劇的なブレイクスルーを果たした！🔥
- が…！「絵文字パス」や「シフトJISマッピング不能文字」が含まれるパスを渡すと**内部で文字化けして死ぬ（ファイルが開けない）**というffmpeg特有の呪いをモロに受け、我々は絶望の淵に立たされた…。😭

### 👑 第四世代: FFmpeg.AutoGen (`FfmpegAutoGenThumbnailGenerationEngine.cs`)
- **[ID: `autogen`]** そして辿り着いた、現在の「絶対王者」。
- `FFMediaToolkit` より更に低レイヤ、C言語の関数とポインタ（`unsafe`コード）をC#から直接ぶん回す真の黒魔術！💀✨
- **最大のメリット**: WindowsのUTF-16文字列から直接UTF-8バイト配列を生成し、`avformat_open_input` のポインタに直接叩き込むことで、**「絵文字パス」の呪いを完全に打ち砕いた！！**🎉
- 現在はこいつが主力（デフォルト）としてバリバリ働いているぞ！

## 2. 🚦 今のルーティング条件（どう選んでるの？）

これらのエンジン達の頂点に立ち、状況に応じて最適なエンジンを即座に選ぶ**超優秀な司令塔**が `ThumbnailEngineRouter.cs` だ！🧠✨

`ThumbnailEngineRouter.ResolveForThumbnail` メソッドを通った時の「採用フロー（時系列順）」は以下の通り！上から順に判定され、条件に合えばそのエンジンが即座に選ばれるぞ！

| 順序 (優先度) | 判定条件 | 選ばれるエンジン (ID) | 理由・背景 |
|:---:|:---|:---|:---|
| **第1位** | `IMM_THUMB_ENGINE` 環境変数が「auto以外」 | **強制指定された奴** | デバッグ用！ユーザーの言うことは絶対！👑 |
| **第2位** | プレイヤーからの「手動作成 (`IsManual`)」 | `ffmediatoolkit` | 画面からの指示。位置指定(TimeSpan)の正確性が高い旧王者をあえて採用！ |
| **第3位** | **絵文字パス / 日本語化けパス含有** | `autogen` | 旧王者が即死する魔のパス。黒魔術（ポインタ直渡し）でしか倒せない！🔥 |
| **第4位** | パネル数10以上 ＆ **超巨大ファイル(4GB以上等)** | `ffmpeg1pass` | メモリリークやシーク遅延でC#側が死なないよう、安定の別プロセスへ退避！🛡️ |
| **第5位** | パネル数10以上 ＆ **高ビットレート(20Mbps以上等)** | `autogen` | 負荷が高いシークは、実は低レイヤな(`autogen`)の方がメモリ効率が良く安定するため！ |
| **第6位** | パネル数10以上 ＆ **超長編(120分以上)** | `ffmpeg1pass` | これも同様に、高負荷のシーク破綻を避けるための安全装置！ |
| **デフォルト** | 上記すべてに当てはまらない通常時 | `autogen` | **現代の絶対的主力！** 最速かつ絵文字に強い君に決めた！🌟 |
| **防衛線** | もし上の奴がコケたら？（Fallback） | 呼び出し順(※) | `autogen` -> `ffmedia` -> `ffmpeg1pass` -> `opencv` と、順番に代役を立てる不屈の設計！ |

*(※) `ThumbnailCreationService.cs` でインスタンスを配列にしてルーターに渡す順序で決まるぞ！*

## 3. 🎯 環境変数での微調整（限界突破の呪文）

デバッガーやユーザーが無理やり条件をいじれるように、便利な環境変数も用意してある！🔥

- **`IMM_THUMB_ENGINE`** : `ffmediatoolkit`, `autogen`, `ffmpeg1pass`, `opencv` を指定すると、判定を全部すっ飛ばしてそいつを使う。
- **`IMM_THUMB_LARGE_FILE_GB`** : 「これ以上はデカいファイル認定！」の基準（デフォルト: `4.0` GB）。
- **`IMM_THUMB_HIGH_AVG_BITRATE_MBPS`** : 「これ以上は高ビットレート認定！」の基準（デフォルト: `20.0` Mbps）。
