# 🚑 動画インデックス破損修復機能の全貌（リカバリーレーンの救済策）2026-03-06

やっほー！今回は、**「古のFLVや壊れかけたMP4がやってきても、もう誰も泣かない！インデックス修復機能」** についてまとめたよ！✨

これのおかげで、一部の動画ファイルが持っている「インデックス部分の破損（シーク情報がぶっ壊れててサムネが作れない現象）」を自動で検知して、システムが勝手に直してくれる最強の仕組みが手に入ったんだ！😎🔥

---

## 🛡️ なぜこの機能が必要だったの？
Indigoのサムネイル生成エンジン（FFmpeg.AutoGenやFFMediaToolkit）は爆速だけど、**インデックスが壊れているファイルに出会うとお手上げ（エラー）**になってしまう弱点があったんだよね。

特に古いFLVファイルとかだと、「再生はできるのにシーク（サムネの切り出し位置へのジャンプ）ができない！」ってことが頻発してたの。
これまでは「エラー画像の生成（CODEC NG）」でお茶を濁すしかなかったけど、これからは違うぜ！💪

## 🚀 修復機能の最強アプローチ（Engine分離アーキテクチャ）
この修復ロジックは、アプリの根幹である `IndigoMovieManager.Thumbnail.Engine` （一番奥深くにいるサムネ職人の部屋）に完全に隔離されているよ！
UIやQueue（処理の待ち行列）を汚さずに、奥底で勝手に修復してくれる美しい設計なんだ！✨

### 動きのフロー
1. **通常レーンはそのまま爆速で！**
   - 最初のサムネイル作成（`AttemptCount == 0`）は、**修復処理なんて一切考えずに全力で走る！** （余計な判定でスピードを落とさないため！）
2. **エラーになったら「リカバリーレーン」へ！**
   - もしエラーになって再試行キュー（`AttemptCount > 0`）に入ってきたら、ここで初めて**「もしかしてインデックス壊れてる？」**と疑い始める（Probe）。
3. **破損検知（Probe）**
   - ファイルを開いたときのエラーコードや、ストリーム情報の欠落（既知の破損シグネチャ）を検知！
4. **自動修復（Repair）開始！**
   - 壊れていると判定されたら、一旦「インデックスに頼らないメモリ読み込み」を試したり、裏でこっそり **一時ファイルへ再MUX（再エンコードなしの超高速詰め替え）** を行って構造を修復！
5. **サムネ生成の完遂！**
   - 直った一時ファイルを使ってサムネイルを生成し、終わったら証拠隠滅（一時ファイルの削除）をしてフィニッシュ！🎉

## 🔒 鉄壁の安全対策（絶対上書きさせないマン）
「一時ファイルとか作って、間違えて元の動画消えたりしない？」
**—— 全く心配ご無用！！！**

修復API（`RepairVideoIndexAsync`）には **「元の動画ファイルと修復先の出力ファイルが同じパスだったら、即座に例外をぶん投げて処理を完全ブロックする」** という絶対防御が入ってる！🛡️
しかも出力は `.mp4` や `.mkv` のような安全コンテナを強制！元動画を傷つけることは1000%ない安心設計だよ！🥰
