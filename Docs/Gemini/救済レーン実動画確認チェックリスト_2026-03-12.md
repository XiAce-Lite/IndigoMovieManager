## 6. チェック項目

### 6.1 通常動画の基準確認

1. 通常動画を一覧表示からサムネイル未作成状態にする。
2. 自動生成を待つ。
3. 次を確認する。
   - `thumbnail-rescue-request` が出ない
   - `thumbnail-rescue-worker` が出ない
   - `thumbnail-create-process.csv` に `success` が1件出る
   - 進捗表示は通常Thread / BigMovie の範囲で収まり、救済専用の揺れが出ない

### 6.2 terminal failure から外部救済 worker へ進むか

1. 重動画を通常キューで自動生成させる。
2. 次を確認する。
   - `FailureDb` に `pending_rescue` が1行追加される
   - 通常キュー drain 後に `thumbnail-rescue-worker` の `rescue worker launched` が1回だけ出る
   - 続いて `rescue worker stdout: rescue leased` が出る
   - `engine attempt start: ... engine=ffmpeg1pass` から始まる
   - 長時間動画では `lease heartbeat` が出続ける
   - 規定時間を超えた試行は `engine attempt exception` または `rescue gave up` へ進む
   - worker 終了後に `rescue worker exited` が出る
   - 同じ動画が通常キューへ戻って粘り続けない

### 6.3 明示救済要求の確認

1. 対象動画に対して `手動等間隔サムネイル作成` または右クリック救済を実行する。
2. 次を確認する。
   - `thumbnail-rescue-request` の `enqueue accepted` が1回だけ出る
   - `FailureDb` に `pending_rescue` が1行追加される
   - duplicate 実行時は `enqueue skipped duplicated` が出る
   - harness / `pwsh` 直叩き時は `launch_requested=True` になること
   - `requiresIdle=true` の経路では active ジョブ中に `worker launch deferred` が出る
   - 空き後に外部 worker が起動する

### 6.4 repair 発火確認

1. repair 候補動画を救済 worker まで流す。
2. 次を確認する。
   - `engine attempt failed` 後に `repair probe start` が出る
   - `repair probe end: detected=True` の時だけ `repair start` が出る
   - `repair end: success=True` の時だけ repair 後の `engine attempt start: ... repair=True` が出る
   - repair 不要動画では `repair probe end: detected=False` または `repair` 系ログなしで `rescue gave up` へ進む

### 6.5 `rescued -> reflected` 同期確認

1. worker が rescue 成功した動画を用意する。
2. 次を確認する。
   - `FailureDb` の親行が `rescued` になる
   - queue drain / 次回起動 / `periodic-ui-tick` のいずれかで `thumbnail-sync` の `rescued sync completed` が出る
   - 反映後に `FailureDb` の親行が `reflected` になる
   - worker success ログの `output='...'` が session 配下ではなく通常の thumb root を指す
   - 出力jpgを消してから同期させた場合は `requeued` が増え、`pending_rescue` へ戻る

### 6.6 placeholder 起点確認

1. 詳細または各タブの表示先を `error*.jpg` にした動画を用意する。
2. 画面更新を発生させる。
3. 次を確認する。
   - `thumbnail-rescue-request` で stale `ERROR` マーカー削除が出る
   - `detail-error-placeholder` または `tab-error-placeholder` 理由の救済要求が1回だけ入る
   - 通常キュー active 中は `worker launch deferred`、空き後に worker 起動となる

### 6.7 `サムネ失敗` タブ確認

1. `.#ERROR.jpg` がある動画を2本以上用意する。
2. `サムネ失敗` タブを開き、`選択救済` と `一括救済` を試す。
3. 次を確認する。
   - `.#ERROR.jpg` 実在個体だけが一覧に出る
   - `選択救済` は選択行だけ、`一括救済` は一覧全件を対象にする
   - 本exe 側では `thumbnail-rescue-request` が出る
   - 実処理は外部 worker 側だけで進み、通常キューへ戻らない

### 6.8 Watch 起点の周辺確認

1. 監視対象フォルダで欠損サムネイル救済または差分検出を発生させる。
2. 次を確認する。
   - `watch-check` の監視ログが出る
   - 同一走査中に通知が過剰に増え続けない
   - `ui-tempo` の初動が目立って悪化しない
   - 救済確認中でも通常一覧や最初の数件表示が重くならない

## 7. 異常として扱うもの

- 通常動画でも毎回 `thumbnail-rescue-request` や `thumbnail-rescue-worker` が出る
- 同じ動画 / 同じタブで `pending_rescue` が重複投入される
- worker が毎回起動するのに `rescue leased` まで進まない
- `engine attempt start` の先頭が `ffmpeg1pass` にならない
- `rescued` になったのに `thumbnail-sync` で `reflected` へ進まない
- `rescue succeeded` の `output` が `RescueWorkerSessions` 配下になっている
- 出力jpg欠損時に `pending_rescue` へ戻らない
- 救済中に通常一覧や初動表示が目に見えて重くなる
