# 現状把握 workthree 失敗動画検証と本線反映方針 2026-03-11

最終更新日: 2026-03-19

## 1. 目的

この文書は、`workthree` 本線で高速化を進める時に、難読動画対応をどの位置づけで扱うかを固定するための資料である。

難読動画検証の主戦場は `future` だが、`workthree` ではその結果を通常動画の高速感を壊さない形で採用する。

## 2. 現在の前提

- `workthree` は UI を含む高速化と安定化の本線である。
- `future` は難読動画検証の実験線である。
- 難読動画対応は価値があるが、通常動画の初動や一覧テンポを悪化させるなら本線へ入れない。
- 取り込み判断は、動画名ではなく一般条件で行う。

## 3. 本線で優先すること

1. UI の詰まりを解消する
2. 一覧表示、キュー投入、サムネイル作成のテンポを保つ
3. 難読動画対応は通常系へ副作用を広げない形で取り込む
4. ログとフォールバックで遅延理由を追えるようにする

## 4. `future` から持ち込む時の見方

`future` の成果は、次の 3 点が揃ったものだけ本線候補にする。

- 成功条件: どの一般条件で効いたか
- 導入位置: `MainWindow` / `Watcher` / `ThumbnailCreationService` / `ThumbnailQueueProcessor` のどこへ入れるか
- 回帰観点: UI テンポ、通常動画、フォールバック順、ログ比較のどこを見るか

## 5. 今の扱い

- UI 系改善は `workthree` で進める
- 難読動画の仮説検証は `future` で進める
- 本線へ返す時は、通常動画のテンポ悪化がないことを優先確認する
- 個別動画名依存の条件分岐は本線へ入れない

## 6. 次に見るファイル

- `MainWindow.xaml.cs`
- `Watcher/MainWindow.Watcher.cs`

- `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailQueueProcessor.cs`

## 7. 2026-03-19 実動画確認で分かったこと

`autogen` の `1x1` 単発で `thumb-sec` を固定して比較したところ、難しさは「超巨大動画かどうか」だけでは説明できなかった。

### 7.1 `sango72GB.mkv`

- `C:\Users\na6ce\Desktop\_読み取り困難ログ用\超巨大動画\72\sango72GB.mkv`
- `thumb-sec=10,30,60,120,300` は短時間成功
- `thumb-sec=600` は `180秒` timeout
- `thumb-sec=2159` は既存ログでも失敗・長時間化が集中

つまり、この動画はサイズよりも「深い秒位置」で急に重くなる性質が強い。

### 7.2 `60fps_scale_2x_prob-3(4)_scale_1x_prob-3(1).mov_NVEcqp16.mp4`

- `C:\Users\na6ce\Desktop\_読み取り困難ログ用\超巨大動画\234\60fps_scale_2x_prob-3(4)_scale_1x_prob-3(1).mov_NVEcqp16.mp4`
- `thumb-sec=10,30,60,120,300,600,1200,1800,2400` をすべて短時間成功

つまり、同じ超巨大動画帯でも秒位置依存は一様ではない。

### 7.3 本線反映の判断

- `opencv` 先頭化のような既定順変更を全体へ広げる根拠にはならない
- 一般化するなら `thumb-sec` の短秒寄せ、または深い秒位置だけ rescue へ逃がす条件化が先
- 本線で守るべきなのは「通常動画の初動」と「短秒位置での素早い成功」である
