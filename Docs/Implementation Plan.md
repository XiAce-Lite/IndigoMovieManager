# Implementation Plan

## 1. 目的
- プロジェクト理解コストを下げる
- 変更時の破壊リスクを下げる
- 検索・サムネイル周りの機能発展を進めやすくする

## 2. 現状課題（優先順）
1. `../MainWindow.xaml.cs` に責務が集中している
2. SQL文字列連結が残っており保守性と安全性に課題がある
3. UI処理と業務処理が密結合でテストしにくい
4. TODO（検索拡張、タグバー等）が未回収

## 3. フェーズ計画

### Phase 0: 可視化（完了）
- 目的: 全体理解と共通言語の作成
- 成果物:
  - `ProjectOverview.md`
  - `DevelopmentSetup.md`
  - `Architecture.md`
  - `DatabaseSpec.md`

### Phase 1: 安全化（短期）
- 目的: 既存挙動を保ったまま壊れにくくする
- 作業:
  1. [進行中] `../DB/SQLite.cs` のSQLを順次パラメータ化
  2. [未着手] DB更新系で例外時のログ出力を整理
  3. [未着手] サムネイルキュー処理のキャンセル/再起動経路を整理
  4. [完了] クリティカル経路（起動、取込、サムネ、検索）の手動テスト観点を文書化
- 完了条件:
  - 主要更新SQLに文字列連結が残らない
  - 主要導線の回帰チェック手順が1枚にまとまっている

## 3.1 進捗ログ（2026-02-23）
- 完了:
  - Phase 0 の成果物を作成済み
  - `../DB/SQLite.cs` の更新系SQLで主要なパラメータ化を実施済み
  - `RegressionChecklist.md` を作成し、主要導線の回帰手順を定義済み
  - `SearchSpec.md` を作成し、検索仕様（入力ルール、履歴更新、既知制約）を明文化済み

  1. `../MainWindow.xaml.cs` 側のSQL組み立てを安全化（必要箇所を分離）
  2. DB例外ログの統一（出力先と粒度を決定）
  3. サムネイルキューの再起動経路を明示（再入・多重起動防止）

### Phase 2: 分割（中期）
- 目的: 機能追加しやすい構造へ移行
- 作業:
  1. `MainWindow` から `SearchService` を分離
  2. `ThumbnailService`（`CreateThumbAsync`周辺）を分離
  3. `WatcherService`（監視処理）を分離
  4. ViewModelへの移管可能な処理を段階移行
- 完了条件:
  - 主要ロジックの入口がサービスクラスに移る
  - `../MainWindow.xaml.cs` の責務がUIイベント中心になる

### Phase 3: 機能拡張（中長期）
- 目的: ユーザー価値の高い未実装項目を解消
- 作業候補:
  1. 検索のNOT/OR対応
  2. 検索履歴保存条件の調整
  3. タグバー代替機能の具体化
  4. 重複判定ロジックの強化
- 完了条件:
  - 優先機能2件以上を本番運用レベルで提供

## 5. リスクと対策
- リスク: WhiteBrowser互換挙動の崩れ
  - 対策: 互換系処理は先に振る舞い確認ケースを作る
- リスク: サムネイル生成の性能低下
  - 対策: 既存処理時間を計測し、分離後と比較する
- リスク: 大規模リファクタで差分が増える
  - 対策: サービス単位で小分けに実施する

## 7. 直近スプリント実行計画（2026-02-23 週）
### 7.1 スプリントゴール
- `../MainWindow.xaml.cs` と `../DB/SQLite.cs` の「安全化対象」を明確化し、Phase 1 を完了に近づける
- DB更新時ログの共通ルールを決め、次スプリントで実装できる状態にする
- サムネイルキューの再起動経路を文書化し、多重起動リスクを下げる

### 7.2 実施タスク（優先順）
1. SQL組み立ての安全化
- 対象: `../MainWindow.xaml.cs` の `GetData(..., $"SELECT * FROM movie order by ...")`
- 方針: 並び替え語はホワイトリスト関数で確定し、SQL本文は固定文に寄せる
- 補足: `watch` テーブル参照の `$"SELECT * FROM watch ..."` は固定文字列へ置換する
2. DB更新ログの統一仕様作成
- 対象: `../DB/SQLite.cs` の `InsertHistoryTable` / `DeleteHistoryTable` / 更新系メソッド
- 方針: 例外時に「処理名・対象ID・SQL種別・メッセージ」を同一形式で出力する
- 成果物: `DevelopmentSetup.md` にデバッグ時の確認方法を追記
3. サムネイルキュー再起動経路の明文化
- 対象: `../MainWindow.xaml.cs` の `CheckFolderAsync` / `CheckThumbAsync` / 起動直後処理
- 方針: キュー投入、キャンセル、再開の順序をシーケンスで整理する
- 成果物: `Architecture.md` に追記するか `ThumbnailFlow.md` を新規作成する

### 7.3 完了判定（このスプリント）
- `Implementation Plan.md` の Phase 1 項目 2, 3 が「設計完了」状態になる
- SQL安全化対象の着手順がファイル・メソッド単位で決まっている
- 回帰チェックで使う重点ケースを `RegressionChecklist.md` から3件選び、実施記録が残る

### 7.4 次スプリントへの引き継ぎ
- `SearchSpec.md` を基準に、検索条件仕様（AND/OR/NOT）の実装差分を解消する
- `ThumbnailService` 分離の前提として、`CreateThumbAsync` の入出力境界を定義する
