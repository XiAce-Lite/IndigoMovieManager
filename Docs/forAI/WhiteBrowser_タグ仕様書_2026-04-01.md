## 4. 結論の先出し
- WhiteBrowser 最終版のタグ情報は、`*.wb` 内の `movie.tag` / `bookmark.tag` 列と `tagbar` テーブルを中心に構成される。
- タグ限定検索として `!tag` が存在する。
- スキン/タグレット側には `addTag` / `removeTag` / `flipTag` / `onModifyTags` がある。
- `tagbar.contents` に `wb://...` を入れると、タグレットとして JavaScript を実行できる。
- 一方で、「階層タグ」「エイリアス」「CSV 一括入出力」は補助資料にはあるが、今回の実DB確認だけでは完全確定ではない。

## 5. タグの保存構造

### 5.1 `movie` / `bookmark` の `tag` 列
- 確認済み
- `movie` テーブルと `bookmark` テーブルの両方に `tag text not null default ''` がある。
- 同じく `title` / `kana` / `roma` / `comment1-3` と並んで保持される。
- 逆アセンブリ仕様書では、`title/kana/roma/tag/comment1-3` は「ユーザー編集/検索用の別フィールド」と整理されている。

### 5.2 `tagbar` テーブル
- 確認済み
- 構造は次の通り。
  - `item_id`
  - `parent_id`
  - `order_id`
  - `group_id`
  - `title`
  - `contents`
- `parent_id` と `order_id` を持つため、タグバーは単純な1列ボタン群ではなく、階層や並び順を意識した構造と見るのが自然。
- `contents` は単なるラベル文字列ではなく、検索文字列または `wb://...` を含む実行内容の格納先である。

### 5.3 `profile` との関係
- 確認済み
- 逆アセンブリ仕様書では、`taglet / key script / startup.js` から呼ぶ `getProfile/writeProfile` はグローバルスコープへ移したとされている。
- つまりタグ関連の拡張動作は、`tag` 列や `tagbar` だけでなく `profile` とも連携し得る。

## 6. タグ検索仕様

### 6.1 一般検索との関係
- 確認済み
- `tag` は単独専用テーブルではなく、動画ごとの検索対象フィールドの1つとして使われる。
- 逆アセンブリ仕様書の整理上、`title/kana/roma/tag/comment1-3` は検索用別フィールドである。

### 6.2 タグ限定検索
- 確認済み
- `!tag` というタグ限定検索がある。
- これは通常のファイル名検索とは別に、「タグだけを見る」専用記法として扱うのが正しい。

### 6.3 かな・ローマ字との関係
- 確認済み
- タグは `title/kana/roma/comment1-3` と同列の検索系フィールド群の1つである。
- ただし「タグ名自体へかな変換やローマ字変換を適用するか」は今回の資料だけでは未確認。

## 7. タグ操作 API

### 7.1 スキン/JavaScript 側の操作
- 確認済み
- `wblib.js` 系 API として次が存在する。
  - `addTag`
  - `removeTag`
  - `flipTag`
  - `modifyField`
- このうちタグ専用操作は `addTag` / `removeTag` / `flipTag` である。
- `flipTag` は「対象タグがあれば外し、無ければ付ける」トグル動作として読むのが自然で、補助資料でもその説明と一致している。

### 7.2 コールバック
- 確認済み
- タグ変更時コールバックとして `onModifyTags` が存在する。
- したがって WB のタグ変更は、単なる DB 更新だけでなく、スキン側の UI 再描画契機も持っていたと考えるべきである。

### 7.3 情報取得結果のタグ
- 確認済み
- `getInfo/getInfos` で返るオブジェクトには `tags` プロパティがある。
- つまりスキン側はタグ一覧を読んで独自UIを描ける。
- ただし、その `tags` の内部表現が配列か文字列か、どの区切りで渡るかは今回の資料だけでは未確定。

## 8. タグバー仕様

### 8.1 タグバーの役割
- 確認済み
- `tagbar` は DB テーブルとして独立しており、タグそのものの保存先というより「タグ操作や検索導線の定義置き場」として機能していたと見るのが妥当である。

### 8.2 `contents` の意味
- 確認済み
- `contents` に `wb://...` を入れるとタグレットとして JavaScript 実行される。
- 逆に `wb://` でない値は、通常の検索語や検索条件として使われる可能性が高い。

### 8.3 マウス操作
- 補助資料ベース
- 要約資料ではタグバーに次の操作があるとされる。
  - 左クリック: 登録ワードで検索
  - 中クリック: 選択中ファイルへタグ付与
  - 右クリック: メニュー表示や永続フィルタ
  - ドラッグ: ボタン移動やグループ化
- ただしこのクリック割り当ては、今回の実DB確認だけでは断定していない。

## 9. タグレット仕様

### 9.1 実行形式
- 確認済み
- タグレットは `wb://...` から始まる JavaScript 実行片である。
- 例として次が仕様書に残っている。
  - `wb://wb.sort('名前かな(昇順)');wb.find('★');`

### 9.2 配置場所
- 確認済み
- `taglet` フォルダが存在し、タグレット JS の置き場になっている。
- 同梱サンプルとして `taglet/checkArgv.js`、`taglet/registMovie.js` がある。

### 9.3 起動引数との連携
- 確認済み
- `-t path` でタグレットを実行できる。
- フルパスでない場合は `taglet` フォルダから探す。
- 起動済みの WhiteBrowser があれば、コマンドライン引数経由で検索やタグレット実行を転送できる。

### 9.4 `startup.js`
- 確認済み
- 0.7.5.0 系では `taglet/startup.js` が起動時またはスキン切替時の自動実行対象と見るのが自然である。
- タグ操作の初期化や補助フックは、ここから行われていた可能性が高い。

## 11. 互換実装時に重要な注意

### 11.1 `WB.db` / `Tags` テーブル前提は採らない
- 確認済み
- 補助資料の一部には `WB.db` や `Tags` テーブル前提の説明がある。
- しかし実DB確認と逆アセンブリ仕様書では、実際の管理DBは `*.wb` であり、タグは `movie.tag` / `bookmark.tag` と `tagbar` によって扱われている。
- したがって互換実装では、こちらを正本として扱うべきである。

### 11.2 タグ文字列の内部区切りは未断定
- 未確認
- `tag` 列が複数タグをどう区切るかは、今回の資料だけでは断定しない。
- 実装側で改行区切りや配列展開を採る場合も、それを「WB の確認済み仕様」とは書かない方が安全である。

### 11.3 `tagbar` はタグ辞書ではなく操作定義として扱う
- 確認済み
- `tagbar.contents` に `wb://...` を入れて実行できる時点で、`tagbar` は「タグ名一覧テーブル」より広い。
- 互換実装でも、検索ボタン、タグ付与ボタン、マクロ実行ボタンを同居できる前提で設計した方が WB に近い。

## 12. 最終整理
- 確認済みの WB タグ中核は次の4点である。
  - `movie/bookmark` の `tag` 列
  - `tagbar` テーブル
  - `!tag` によるタグ限定検索
  - `addTag/removeTag/flipTag/onModifyTags` とタグレット
- 逆に、階層タグ、エイリアス、CSV 一括編集は魅力的だが、今回の根拠だけでは補助資料扱いに留めるのが誠実である。
- IndigoMovieManager 側で互換を取るなら、まずはこの「確認済みの4点」を先に固めるのが筋である。
