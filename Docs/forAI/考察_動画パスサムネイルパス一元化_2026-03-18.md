# 考察: 動画パス・サムネイルパス取得の実情と一元化方針 (2026-03-18)

作成日: 2026-03-18

## 1. 調査の目的

コードベース全体で動画パスとサムネイルパスがどのように取得・構築・正規化・比較されているかを洗い出し、
重複や散在を整理した上で、一元化の可否と方針を考察する。

## 2. 現状マップ

### 2.1 動画パスのライフサイクル

動画パスは以下の 5 段階を経て使われている。

```
[DB保存] → [読取/モデル化] → [正規化] → [比較/検索] → [存在確認/表示]
```

各段階の実態:

| 段階 | 主管 | 実装場所 |
|------|------|---------|
| DB 保存 | `movie_path text not null` | DB/SQLite.cs (INSERT/UPDATE) |
| 読取 | `DataRow → Movie_Path` | Views/Main/MainWindow.xaml.cs (CreateMovieRecordFromDataRow) |
| 正規化 (UI用) | `MovieCore.NormalizeMoviePath()` | Models/MovieCore.cs L100-125 |
| 正規化 (比較用) | `QueueDbPathResolver.NormalizePathForCompare()` | src/.../QueueDbPathResolver.cs L57-93 |
| 存在確認 | `Path.Exists(movie_path)` | 7箇所以上に散在 |

### 2.2 サムネイルパスのライフサイクル

サムネイルパスは以下の 4 段階で構築される。

```
[ThumbRoot決定] → [Layout別フォルダ] → [ファイル名生成] → [フルパス結合]
```

| 段階 | 主管 | 実装場所 |
|------|------|---------|
| ThumbRoot 決定 | `ThumbRootResolver.ResolveRuntimeThumbRoot()` | src/.../ThumbRootResolver.cs |
| Layout フォルダ | `ThumbnailLayoutProfile.BuildOutPath()` | src/.../ThumbnailLayoutProfile.cs |
| ファイル名生成 | `ThumbnailPathResolver.BuildThumbnailFileName()` | Thumbnail/ThumbnailPathResolver.cs |
| フルパス結合 | `ThumbnailPathResolver.BuildThumbnailPath()` | Thumbnail/ThumbnailPathResolver.cs |

サムネイルパスの **構築ロジック自体は一元化されている**。
問題は構築後の **設定（代入）箇所が分散** していること。

### 2.3 ThumbPath* プロパティへの代入箇所（6箇所）

| 箇所 | ファイル | 用途 |
|------|---------|------|
| A | Views/Main/MainWindow.xaml.cs (CreateMovieRecordFromDataRow) | 通常 DB 読取 |
| B | Views/Main/MainWindow.Startup.cs (CreateStartupMovieRecordFromSource) | 起動 Feed |
| C | Thumbnail/MainWindow.ThumbnailFailureSync.cs | サムネ生成完了時 |
| D | Views/Main/MainWindow.xaml.cs (ファイルリネーム処理) | リネーム時の差し替え |
| E | BottomTabs/Extension/MainWindow.BottomTab.Extension.DetailThumbnail.cs | 詳細パネル |
| F | BottomTabs/Bookmark/MainWindow.BottomTab.Bookmark.cs | ブックマーク |

A と B は **ほぼ同じロジックが 2 箇所に書かれている**。

## 3. 発見された問題

### 3.1 正規化ロジックの 3 系統問題 🔴

動画パスの正規化が 3 つの異なる実装で行われている。

**系統 1: MovieCore.NormalizeMoviePath()** — UI 用
- Trim → クォート除去 → `Path.GetFullPath()` （論理パスは温存）
- 小文字化しない、パス区切り統一しない

**系統 2: QueueDbPathResolver.NormalizePathForCompare()** — 比較用
- Trim → クォート除去 → Windows 拡張パス接頭辞除去 → `Path.GetFullPath()` → `Replace('/', '\\')` → `ToLowerInvariant()`
- 系統 1 の上位互換 + 比較用追加処理

**系統 3: MainWindow.DbSwitch.cs 等の散在コード** — 個別対応
- `path.Trim().Trim('"')` を手書き
- `normalized.Replace('/', '\\')` を手書き
- `Path.GetFullPath(directoryPath.Trim().Trim('"'))` を手書き

散在箇所（確認済み 10 箇所以上）:
- Views/Main/MainWindow.DbSwitch.cs — 4 箇所
- Thumbnail/ThumbnailEnvConfig.cs — 1 箇所
- Watcher/EverythingLiteProvider.cs — 3 箇所
- Watcher/EverythingFolderSyncService.cs — 1 箇所
- Thumbnail/MainWindow.ThumbnailRescueLane.cs — 1 箇所

### 3.2 MovieRecord 生成の 2 経路重複 🟡

`CreateMovieRecordFromDataRow()` と `CreateStartupMovieRecordFromSource()` が
ほぼ同じ ThumbPath 設定ロジックを持っている。
起動段階ロード（Phase 2）で分離された結果だが、変更時に片方だけ直す事故の温床になる。

### 3.3 存在確認の散在 🟡

動画ファイルの `Path.Exists` / `File.Exists` が 7 箇所以上に散在。
サムネイルファイルの存在確認も 8 箇所以上に散在。
それぞれ呼び出し前の前処理（パス正規化、null チェック）が個々にバラバラ。

### 3.4 NoLockImageConverter の metadata 取得問題 🟡

前回レビュー（ページUpDown引っかかり計画）でも指摘した通り、
`NoLockImageConverter` はキャッシュヒット時でも毎回 `Path.Exists` + `FileInfo` を叩く。
パス取得の一元化とは別問題だが、存在確認コストの議論と接続する。

## 4. 何が一元化されていて、何がされていないか

### ✅ 既に一元化されているもの

| 対象 | 一元管理者 | 状態 |
|------|-----------|------|
| サムネイルファイル名生成 | `ThumbnailPathResolver.BuildThumbnailFileName()` | ◎ 完全に集約 |
| サムネイル出力フォルダ決定 | `ThumbRootResolver` + `ThumbnailLayoutProfile` | ◎ 完全に集約 |
| エラーマーカー判定 | `ThumbnailPathResolver.IsErrorMarker()` | ◎ 完全に集約 |
| CRC32 ハッシュ計算 | `MovieHashCalculator.GetHashCrc32()` | ◎ 完全に集約 |
| QueueDB 用パスキー | `QueueDbPathResolver.CreateMoviePathKey()` | ◎ 完全に集約 |
| tabIndex → Layout 決定 | `ThumbnailLayoutProfileResolver.Resolve()` | ◎ 完全に集約 |

### ❌ 一元化されていないもの

| 対象 | 散在状況 | 影響度 |
|------|---------|--------|
| 動画パスの正規化 | 3 系統 + 散在 10 箇所以上 | 🔴 高 |
| ThumbPath* の代入 | 6 箇所（うち 2 箇所は重複ロジック） | 🟡 中 |
| 存在確認の前処理 | 15 箇所以上で個別実装 | 🟡 中 |
| DB パスの正規化 | 散在 4 箇所 (MainWindow.DbSwitch.cs) | 🟡 中 |

## 5. 一元化の考察

### 5.1 一元化すべきもの

#### A. パス正規化の統合 — 最優先

**現状 3 系統を 2 系統に整理する。**

- **表示用正規化**: `MovieCore.NormalizeMoviePath()` を維持。これは UI 表示とモデル保持用。
- **比較用正規化**: `QueueDbPathResolver.NormalizePathForCompare()` を維持。これはキー生成と重複判定用。
- **散在コードの撲滅**: `Trim().Trim('"')` や `Replace('/', '\\')` を手書きしている箇所は、上記 2 つのどちらかへ寄せる。

2 系統が必要な理由:
- 表示用は大文字小文字を保持する必要がある（ユーザーが見るパス）
- 比較用は小文字統一・拡張接頭辞除去が必要（一致判定の精度）

**撲滅対象の散在コード:**

| ファイル | 行 | 推奨置き換え先 |
|---------|-----|---------------|
| MainWindow.DbSwitch.cs L324 | `Path.GetFullPath(selectedPath.Trim().Trim('"'))` | `MovieCore.NormalizeMoviePath()` |
| MainWindow.DbSwitch.cs L363-373 | `dbFullPath.Trim().Trim('"')` + `Replace` | 専用の `NormalizeDbPath()` 新設 |
| MainWindow.DbSwitch.cs L385-391 | `directoryPath.Trim().Trim('"')` + `Replace` | 同上 |
| ThumbnailEnvConfig.cs L234 | `configuredPath.Trim().Trim('"')` | `MovieCore.NormalizeMoviePath()` |
| EverythingLiteProvider.cs L424,437 | `Path.GetFullPath(candidatePath)` | `MovieCore.NormalizeMoviePath()` |
| EverythingFolderSyncService.cs L491-492 | `Path.GetFullPath(path).TrimEnd(...)` | ディレクトリ用正規化を新設 |

なお、`NormalizeMoviePath` と `NormalizePathForCompare` の共通部分（Trim、クォート除去、GetFullPath）を
private な共通ヘルパーとして切り出し、差異部分（小文字化、拡張接頭辞除去）だけを上位で足す形にすると、
2 系統間の挙動差がコードレベルで明確になる。

#### B. DB パス正規化の新設 — 中優先

`MainWindow.DbSwitch.cs` に散在するDB パス正規化を `DbPathNormalizer` 等として切り出す。
動画パスとは性質が異なる（ディレクトリ末尾の `\` 処理等が入る）ため、MovieCore とは別にする。

#### C. ThumbPath 代入の共通化 — 低優先

`CreateMovieRecordFromDataRow` と `CreateStartupMovieRecordFromSource` で重複している
ThumbPath 設定ロジックを、共通メソッド（例: `ApplyThumbnailPaths(MovieRecords, thumbPaths[])`）へ切り出す。

### 5.2 一元化すべきでないもの

#### 存在確認の一括集約

`Path.Exists` を全箇所で共通ラッパーに通すのは過剰。理由:

- 存在確認の目的が箇所ごとに異なる（表示判定、生成前チェック、走査フィルタ、エラー検知）
- ラッパーの抽象化レベルが低すぎて、呼び出し側にとって意味のある抽象にならない
- `NoLockImageConverter` の metadata キャッシュは別の問題（パス一元化ではなく I/O 最適化の領域）

ただし、**同じパスに対して短時間に複数回 `Path.Exists` を叩いている箇所**（例: NoLockImageConverter）は
存在確認結果の短時間キャッシュとして対処する方が適切。これはパス一元化ではなく I/O 最適化として扱う。

#### サムネイルパス構築ロジックの再集約

`ThumbnailPathResolver` と `ThumbRootResolver` と `ThumbnailLayoutProfile` は既に一元化されている。
これ以上の集約は過剰。

## 6. 推奨アクション

### 優先度順

| 順序 | アクション | 影響範囲 | リスク | 効果 |
|------|-----------|---------|--------|------|
| 1 | 散在している `Trim().Trim('"')` + `GetFullPath` を `NormalizeMoviePath` へ寄せる | 10箇所程度 | 低 | 正規化バグの予防 |
| 2 | `NormalizeMoviePath` と `NormalizePathForCompare` の共通部を private ヘルパーへ切り出す | 2ファイル | 低 | 2系統の差異の明確化 |
| 3 | DB パス正規化を切り出す (`DbSwitch.cs` の 4 箇所) | 1ファイル | 低 | DB 切替バグの予防 |
| 4 | ThumbPath 代入の共通メソッド化 (A, B 経路) | 2ファイル | 中 | 起動ロード改修との接続 |

### やらないこと

- 存在確認の全箇所共通ラッパー化
- `MovieCore` と `QueueDbPathResolver` の統合（目的が異なる 2 系統は維持する）
- サムネイルパス構築ロジックの再構成（既に集約済み）

## 7. 起動段階ロード計画との接続

起動段階ロード計画（Implementation Plan 大DB起動段階ロード化）の Phase 2 で
`MovieListItem` を新設する際、ThumbPath の持ち方が論点になる。

現在の `MovieRecords` は 6 つの ThumbPath プロパティを持つが、
`MovieListItem` では **表示中タブの 1 パスだけ** 持てばよい可能性がある。
この設計判断は、本考察のアクション 4（ThumbPath 代入の共通メソッド化）の後に行うのが自然。

## 8. まとめ

**サムネイルパスの構築ロジックは既に十分一元化されている。** `ThumbnailPathResolver` + `ThumbRootResolver` + `ThumbnailLayoutProfile` の 3 者で責務が明確に分かれている。

**最大の問題は動画パスの正規化ロジックの散在。** 3 系統 + 散在 10 箇所以上が存在し、`Trim().Trim('"')` のような断片が個別に手書きされている。ここを `MovieCore.NormalizeMoviePath()` へ寄せるのが最も費用対効果の高い一元化である。
