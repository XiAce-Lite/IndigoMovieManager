# 本家で本当に手を動かさない運用を目指す場合の実現条件と代替案（2026-04-05）

## 1. 結論

- 現在の構成では、**「本家で完全に手を一切触らない」は 2026-04-05 時点では成立しません**。
- これは `Private repo` の成果物取得が `Private` 側認証前提になっているためです。
- ただし、**初回セットを 1 回だけ入れれば、その後は運用がほぼテンプレ化**できます。

## 2. 現行を本家に残しつつ最小化する現実解

最低限必要になるものは以下の 2 つです。

- `INDIGO_ENGINE_REPO_TOKEN`（シークレット）
- `private_engine_release_tag` または `private_engine_run_id`（または repo variable 連動）

この状態なら、`Public/Private` の分離運用はそのまま使えます。

## 3. 本当にゼロタッチを作る場合（設計変更）

### 3.1 方式A: public ミラーから読む

- Private の worker ZIP / package asset を公開可能なミラーへ定期転送。
- Public workflow は private リポジトリ API を叩かずに、公開ミラーから取りに行く。
- メリット: 本家の Secrets 依存が消せる。
- デメリット: Private ソース資産を公開ミラーに出す設計変更が必要。

### 3.2 方式B: GitHub App / OIDC 認証化

- 本家の `INDIGO_ENGINE_REPO_TOKEN` を外し、GitHub App 権限で Private 取得。
- 本家側は App インストール/権限許可が必要。
- メリット: シークレット管理を `token` ベースから App に寄せやすい。
- デメリット: App 導入と権限設計の追加が必要。

## 4. 今回の提案

「すぐ効かせる」の順なら、まずは現行を `初期セット2点` で走らせ、
実績を固めてから方式A or B を選ぶのが安全です。

- v1 の既存運用（install/upgrade/uninstall）には影響を与えず、追加改修を段階的に進められる。
- まずは `事実ベース` の説明を upstream に共有して、誤解（「本家は何もしないで OK」）を潰す。
