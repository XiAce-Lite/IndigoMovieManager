# UNC 動作確認手順 2026-04-01

## 目的

- UNC 上の MainDB が開けることを確認する
- UNC 監視フォルダの登録が保持されることを確認する
- UNC 監視フォルダへ動画追加時に watch 反映されることを確認する
- QueueDB / FailureDB が巻き添えで異常化しないことを確認する

## 事前準備

- UNC 共有を 1 つ用意する
  - 例: `\\server\share`
- 動作確認用フォルダを共有内に作る
  - `\\server\share\imm-test\db`
  - `\\server\share\imm-test\watch`
- 小さめの動画ファイルを 1 本用意する
  - 例: `sample.mp4`
- IndigoMovieManager を起動する

## 確認 1: UNC 上の MainDB を開く

1. `\\server\share\imm-test\db\unc-test.wb` を新規作成または既存選択で開く
2. 例外ダイアログが出ないことを確認する
3. 一覧画面が通常どおり表示されることを確認する

期待結果:

- DB が開ける
- 直後にクラッシュしない
- Debug タブが使える構成なら、現在 DB パスが UNC のまま見える

## 確認 2: UNC 監視フォルダを登録する

1. 監視フォルダ編集を開く
2. `\\server\share\imm-test\watch` を登録する
3. 画面を閉じる
4. 再度 監視フォルダ編集を開く

期待結果:

- パスが `\\server\share\imm-test\watch` のまま残る
- 先頭の `\\` が欠けない
- 重複登録扱いにならず 1 件として保持される

## 確認 3: watch 反映

1. 監視対象に登録した `\\server\share\imm-test\watch` へ `sample.mp4` をコピーする
2. 数秒待つ
3. 一覧または対象タブに動画が現れるか確認する

期待結果:

- 新規動画が登録される
- watch 例外で落ちない
- 監視登録済みパスが消えない

## 確認 4: QueueDB / FailureDB の健全性

1. Debug タブが使える場合は以下を確認する
   - 現在 QueueDB パス
   - 現在 FailureDB パス
   - レコード数の取得成否
2. サムネイル作成を 1 件流す
3. 必要なら失敗動画も 1 件流して FailureDB の更新を確認する

期待結果:

- QueueDB が開けない系の例外が出ない
- FailureDB が開けない系の例外が出ない
- レコード数取得が失敗しない

## ログ確認先

- アプリログ: `%LOCALAPPDATA%\{AppName}\logs\`
  - 例: `%LOCALAPPDATA%\IndigoMovieManager\logs\`
  - 例: `%LOCALAPPDATA%\IndigoMovieManager_fork\logs\`
- 確認したい観点:
  - DB オープン失敗
  - schema validation retry / retry exhausted
  - watch 登録失敗
  - watcher create canceled
  - watcher started
  - queue db 関連エラー
  - failure db 関連エラー

## NG の見分け方

- DB を開く時点で例外が出る
- 監視フォルダ再表示時に UNC の先頭 `\\` が崩れる
- watch 登録後に動画を置いても反映されない
- QueueDB / FailureDB の取得でエラーが出る

## 補足

- 今回の改修で、接続文字列生成と UNC 文字列保持のガードは入っている
- MainDB を UNC で開く時は、共有の瞬断を吸収するため短い retry と timeout を入れている
- ただし NAS/UNC 実運用では共有権限、SMB 切断、ネットワーク遅延の影響は別途残る
- そのため最終判断は実共有で 1 回流して決める
