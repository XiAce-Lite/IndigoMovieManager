# 公開ミラー方式A 初回セットアップと実行手順 2026-04-06

## 1. 結論

- 方式Aの最短運用では、`Public repo` 側は workflow 既定の公開ミラー設定をそのまま使え、必要なら `PUBLIC_ENGINE_MIRROR_REPO` で上書きすればよい。
- `INDIGO_ENGINE_REPO_TOKEN` は不要にできる。
- ただし、**公開ミラー運用では `private_engine_release_tag` 指定を基本にする**。
- `private_engine_run_id` ルートは private repo の Actions artifact を見るため、通常運用では使わず復旧時だけ使う。

今回の mirror repo:

- `https://github.com/T-Hamada0101/IndigoMovieEngine-Mirror.git`
- repo 名として workflow / variable に入れる値は `T-Hamada0101/IndigoMovieEngine-Mirror`

## 2. Public repo 側で必要ならやること

GitHub の `Settings > Secrets and variables > Actions > Variables` で、必要なら次を追加する。

- Name: `PUBLIC_ENGINE_MIRROR_REPO`
- Value: `T-Hamada0101/IndigoMovieEngine-Mirror`

未設定でも `github-release-package.yml` は既定で `T-Hamada0101/IndigoMovieEngine-Mirror` を見る。
別の mirror repo に切り替えたい時だけ設定する。

## 3. 公開ミラーに置くもの

### 3.1 worker asset

次の命名規則に合う zip を release asset として置く。

- `IndigoMovieManager.Thumbnail.RescueWorker-<tag>-win-x64-compat-*.zip`

例:

- `IndigoMovieManager.Thumbnail.RescueWorker-v1.0.3.5-win-x64-compat-1.zip`

### 3.2 private engine packages

次の 3 本を release asset として置く。

- `IndigoMovieEngine.Thumbnail.Contracts.<version>.nupkg`
- `IndigoMovieEngine.Thumbnail.Engine.<version>.nupkg`
- `IndigoMovieEngine.Thumbnail.FailureDb.<version>.nupkg`

推奨:

- `engine-manifest.json`

任意:

- `private-engine-packages-manifest.json`

## 4. 実行手順

### 4.1 preview 実行

`Public repo` の `Actions > github-release-package > Run workflow` から手動実行する。

入力は次を基本にする。

- `private_engine_release_tag`: mirror release に置いた tag
- `private_engine_run_id`: 空

例:

- `private_engine_release_tag = v1.0.3.5`

### 4.2 release 実行

- Public repo で `v*` tag を push する
- workflow は同じ tag 名で mirror release asset を探す

## 5. 成功判定

次の step が通れば、方式Aの最短導線は成立している。

- `Try sync private engine publish source`
- `Try sync private engine packages`
- `Create release package`
- `Create WiX installer`

artifact は次が出る。

- `github-release-package`
- `github-release-installer`
- `github-release-body-preview`

## 6. よくある詰まりどころ

### 6.1 mirror repo に Actions が無い

問題ない。方式Aの最短運用では、mirror repo は **release asset 置き場** であればよい。

### 6.2 `Run workflow` で `private_engine_run_id` を入れたくなる

公開ミラーだけで始めるなら避ける。

- mirror 側に Actions artifact が無い
- token も使わない

この条件では `run_id` ルートは成立しない。

### 6.3 token を消したのに落ちる

次を確認する。

- `PUBLIC_ENGINE_MIRROR_REPO` を上書きしているなら値が正しいか
- `private_engine_release_tag` を入れているか
- mirror release に `engine-manifest.json`、worker zip、nupkg 3 点が揃っているか

## 7. この方式の意味

- 本家に `INDIGO_ENGINE_REPO_TOKEN` を要求しない
- 本家に private repo 参照権限を要求しない
- 本家は `Public repo` の workflow 実行だけ見ればよい

つまり、**本家に「秘密情報の準備」をさせない最短案**として成立する。
