# 完全自動化案 公開ミラーmanifest正本化 2026-04-07

## 1. 先に結論

完全自動化の完成形では、次の2つを人手設定から外す。

- `PUBLIC_ENGINE_MIRROR_REPO`
- `PRIVATE_ENGINE_PUBLISH_RUN_ID`

最終的な正本は、公開ミラー release 上に置く `engine-manifest.json` とする。

つまり、

- 公開ミラーの repo 名は workflow 既定値にする
- どの private publish run を使うかは manifest に書く
- Public 側 workflow は manifest を読んで engine package / worker artifact を取得する

この形にすると、GitHub Variables を人が手で更新しなくてよくなる。

## 2. いま手で持っている値

2026-04-07 時点で、運用上人が意識している値は次の2つである。

- `PUBLIC_ENGINE_MIRROR_REPO`
  - 現在値: `T-Hamada0101/IndigoMovieEngine-Mirror`
- `PRIVATE_ENGINE_PUBLISH_RUN_ID`
  - 現在の成功確認済み値: `23997659256`

ただし、この2つは性質が違う。

- `PUBLIC_ENGINE_MIRROR_REPO` はほぼ固定値
- `PRIVATE_ENGINE_PUBLISH_RUN_ID` は private 側の成功 run に応じて変わる可変値

したがって、完全自動化では次のように扱う。

- 固定値は workflow 既定値へ寄せる
- 可変値は manifest 化して workflow が自動解決する

## 3. 最終完成形

### 3.1 `PUBLIC_ENGINE_MIRROR_REPO`

手動 variable ではなく、Public 側 workflow の既定値にする。

既定値:

- `T-Hamada0101/IndigoMovieEngine-Mirror`

必要なら override 用 variable を残してもよいが、通常運用では未設定でも動くようにする。

### 3.2 `PRIVATE_ENGINE_PUBLISH_RUN_ID`

人が設定する variable から外す。

代わりに、公開ミラー release に置く `engine-manifest.json` へ次を記録する。

- releaseTag
- privateRunId
- packages asset 名
- worker asset 名
- channel
- 生成時刻

Public 側 workflow はこの manifest を読んで、必要な asset を自動取得する。

## 4. リポジトリの役割

### 4.1 エンジン本体（Private）

- engine source の正本
- `Contracts / Engine / FailureDb / RescueWorker` の実装を保持
- package / worker artifact の生成元

### 4.2 エンジンミラー（Public）

- source mirror ではなく asset-only mirror
- engine source は置かない
- 公開可能な release asset と `engine-manifest.json` の配置先

### 4.3 本体アプリ（Public）

- UI / 周辺ロジック / 配布導線を保持
- engine source は持たない
- build / release 時に manifest を見て package / artifact を取得する

## 5. manifest の例

```json
{
  "schemaVersion": 1,
  "channel": "stable",
  "releaseTag": "v1.0.3.5-r1",
  "privateRunId": 23997659256,
  "packagesAssetName": "private-engine-packages.zip",
  "workerAssetName": "IndigoMovieManager.Thumbnail.RescueWorker-v1.0.3.5-win-x64-compat-abcdef.zip",
  "generatedAt": "2026-04-07T00:00:00Z"
}
```

preview 用は tag 固定ではなく、`preview-latest` のような channel release を使う。

## 6. stable release の自動フロー

1. Private engine repo で publish workflow が成功する
2. packages と worker artifact を生成する
3. 同 workflow が公開ミラー release を更新する
4. asset と `engine-manifest.json` を公開ミラーへ upload する
5. Public repo の `github-release-package.yml` は mirror release の manifest を読む
6. 必要な engine 部品を取得する
7. Zip / Installer を生成して GitHub Release へ公開する

## 7. preview の自動フロー

1. Private engine repo の preview publish が成功する
2. 公開ミラーの `preview-latest` release を更新する
3. `engine-manifest.json` を最新 preview run に差し替える
4. Public repo の `workflow_dispatch` は preview manifest を読んで build する

## 8. Public 側 workflow の取得順

最終形では次の順に解決する。

1. tag push なら同名 tag の公開ミラー release を見る
2. manual preview なら `preview-latest` release を見る
3. release の `engine-manifest.json` を読む
4. manifest に書かれた asset 名で package / worker を取得する
5. mirror 不整合時だけ private API fallback を使う
6. `PRIVATE_ENGINE_PUBLISH_RUN_ID` fallback は互換期間だけ残し、安定後に削除する

## 9. 必要な認証

完全自動化でも、最初のセットアップだけは必要である。

必要な権限の考え方:

- Private repo:
  - `Actions read`
  - `Contents read`
- Mirror repo:
  - `Contents write`
- Public repo:
  - `Actions write`
  - 必要なら `Contents write`

実装手段は次のどちらか。

- GitHub App
- PAT

長期運用では GitHub App を推奨する。

## 10. この構成で消える手作業

- `PUBLIC_ENGINE_MIRROR_REPO` の手入力
- `PRIVATE_ENGINE_PUBLISH_RUN_ID` の手更新
- 成功した private run を人が探す作業
- public workflow 実行前に「どの run を使うか」を人が判断する作業

## 11. 段階的移行案

### Phase 1 最小変更

- `github-release-package.yml` に `PUBLIC_ENGINE_MIRROR_REPO` の既定値を入れる
- variable 未設定でも動くようにする
- `PRIVATE_ENGINE_PUBLISH_RUN_ID` は当面残す

### Phase 2 manifest 導入

- Private 側 publish workflow で `engine-manifest.json` を生成する
- 公開ミラー release に manifest を upload する
- Public 側 sync script を manifest 読み取り対応にする

### Phase 3 完全自動化

- Public 側の既定参照を manifest 正本へ切り替える
- `PRIVATE_ENGINE_PUBLISH_RUN_ID` を互換 fallback に格下げする
- 安定後に docs と workflow から `PRIVATE_ENGINE_PUBLISH_RUN_ID` 必須運用を削除する

## 12. 実装対象のイメージ

主に次のどこかへ入る。

- `.github/workflows/github-release-package.yml`

## 13. いまの判断

完全自動化の本質は、
`run_id を設定値として人が持つのをやめて、公開ミラーの manifest を正本にする`
ことにある。

このため、今後の実装は

1. `PUBLIC_ENGINE_MIRROR_REPO` の既定値化
2. manifest 生成
3. manifest 読み取り
4. `PRIVATE_ENGINE_PUBLISH_RUN_ID` の縮退

の順で進めるのが自然である。
