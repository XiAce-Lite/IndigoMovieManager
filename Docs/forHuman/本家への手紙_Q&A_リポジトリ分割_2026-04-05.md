# 本家への手紙 Q&A：リポジトリ分割について 2026-04-05

## Q1: エンジン/Worker用の非公開リポジトリは、誰がどこに作ればいいですか？

**A: 現在は、Private 側の repository / workflow / token は整備済みです。**
**ただし、本家がそのまま運用を乗せる場合、完全無操作にはなりません。**

まずはこの 2 点を本家側で揃える想定です（最小セット）。

- `INDIGO_ENGINE_REPO_TOKEN`（Private repository 用）
- `private_engine_release_tag` または `private_engine_run_id`（または repo variable での代替）

具体的にはこういう構成になっています。

| 項目 | 内容 |
|:---|:---|
| **リポジトリ名** | `IndigoMovieEngine`（仮称） |
| **オーナー** | `T-Hamada0101`（私のアカウント） |
| **公開設定** | **Private（非公開）** |
| **役割** | サムネイル生成エンジン・RescueWorkerのビルド専用 |

**本家側（オリジナルのリポジトリ）で行うのは、初期 1 回の接続整備が基本です。**
本家には「エンジンのソースコードは見えない」という前提を維持しつつ、
「完成済みのバイナリが ZIP / Setup として配布される」形をそのまま維持します。

### 本家が最小限必要なこと

- `INDIGO_ENGINE_REPO_TOKEN` を Secrets に入れる
- `Private Engine` の参照先が変わる場合は、`Private repo` 参照先の変数 or スクリプト実行引数を本家向けに合わせる
- `github-release-package.yml` の `workflow_dispatch` が使える状態（`workflow_dispatch` 実行権限）

### それでも「本家は何も触らず完全ゼロ」にするための前提

現行設計では、Private 側取得に認証が必要なため完全ゼロは未対応です。
将来的に 0 操作を目指すなら、少なくとも次のどちらかが必要です。

1. `Private` 取得を公開可能なミラー（パッケージ/アセット供給）に寄せる
2. GitHub App（OIDC）を介したクロス repo アクセスを実装して Secret を消す

現時点の `v1` と `運用ルート` は、上記 1/2 実装前提にはなっていません。

2026-04-05 時点の事実としては、「本家での初期接続セットだけは必要」です。

## Q2: ユーザーへの配布（リリース）はどうなりますか？

**A: GitHub Release に「ポータブル版ZIP」と「インストーラー（.exe）」の2種類を自動で並べる体制を構築済みです。**

ユーザーは GitHub Releases ページから好みの方を選んでダウンロードするだけです。

| 配布物 | 内容 | 対象ユーザー |
|:---|:---|:---|
| `IndigoMovieManager-v*.zip` | 解凍してすぐ使えるポータブル版 | 既存の上級ユーザー・WhiteBrowserから移行中のユーザー |
| `IndigoMovieManager-Setup-v*.exe` | ダブルクリックでインストール完了 | 新規ユーザー・初心者 |

### 配布フロー（自動化済み）

```
git tag v1.0.0 → git push
    ↓
GitHub Actions（github-release-package.yml）が自動起動
    ↓
Private リポジトリから エンジンバイナリを「お取り寄せ」
    ↓
ZIP + WiX インストーラー（.exe）を同時生成
    ↓
GitHub Release へ 2ファイルを自動添付して公開完了
```

**エンジンのバイナリ（非公開コード由来）は ZIP / インストーラーの中に同梱されますが、ソースコードは一切 Public リポジトリへ入りません。**

## Q3: インストーラーは何を使っていますか？アンインストールは？

**A: WiX Toolset（v5〜v6）の Burn バンドル形式で生成しています。Windowsの「設定 > アプリ」から正式にアンインストール可能です。**

| 機能 | 対応状況 |
|:---|:---|
| インストール | ✅ ダブルクリックで完了 |
| アップグレード（上書き更新） | ✅ 同じインストーラーを実行するだけ |
| アンインストール | ✅ 「設定 > アプリ」から削除可能 |
| .NET ランタイム自動取得 | ✅ Burn が未インストールを検知して自動ダウンロード |
| 自動バックグラウンドアップデート | 🔄 v2 で実装予定 |

### WiX ライセンスについて

WiX Toolset は **MS-RL（Microsoft Reciprocal License）** のオープンソースツールです。
- 商用製品のインストーラー生成に使用しても、自社コードの公開義務は発生しません。
- 無料で商用利用可能です。（公式サポートが必要な場合は有償プランあり）

## Q4: アップデートの仕組みは？

**A: v1 は「インストーラーを上書き実行するだけでアップデート完了」という手動方式です。v2 でアプリ内通知 + 自動ダウンロードを実装予定です。**

### v1（現在）
- GitHub Releases ページから最新のインストーラーを手動ダウンロードして実行するだけで上書き更新完了
- WiX が既存インストールを検知し、設定・DBを保持したままクリーンに更新

### v2（予定）
- アプリ起動時に GitHub Releases API を確認
- 新バージョンがある場合は「アップデートしますか？」と通知
- OKを押すと最新インストーラーをバックグラウンドでダウンロードし、インストーラーがサイレントで実行される
- アプリが自動的に再起動して更新完了（Discord / Slack ライクな体験）

## Q5: 「配布デプロイ専用アプリ」とは何ですか？

**A: 開発者・メンテナー向けの GUI リリースツール「IndigoReleaseManager」を実装済みです。これは一般ユーザー向けではありません。**

現在の配布作業は以下の PowerShell スクリプト群を正しい順番で実行する必要があり、手順ミスが起きやすい状態でした。

```
1. Private エンジンの同期
2. バージョン番号の更新
3. Release ビルド
4. ZIPとインストーラーの生成
5. Git tag 作成 → push
6. GitHub Actions の完了確認
```

`IndigoReleaseManager`（WPFアプリ）がこれを **3ボタン・GUIで操作できる形** に整理します。

| 機能 | 内容 |
|:---|:---|
| 環境カード表示 | Public/Private リポジトリの状態・branch・URL を一覧表示 |
| ① Private release ボタン | エンジンのビルド・パッケージ化を実行 |
| ② Public preview ボタン | GitHub Actions で動作確認（preview run） |
| ③ Public release ボタン | 本番タグ作成・push → GitHub Release 自動公開 |
| ログパネル | 全実行結果をリアルタイム表示、ログファイルへ保存 |

**本体スクリプト（PowerShell）を正本として維持しつつ、GUI はその薄いラッパーとして機能します。CLIでも同じ手順を実行可能です。**
