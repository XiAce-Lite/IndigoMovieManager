# 調査結果（UIボトルネック解消）workthree

最終更新日: 2026-03-17

## 1. この文書の位置づけ

- 本書は `workthree` 本線で進める UI 詰まり解消の調査結果と実装メモである。
- `workthree` は、一覧表示、キュー投入、サムネイル作成まで含めて、ユーザーが感じるテンポ感を最優先に改善する本線として扱う。

## 2. 要点

- メイン一覧の重さの本命は、仮想化不足ではなく全件再構築・全件再バインドだった。
- 効いた初手は `FilteredMovieRecs` への一本化と、`ItemsSource` 再代入 / `Items.Refresh()` の撤去だった。
- 次点として、`FilterAndSort` 内の同期 `SELECT *` を UI スレッド外へ逃がした。

## 3. workthree での扱い

- この知見は `workthree` 本線の改善判断に直接使う。
- UI 改善は、通常動画の初動、一覧応答、キュー投入テンポを優先して進める。
- `future` の難読動画検証成果を取り込む時は、この文書の観点でテンポ悪化がないかを先に確認する。

## 4. 当面の判断

- `workthree` では、UI 改善を最優先の実装対象として扱う
- 一覧、Watcher、Queue、サムネイル生成のテンポをつないで改善する
- `future` 側コミットを持ち込む時は、この文書を入口に個別判定する

## 5. 2026-03-11 workthree 次対応

- `FilterAndSort` の `filterList` 全件コピーと全タブ `ItemsSource` 再代入をやめ、`FilteredMovieRecs` の中身更新へ寄せた。
- `SortData` も表示中一覧だけを並べ替える形へ縮退し、全タブ再バインドを外した。
- `Refresh()` の全タブ `Items.Refresh()` をやめ、詳細ペイン同期だけにした。
- `Tabs_SelectionChangedAsync` と watcher の表示中一覧参照を `FilteredMovieRecs` 基準へ寄せた。
- `FilterAndSort` 内の同期 `SELECT *` を UI スレッド外へ逃がし、古い再読込結果で表示を巻き戻さないよう request revision を追加した。
- `DebugRuntimeLog` へ UI 計測ログを追加し、`FilterAndSort`、`SortData`、タブ切替の内訳時間を追えるようにした。
- タブ切替時の `Task.Delay(1000)` 待機を UI 経路から外し、不足サムネ再投入は遅延バックグラウンド処理へ逃がした。
- プレイヤーの時間スライダーは、`Timer_Tick` と `ValueChanged` の相互更新をガードし、確定操作時だけ seek する形へ整理した。
- `サムネイル進捗` タブの `CPU / GPU / HDD` 表示を削除し、対応する `PerformanceCounter` 取得処理も停止した。

今回の狙いは、`workthree` 本線で最優先とする「ユーザーが感じるテンポ感」を、低リスクに改善することにある。

## 6. 2026-03-17 追加の重さ対策

- `FilterAndSortAsync` 完了時に毎回 `RefreshThumbnailErrorRecords()` を叩くのをやめ、失敗タブを dirty 管理へ切り替えた。
- 失敗タブは、通常の一覧更新では「古くなった」という印だけ付け、実際の全件再構築はタブ表示中か明示操作時だけ行う。
- `ThumbnailFailureSync` からの反映も同じ dirty 管理へ寄せ、失敗タブが見えていない時の無駄な全件走査を減らした。
- `ThumbnailPathResolver.TryFindExistingSuccessThumbnailPath()` は、出力フォルダごとの success jpg インデックスを 1 秒だけ使い回す形にし、同一フォルダの再走査を減らした。
- `upper tab viewport` の高頻度 DEBUG ログは停止し、`debug-runtime.log` は 20MB を超えたら追記停止するようにした。
- これにより、一覧更新時の UI スレッド負荷と、DEBUG 実行時の同期ファイル I/O を先に抑える。
