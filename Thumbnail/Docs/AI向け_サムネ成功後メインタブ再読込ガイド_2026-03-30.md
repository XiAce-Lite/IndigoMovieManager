# AI向け サムネ成功後メインタブ再読込ガイド 2026-03-30

最終更新日: 2026-03-30

変更概要:
- サムネ成功後に main tab の画像が変わらない時の正本を追加
- 即時反映と `Reload` 相当再構築の使い分けを明文化
- 今後 AI が様々な導線から再利用できる判断基準を整理

## 1. この資料の目的

- サムネ成功後の main tab 再読込まわりを AI が迷わず触れるようにする
- `path 差し替え` と `Reload 相当再構築` の役割差を固定する
- 「成功したのに画像が変わらない」時に、どこを見るかを共通化する

## 2. 現在の結論

- 軽い成功反映だけで済む場合:
  - `TryApplyThumbnailPathToMovieRecord(...)`
  - `RefreshVisibleThumbnailUiAfterImmediateThumbnailSuccess(...)`
- それでも main tab が変わらない場合:
  - `RequestMainTabFullReloadAfterThumbnailSuccess(...)`
  - 後段で `FilterAndSort(MainVM.DbInfo.Sort, true)` を1回だけ流す

今回の実機確認では、
- 軽い再描画だけでは main tab が変わらないケースがある
- ヘッダーの `Reload` ボタンなら更新される

このため、今の正本は
- まず軽い即時反映
- 次に短いデバウンス付きの `Reload` 相当再構築

の二段構えである。

## 4. 処理の流れ

### 4.1 通常の成功反映

入口:
- `CreateThumbAsync(...)`

主処理:
- `TryApplyThumbnailPathToMovieRecord(...)`
- `TryReflectRescuedThumbnailIntoUpperTabRescueItems(...)`
- `TryReflectCreatedThumbnailIntoUpperTabDuplicateItems(...)`

役割:
- まず対象 `MovieRecords` の `ThumbPath*` を差し替える
- 同一パス上書きでも空経由で再通知する
- 特殊タブ側のコピー済み ViewModel も合わせて差し替える

### 4.2 軽い visible UI 再評価

入口:
- `RefreshVisibleThumbnailUiAfterImmediateThumbnailSuccess(...)`

役割:
- `InvalidateThumbnailErrorRecords(refreshIfVisible: true)`
- `Refresh()`
- `RequestUpperTabVisibleRangeRefresh(immediate: true, reason: ...)`
- `RequestThumbnailErrorSnapshotRefresh()`
- `RequestThumbnailProgressSnapshotRefresh()`

用途:
- 即時に見た目を追従させたい時の第一段
- manual rescue の即時成功
- preferred なユーザー明示作成成功

### 4.3 `Reload` 相当の後段再構築

入口:
- `RequestMainTabFullReloadAfterThumbnailSuccess(...)`

内部:
- `DispatcherTimer` で短く圧縮
- `ThumbnailSuccessMainTabReloadTimer_Tick(...)`
- `FilterAndSort(MainVM.DbInfo.Sort, true)`

役割:
- ヘッダー `Reload` ボタンと同じ再構築筋へ寄せる
- 複数成功が連続しても最新1回へ圧縮する
- `path 差し替え` だけで届かない main tab の表示更新を拾う

## 5. なぜ `Reload` 相当が必要か

今回の観測事実:
- 成功後に画像が変わらない
- ただしヘッダー `Reload` ボタンなら更新される

この事実から、
- 単なる `PropertyChanged`
- 軽い `Refresh()`
- visible range 再評価

だけでは足りず、実際には
- 一覧ソース再構築
- サムネパス再解決

まで含む `Reload` 系の筋が効いていると判断した。

## 6. どの導線で使うか

使う:
- `等間隔サムネイル作成` のようなユーザー明示作成
- manual rescue 成功
- 今後増える「ユーザーが結果をすぐ見たい単動画処理」

慎重に使う:
- 自動 watch 起点の大量成功
- バックグラウンドで連続発火する bulk 成功

理由:
- `FilterAndSort(..., true)` はヘッダー `Reload` 相当で比較的重い
- 自動系へ無差別に入れるとテンポを壊す

## 7. AI が守るべきルール

- まずは `TryApplyThumbnailPathToMovieRecord(...)` を壊さない
- 次に軽い即時反映 helper を使う
- それでも main tab が追従しない時だけ `RequestMainTabFullReloadAfterThumbnailSuccess(...)` を使う
- 自動系 bulk 成功へ安易に `FilterAndSort(..., true)` を直書きしない
- 新しい成功導線を増やしたら、この資料と `Thumbnail/README.md` を同時更新する

## 8. 実装上の注意

- `ThumbPathBig10` は `PropertyChanged` 名ミスがあると更新が飛ばない
- 同一 jpg の上書きでは、空経由の forced rebind が必要
- `RequestMainTabFullReloadAfterThumbnailSuccess(...)` は UI スレッドで扱う
- timer は最新1回に圧縮する前提で使う

## 9. テスト観点

最低限:
- `TryApplyThumbnailPathToMovieRecord(...)` が同一パスでも2回通知する
- `ThumbPathBig10` が正しい property 名で通知する
- `ShouldRefreshVisibleThumbnailUiAfterCreate(...)` が preferred のみ true

実機観点:
- main tab で成功直後に画像が変わるか
- 少し遅れてでも `Reload` 相当で最終的に追従するか
- 連続成功時に過剰な全面再読込になっていないか

## 10. 迷った時の判断

- 「軽く直る」なら軽い即時反映で止める
- 「Reload でしか直らない」なら後段 full reload を使う
- 「自動系にも入れたい」時は、まずテンポ悪化リスクをレビューする

今後 AI がこの領域を触る時は、この資料を正本として更新してからコードを変えること。
