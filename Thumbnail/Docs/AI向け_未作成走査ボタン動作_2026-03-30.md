# AI向け 未作成走査ボタン動作 2026-03-30

最終更新日: 2026-03-30

変更概要:
- `未作成走査` ボタンの入口、処理内容、投入先を整理
- `救済worker` 直送ではなく通常キュー再投入であることを明記
- 次に仕様変更や導線整理をする時の確認ポイントを残す

## 1. この資料の目的

- `未作成走査` ボタンが何をするかを AI がすぐ追えるようにする
- `通常キュー` と `救済worker` のどちらへ流れるかを誤認しないようにする
- 今後このボタンの仕様を変える時の起点を残す

## 2. 現在の結論

- `未作成走査` は、指定タブのサムネイルが未作成な動画を DB 全体から探す
- 見つかった欠損動画は `QueueObj` を作って通常サムネ作成キューへ再投入する
- `救済worker` へ直接送る処理ではない
- `ERROR` マーカーや open rescue request がある個体は、この走査では投入しない

## 3. 入口

UI 定義:
- `BottomTabs/ThumbnailProgress/ThumbnailProgressTabView.xaml`
  - `Content="未作成走査"`

イベント配線:
- `Views/Main/MainWindow.xaml`
  - `MissingThumbnailScanRequested="ThumbnailProgressMissingScan_Click"`

クリック処理:
- `BottomTabs/ThumbnailProgress/MainWindow.BottomTab.ThumbnailProgress.cs`
  - `ThumbnailProgressMissingScan_Click(...)`

## 4. クリック時の前処理

`ThumbnailProgressMissingScan_Click(...)` では、先に次を行う。

- `_isThumbnailProgressMissingScanRunning` で二重実行防止
- `MainVM.DbInfo.DBFullPath` が無い時はエラー表示
- `DBName` / `ThumbFolder` が無い時はエラー表示
- 走査対象タブを決定

## 5. 対象タブの決め方

対象タブは `GetCurrentThumbnailActionTabIndex()` で取る。

ただし:
- `0..4` 以外なら `Grid` にフォールバックする

つまり:
- `Small / Big / Grid / List / Big10` 上で押した時はそのタブを走査
- `サムネ救済` や特殊タブで押した時は `Grid` を走査

## 6. 本体処理

本体は `Watcher/MainWindow.Watcher.cs` の
- `EnqueueMissingThumbnailsAsync(...)`

ここで次を行う。

1. 対象タブのサムネ出力先を決定
2. DB の `movie` テーブルから `movie_id, movie_path, hash` を全件取得
3. 出力フォルダ内の既存サムネファイル名一覧を作る
4. FailureDb の open rescue request 一覧を読む
5. 各動画について期待サムネファイル名を計算
6. 欠損なら enqueue 候補にする

## 7. skip 条件

次の個体は投入しない。

- `movie_path` が空
- `Path.GetFileNameWithoutExtension(path)` が取れない
- 動画ファイル自体に触れない
- `.#ERROR.jpg` がある
- 同じ `movie/tab` の open rescue request がある

関連ロジック:
- `ResolveMissingThumbnailAutoEnqueueBlockReason(...)`
- `BuildMissingThumbnailRescueBlockKey(...)`

## 8. 投入先

ここが重要。

`EnqueueMissingThumbnailsAsync(...)` は候補を `QueueObj` にして、
- `Priority = ThumbnailQueuePriority.Normal`
- `Tabindex = targetTabIndex`

で `batch` に積む。

その後:
- `FlushPendingQueueItems(batch, "RescueMissingThumbnails")`

が呼ばれ、内部で
- `TryEnqueueThumbnailJob(pending, bypassDebounce)`

へ流れる。

結論:
- 投入先は通常サムネ作成キュー
- `救済worker` 直送ではない

## 9. debounce とバッチ

- 100件単位で flush する
- flush の間に `await Task.Delay(50)` で少し息継ぎする
- `folderPath == "RescueMissingThumbnails"` の時は debounce を bypass する

狙い:
- 欠損走査の大量投入で UI を固めにくくする
- 既に同一走査で拾った分を途中で落としにくくする

## 10. watch 起点との違い

`EnqueueMissingThumbnailsAsync(...)` は watch 側からも使われる。

ただし manual ボタン起点では:
- `isWatchMode: false`
- stale scope / UI suppression の watch 特有制御は素通り

つまり:
- ボタン起点は素直なワンショット走査
- watch 起点は stale scope や UI suppression を見る

## 11. ユーザー向け挙動

成功時:
- 進捗スナップショット更新要求
- ログ出力

失敗時:
- `未作成走査の開始に失敗しました。` の `MessageBox`

補足:
- 成功時に「何件入れたか」の popup は今は出さない

## 12. AI が守るべきポイント

- `未作成走査` を `救済worker` ボタンと誤認しない
- `ERROR` 個体まで拾いたいなら、このボタンの責務変更になる
- `Priority` を変えると体感と混雑特性が変わる
- `Grid` フォールバックを変える時は UI 意味づけも一緒に見直す

## 13. 変更検討時の論点

- `未作成走査` を通常キューのままにするか
- 単動画時だけ優先度を上げるか
- 成功件数を overlay で返すか
- `サムネ救済` タブ上で押した時の `Grid` フォールバックが妥当か
