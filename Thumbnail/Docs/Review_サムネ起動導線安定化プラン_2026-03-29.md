# Review_サムネ起動導線安定化プラン_2026-03-29

最終更新日: 2026-03-29

変更概要:
- `Implementation Plan_サムネ起動導線安定化_2026-03-29.md` の妥当性をレビュー
- 過剰設計の芽と見落としやすいリスクを先に整理

## 1. レビュー結論

- 方針は妥当。
- ただし「導線共通化」を急ぎすぎて一気に抽象化すると、`workthree` の体感テンポを落とす危険がある。
- 実装は 1 回で完成形へ行かず、まず選択文脈の是正と結果型の明確化から入るべき。

## 2. 良い点

- route 差分を単なるログ追加ではなく、入口契約の差として捉えている。
- 下部 `サムネ失敗` タブの右クリック文脈ズレを、局所現象ではなく設計上の穴として明示できている。
- manual slot busy を「受理したのに始まらない」体験の揺れとして扱っており、ユーザー体感の観点に合っている。

## 3. リスク

### 3.1 dispatcher 化を一気にやると変更面が広い

- `MainWindow` partial 群、`Watcher`、`BottomTabs`、`UpperTabs` をまたぐため、1 回で大きく寄せると回帰範囲が広がる。
- 先に `SelectionSource` と `StartResult` だけ導入し、route の本体委譲先は既存メソッドを再利用した方が安全。

### 3.2 direct index repair は rescue request と同列には寄せ切れない

- これは FailureDb を経由しない別物なので、無理に rescue request へ統合すると責務が濁る。
- 今回は「busy / unsupported / started を明確に返す」までに留めるのが現実的。

### 3.3 `等間隔サムネイル作成` の意味を変えると体感が変わる

- 既存では rescue request 扱いで動いている。
- ここを通常キューへ戻すか、明示再作成として rescue のまま整理するかで体感が変わる。
- 実装前に「この操作は既存成功 jpg を上書き再作成したいのか」を決めてから触る必要がある。

### 3.4 ポップアップ義務は自動経路へ漏らさない

- 今回の追加要件は「ユーザー要請の場合」だけである。
- これを可視範囲救済や placeholder 自動救済へ混ぜると、通常運用でポップアップが過多になりテンポを壊す。
- 要求生成時に `IsUserInitiated` を持たせるなど、境界を明確にした方が安全。

## 4. 推奨する実装順の補正

1. 下部 `サムネ失敗` タブの選択文脈修正
2. route 共通の `StartResult` 導入
3. manual rescue の busy 再評価補強
4. `等間隔サムネイル作成` の意図整理
5. その後に DTO / dispatcher を薄く導入

## 5. 追加で見るべき確認点

- `menuContext` を使う他の下部タブでも、上側選択前提の誤用がないか
- `List_SelectionChanged` の共通処理が下部 `サムネ失敗` タブ選択時に副作用を起こしていないか
- manual slot 2 本が busy の時、FailureDb 側の pending がどれくらいの周期で default slot に拾われるか
- 進捗 UI の文言が `Started` と `Accepted but waiting` を区別できるか
- ポップアップ文言が route ごとにばらけず、同じ結果型なら同じ説明になるか

## 6. レビュー後の最終判断

- プランはこのまま進めてよい。
- ただし最初の実装単位は小さく切るべき。
- 第一コミット候補は「下部 `サムネ失敗` タブ右クリック文脈修正 + そのテスト」。

## 7. 実装反映メモ（2026-03-29 追記）

- 追加要件の「ユーザー要請なら必ずポップアップで答える」は、右クリック系だけでなく上側 `サムネ救済` タブと下部 `サムネ失敗` タブのボタン操作にも反映した。
- manual slot busy の返答も共通 builder に揃え、`空いてから再実行してください。` を標準案内にした。
- `dispatcher / DTO` は、入口全置換ではなく件数集計と開始結果の共通化に留める形で実装し、過剰抽象化は避けた。
