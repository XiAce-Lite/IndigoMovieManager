# Implementation Plan + tasklist（サムネイル並列レーン化と大動画低優先制御 2026-03-05）

## 0. 背景
- `+ / -` で並列数設定は更新できるが、体感上「増やしても使われない」ケースがある。
- 大動画が先に実行枠を占有すると、短時間で完了できる小動画の表示反応が鈍る。
- 既存の `Task + ThreadPool` 構成は維持しつつ、ジョブ優先度制御で詰まりを減らしたい。

## 1. 目的
- 総並列数 `N` の範囲内で、小動画の完了を優先して体感レスポンスを改善する。
- 大動画レーンは「低優先」で実行し、UI操作時や高負荷時の詰まりを抑える。
- 設定変更（並列数変更）を実行中バッチにも即時反映する。

## 2. 要件（今回）
- 総並列数は常に `N` を上限とし、過並列にしない。
- レーン構成
  - 小動画レーン: 1枠（優先度高）
  - 大動画レーン: 1枠（優先度低）
  - 通常レーン: `N - 2` 枠（`N < 3` の場合は自動縮退）
- 大動画レーン低優先ルール
  - 空き枠がある時のみ実行
  - 高負荷時は抑制（起動閾値あり）
  - 小動画/通常レーンが空なら枠を貸与可能
- 設定変更時は、進捗タブ表示と実行並列の両方が即時に追従する。

## 3. スコープ
- IN
  - `ThumbnailQueueProcessor` のスケジューリング制御
  - レーン別ジョブ分類（小/大/通常）
  - 大動画レーン低優先ポリシー
  - 設定値の即時反映とログ
- OUT
  - サムネイル生成エンジン自体のアルゴリズム変更
  - QueueDB スキーマ大改修（列追加なしで開始）
  - UIデザイン全面改修

## 4. 設計方針
- OSスレッド優先度ではなく、アプリ内スケジューラで優先度制御する。
- `Parallel.ForEachAsync` は維持し、動的ゲートで同時実行数を制御する。
- 大動画低優先は「実行枠配分」と「起動条件」で実現し、`Thread.Priority` は使わない。
- ログで挙動を可視化し、チューニングは設定値で行えるようにする。

## 5. 実装詳細

### 5.1 レーンモデル
- ジョブを以下で分類する。
  - Small: `MovieSizeBytes <= SmallThresholdBytes`
  - Large: `MovieSizeBytes >= LargeThresholdBytes`
  - Normal: 上記以外
- `N` に応じた枠配分
  - `N=1`: 通常レーン1
  - `N=2`: 小動画1 + 通常1（大動画専用枠なし）
  - `N>=3`: 小動画1 + 大動画1 + 通常`N-2`

### 5.2 大動画低優先ポリシー
- 大動画レーンの起動条件
  - `queueActiveCount >= LargeLaneStartActiveThreshold`
  - 直近の失敗率が閾値未満
- 抑制条件
  - UI高負荷（CPU閾値超え）時は一時停止
  - 小動画待機が多い時は大動画枠を通常へ貸与
- 復帰条件
  - 抑制解除後、クールダウン経過で再開

### 5.3 リース戦略
- 小動画候補はサイズ昇順で優先取得。
- 大動画候補はサイズ降順だが、取得件数は最小（例: 1件）に制限。
- 通常レーンは既存の優先タブロジックを維持。

### 5.4 即時反映
- `+ / -` や設定画面スライダー変更時に
  - 表示: `ConfiguredParallelism` を先行反映
  - 実行: 動的ゲートの上限を即時更新
- レーン配分も同タイミングで再計算する。

### 5.5 計測ログ
- 追加ログ
  - `lane assign`: small/normal/large の割当結果
  - `lane throttle`: large 抑制/復帰イベント
  - `parallel apply`: 実適用並列の変更
- 指標
  - 小動画完了までの P50/P95
  - 大動画レーンの稼働率
  - 抑制発生回数

## 6. フェーズ

### Phase A: 骨格実装
- レーン分類関数と枠配分計算を追加。
- 動的ゲートと統合。

### Phase B: レーン別取得
- 小/大/通常の取得順制御を実装。
- 既存優先タブロジックとの整合を取る。

### Phase C: 大動画低優先化
- 起動閾値、抑制、貸与、復帰を実装。
- 低優先時のログを追加。

### Phase D: 設定・UI連動
- 設定変更時の即時反映を確実化。
- 進捗タブ表示と実行値の乖離を解消。

### Phase E: 検証
- 単体テスト/回帰テスト追加。
- 実データで高負荷検証。

## 8. リスクと対策
- リスク: レーン固定で一方が遊休になる
  - 対策: 枠貸与（borrow）を必須実装する。
- リスク: 大動画が進まず滞留する
  - 対策: 抑制しつつも最低実行率を保証する。
- リスク: ロジック複雑化で回帰
  - 対策: テストを配分・抑制・復帰で分離し段階導入する。
