# Implementation Plan（サムネイル進捗ミニパネル表示遅延対策 2026-03-04）

## 1. 目的
- サムネイル生成完了後、右側ミニパネルへ「すぐ表示される」体感を取り戻す。
- UI更新詰まりを減らし、一覧操作や再生操作の反応低下を防ぐ。
- WhiteBrowser互換サムネイル形式は維持し、保存フォーマットは変更しない。

## 2. 現状の詰まりポイント（コード観測）
- `Converter/NoLockImageConverter.cs`
  - `LoadBitmapNoLock` で `WriteableBitmap` を毎回生成しており、ミニ表示でもフル画像をデコードしている（行147-173）。
- `MainWindow.xaml`
  - ミニパネル画像は `PreviewImagePath` の単一バインドのみ（行1014）。
- `ModelViews/ThumbnailProgressViewState.cs`
  - `SetField` が同値時に通知しないため、同一パス上書き時は画像更新通知が飛ばない（行259-268）。
  - `QueueLogs.Clear()` + 再追加を500msごとに実施しており、UI差分更新になっていない（行125-131）。
- `MainWindow.xaml.cs`
  - 500msタイマー内でスナップショット適用＋メーター取得をUIスレッドで一括実行している（行576-595）。

## 3. 方針（WriteableBitmap側 / WPF側）
- WriteableBitmap側:
  - ミニ表示専用の軽量デコード経路を作り、読み込みサイズを制限する。
  - ファイルI/Oリトライの挙動を安定化し、保存直後の取りこぼしを減らす。
- WPF側:
  - 同一パス更新でも再描画トリガーを確実に発火させる。
  - 500msごとの全量書き換えを減らし、差分更新中心にする。

## 4. 実装フェーズ

### Phase 0: 計測追加（先に事実を取る）
- 追加計測
  - `MarkThumbnailSaved` 時刻
  - コンバータで画像が実際に `ImageSource` 化された時刻
  - `UpdateThumbnailProgressUi` 1回あたり処理時間
- 成果物
  - `logs/thumbnail-progress-latency.csv`（新規）
- 完了条件
  - 「保存完了→ミニ表示」遅延のP50/P95が採取できる。

### Phase 1: WriteableBitmap側の軽量化
- 対応内容
  - `NoLockImageConverter` に「ミニ表示向けデコードサイズ指定」を追加。
  - ミニパネル（高さ72）では `DecodePixelHeight` を使い、必要サイズでデコードする。
  - `WriteableBitmap` 固定生成をやめ、`BitmapFrame/BitmapImage` の軽量経路を優先する。
  - キャッシュキーに「表示種別（通常/ミニ）」を含め、混線を防ぐ。
- 完了条件
  - ミニパネル画像デコード時間が現状比で有意に短縮。
  - 画像品質が実用上劣化しない（72px表示で破綻しない）。

### Phase 2: WPF側の再描画保証と差分化
- 対応内容
  - `PreviewImagePath` に加えて `PreviewRevision`（または `PreviewLastWriteTicks`）をViewStateへ追加。
  - 同一ファイルパス上書き時も `PreviewRevision` を増分して再バインドさせる。
  - 画像バインドを `MultiBinding`（Path + Revision）にして、確実に再評価させる。
  - `QueueLogs` は全クリア再追加をやめ、差分更新に変更する。
  - `WorkerPanels` 更新で未変更項目への代入を最小化する。
- 完了条件
  - 同一パス上書きでもミニパネルが更新される。
  - 高負荷時でもUIスレッド詰まりが目視で減る。

### Phase 3: 更新頻度の最適化
- 対応内容
  - 進捗テキスト更新（500ms）と画像更新（イベント駆動）を分離。
  - メーター取得はバックグラウンド採取値をUIへ反映する方式に切り替え。
- 完了条件
  - サムネイル大量作成中のスクロール/選択操作の引っかかりが許容範囲。

## 5. 受け入れ条件（今回）
- サムネイル保存完了からミニパネル表示まで:
  - P95 <= 800ms
  - Max <= 2000ms（例外ケースを除く）
- 同一パス再生成時に表示が古いまま残らない。
- サムネイル進捗タブ表示中でも、一覧操作体感を悪化させない。

## 6. 実装順（推奨）
1. Phase 0（計測）を先に実装して、遅延の主因比率を数値化する。
2. Phase 1（WriteableBitmap軽量化）でデコード負荷を先に下げる。
3. Phase 2（WPF再描画保証）で同一パス更新漏れを潰す。
4. 必要ならPhase 3（更新頻度最適化）を追加する。

## 7. リスクと対策
- リスク: デコード縮小で画質劣化
  - 対策: ミニパネル専用経路に限定し、通常一覧は現行品質を維持する。
- リスク: MultiBinding化でバインド記述が複雑化
  - 対策: 対象をミニパネル画像バインド1箇所に限定する。
- リスク: 計測ログ追加でI/O増加
  - 対策: DEBUG限定またはサンプリング率を下げる。
