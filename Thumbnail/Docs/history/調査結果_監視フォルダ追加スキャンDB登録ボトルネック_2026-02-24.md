# 調査結果（監視フォルダ追加時のスキャン・DB登録ボトルネック）

## 1. 調査目的
- 「監視フォルダ追加時の遅さ」を処理区間ごとに分解し、まず律速点を特定する。

## 2. 対象ログ
- `%LOCALAPPDATA%\IndigoMovieManager_fork\logs\debug-runtime.log`
- 重点ケース（重負荷）  
  `2026-02-24 21:54:33.711` 開始  
  `2026-02-24 22:01:01.803` 終了  
  `scanned=2034, new=2034, enqueued=2034`

## 3. 分解計測結果（重負荷ケース）
- 全体時間: `388.092秒`（`elapsed_ms=388714`）
- スキャン開始 -> 初回100件キュー投入: `33.104秒`
- キュー投入バッチ数: `21`（`100 x 20 + 34`）
- キュー投入間隔（平均）: `17.718秒 / 100件`
- キュー投入間隔（最大）: `26.973秒 / 100件`
- `enqueue -> persister upsert` 反映遅延（平均）: `157.48ms`
- `enqueue -> persister upsert` 反映遅延（最大）: `195ms`
- スキャン区間（scan start -> scan end）: `387.466秒`

## 4. 比較用の軽負荷ケース
- `scanned=240, new=0` の場合: `0.916秒`
- 差分なしケースは短時間で終わるため、律速は「新規あり時の後段処理」。

## 5. ボトルネック判定

### 5.1 QueueDB（Persister）は主因ではない
- `enqueue` の直後に `persister upsert` が走っており、遅延は `~0.16秒` 程度。
- したがって「キュー永続化」は律速ではない。

### 5.2 主因は `CheckFolderAsync` の新規1件ごとの直列重処理
- `MovieInfo` 生成（動画解析 + ハッシュ）
- `InsertMovieTable` 実行（接続/採番/INSERT）
- 1件ごとのUI再クエリ/再反映

## 6. 該当コード（確認済み）
- 新規1件ごとの処理ループ  
  `MainWindow.Watcher.cs:341` `MovieInfo mvi = new(movieFullPath);`  
  `MainWindow.Watcher.cs:342` `await InsertMovieTable(...)`  
  `MainWindow.Watcher.cs:382` `DataTable dt = GetData(...)`
- DB挿入側の加算要因  
  `DB/SQLite.cs:454` `select max(movie_id) from movie`  
  `DB/SQLite.cs:476` `if (Path.Exists("sinku.exe"))`  
  `DB/SQLite.cs:574` `await Task.Delay(5);`

## 7. 速報結論
- 今回の遅延は「スキャン自体」より「スキャン後の新規1件処理の直列構造」が支配的。
- 最優先は以下。
1. 1件ごとのUI再クエリを廃止して、フォルダ単位で反映する。
2. `InsertMovieTable` 末尾の `await Task.Delay(5)` を除去する。
3. movie登録をバッチ化し、`max(movie_id)` 取得回数を削減する。

## 8. 次回計測で追加すべき分解点
- `MovieInfo` 生成時間（合計/平均）
- DB登録時間（合計/平均）
- UI反映時間（合計/平均）
- フォルダ列挙時間（合計/平均）

これを `CheckFolderAsync` 内で区間タイムを記録し、改善前後を同条件（同一2034件）で比較する。
