# 動画・サムネパス取得 一元化考察 + UI 詰まり防止 🎬

作成: 2026-03-18
更新: 2026-03-18（UI詰まり防止・Everything考察を追加）

> **調査の本来目的**: 大量パス取得でUIスレッドが詰まることの防止。
> Everything 利用の適否も含めて考察する。

---

## 0. 問題の本質：なぜ UI が詰まるのか？

大量の動画・サムネパスを取得する処理には主に 2 種類ある。

| 処理 | 同期/非同期 | UI 詰まりリスク |
|---|---|---|
| ファイルシステムスキャン（Directory.Enumerate） | 同期 I/O | **高**。フォルダ内ファイル数 × ディスクアクセス |
| Everything IPC クエリ（`EverythingProvider`） | 同期（SearchClient.Search） | **中**。IPC 応答待ち 1.5 sec タイムアウトあり |
| EverythingLite インデックス再構築（`EverythingLiteProvider`） | `.GetAwaiter().GetResult()` でブロック | **高**。同期ブロック確定 |
| サムネイルボディ収集 `CollectThumbnailBodies` | 同期 Directory.EnumerateFiles | **中**。サムネ枚数 × ディスクアクセス |
| Watcher スキャン（`MainWindow.Watcher.cs`） | async Task で包まれている | **低**。ただし UI Dispatcher.InvokeAsync 呼び出し頻度次第 |

**実際の呼び出しスタック**
```
[UI イベント or ポーリングタイマ]
  ↓
EverythingFolderSyncService / ScanLogic（async）
  → IndexProviderFacade.CollectMoviePathsWithFallback（呼び出し元 thread 依存）
      → EverythingProvider.CollectMoviePaths（同期 IPC ブロック）
      → EverythingLiteProvider.CollectMoviePaths
          → QueryIndexedItems ← RebuildIndexAsync().GetAwaiter().GetResult() 🔥同期ブロック!
  → CollectThumbnailBodiesWithFallback
      → EverythingLiteProvider.CollectThumbnailBodies（同期 Directory.EnumerateFiles）
```

## 1. 現状のパス解決チェーン

```
DBInfo.ThumbFolder (設定から読んだ生の値)
    ↓
ThumbRootResolver.ResolveRuntimeThumbRoot(DBFullPath, DBName, ThumbFolder)
    = "実際に使うサムネ根フォルダ" を決定
    ↓
ResolveThumbnailOutPath(tabIndex, dbName, thumbFolder)
    = TabIndex × ThumbRootResolver で "タブ別出力フォルダ" を決定
    ↓
ThumbnailPathResolver.BuildThumbnailPath(outPath, movieNameOrPath, hash)
    = "最終サムネファイルパス" を生成
```

上の3段が、メインウィンドウ側では `MainWindow.ThumbnailPaths.cs` の
`BuildCurrentThumbnailPath(tabIndex, movieNameOrPath, hash)` に一本化されている。

## 2. 問題: サムネ根の取得が散らばっている

`ThumbRootResolver.ResolveRuntimeThumbRoot(...)` または `GetDefaultThumbRoot(...)` を
直接呼んでいる箇所が複数ある。

| ファイル | 内容 |
|---|---|
| `Views/Main/MainWindow.xaml.cs:1156` | DB 開き時に `DbInfo.ThumbFolder` を初期化 |
| `Thumbnail/MainWindow.ThumbnailPaths.cs:34` | `ResolveRuntimeThumbnailRoot()` でラップ済み |
| `Thumbnail/ThumbnailRescueWorkerLauncher.cs:281` | ワーカー起動時のパス解決 |
| `src/IndigoMovieManager.Thumbnail.Engine/ThumbnailMovieMetaResolver.cs:52` | エンジン側の既定パス解決 |
| `src/IndigoMovieManager.Thumbnail.RescueWorker/RescueWorkerApplication.cs:3530` | ワーカー側の既定パス解決 |
| `BottomTabs/DebugTab/MainWindow.BottomTab.Debug.cs:511` | デバッグ表示用 |

→ `ThumbnailPaths.cs` のラッパーを通らずに直呼びしているケースが残存。

```
MainWindow.ThumbnailRescueLane.cs  : ThumbFolder を 3 箇所で直接取得

```

これらは全部「実運用のサムネ根を取る」用途で同じことをしている。

## 4. 問題: `Movie_Path` の直接参照

動画パス自体は `MovieRecords.Movie_Path` が正規プロパティで、それを各ファイルが直接触っている。
`MovieRecords` に `Movie_Path_Normalized`（スラッシュ正規化版）があるので DB クエリ・UI バインドに使い分けあり。

主要な利用カテゴリ:
- **ファイル操作**: コピー・移動・削除 (`MenuActions.cs`)
- **プレイヤー起動**: パスをそのまま渡す (`Player.cs`)
- **Watcher/スキャン**: ルックアップ用 HashSet の素材
- **サムネ生成**: `ThumbnailCreation.cs` / `RescueLane.cs` でファイル名抽出に使用
  → ここは `ThumbnailPathResolver.BuildThumbnailFileName(movie.Movie_Path, hash)` に委ねてよい

`Movie_Path` を直接使う箇所は「動画そのもの」を扱う操作なので、これをラップしすぎるとかえって不明瞭になる。**一元化は不要**。

## 5. 問題: `MovieRecords` のサムネパスプロパティが6種ある

```csharp
ThumbPathSmall   // 小サムネ
ThumbPathBig     // 大サムネ
ThumbPathGrid    // グリッドタブ
ThumbPathList    // リストタブ
ThumbPathBig10   // 大き目サムネ
ThumbDetail      // Extension/Bookmark の詳細タブ
```

これら「どのタブ index → どのプロパティ」の対応を知る必要があるコードが、
現在は呼び出し側に分散して `if(tabIndex == 99) ThumbDetail else ThumbPathXxx...` 系の条件分岐を書いている箇所がある（`MissingThumbnailRescuePolicy.TryApplyThumbnailPathToMovieRecord` など）。

## 6. 一元化できる箇所・できない箇所の整理

### ✅ 一元化できる（済み or 改善余地あり）

#### A. サムネ根の取得（高優先）
`MainVM?.DbInfo?.ThumbFolder` を複数箇所で直接引いている部分は、
`MainWindow.ThumbnailPaths.cs` に既にある `ResolveCurrentThumbnailRoot()` /
`ResolveCurrentThumbnailOutPath(tabIndex)` に統合できる。

> 具体的には `Thumbnail/MainWindow.ThumbnailRescueLane.cs` などが
> `MainVM?.DbInfo?.ThumbFolder` に直触りしているので、
> `ResolveCurrentThumbnailOutPath(tabIndex)` 経由に書き換えればよい。

#### B. サムネパス生成（ほぼ済み）
`ThumbnailPathResolver` が一元化されており、直接 Path.Combine してる箇所は既にない。
`BuildCurrentThumbnailPath` を通していない外部プロセス側（Engine/RescueWorker）は
引数受け渡しなので問題なし。

#### C. tabIndex → ThumbPathプロパティの対応（改善余地あり）
`TryApplyThumbnailPathToMovieRecord` のような「tabIndex を見てどのプロパティを書くか」
判断はすでにメソッドに抽出されており良い形。ただしこの対応表を
`MovieRecords` 側もしくは専用クラスに持たせる余地はある。

### ❌ 一元化しなくてよい

| 対象 | 理由 |
|---|---|
| `Movie_Path` 直接参照 | 動画ファイルの操作・再生はそのままパスを扱う自然な処理。ラップは逆効果 |
| エンジン/ワーカー側の `ThumbRootResolver` 直呼び | 別プロセス起動なので UI 側クラスに依存できない |
| `Watcher.cs` のスナップショット変数 | 非同期スキャン中に値が変わらないよう意図的に一時コピーしている |

## 7. 推奨アクション（優先順）

### 🔥 高優先
**`MainVM?.DbInfo?.ThumbFolder` 直引きを `ResolveCurrentThumbnailOutPath(tabIndex)` に統合**

対象ファイル:
- `Thumbnail/MainWindow.ThumbnailRescueLane.cs`（3 箇所）
- `Thumbnail/MainWindow.ThumbnailQueue.cs`（1 箇所）
- `Thumbnail/MainWindow.ThumbnailFailureSync.cs`（1 箇所）
- `Thumbnail/MainWindow.ThumbnailRescueWorkerLauncher.cs`（partial, 1 箇所）

```csharp
// Before
string thumbFolder = MainVM?.DbInfo?.ThumbFolder ?? "";
string outPath = ResolveThumbnailOutPath(tabIndex, currentDbName, thumbFolder);

// After
string outPath = ResolveCurrentThumbnailOutPath(tabIndex);
```

### 💡 中優先
`ThumbnailFailedTab.cs` のサムネ根取得も同じ流れで統合できるが、
コンテキスト構造体 (`ThumbnailErrorTabContext`) 経由で渡している設計を活かしつつ
初期化箇所だけを `ResolveCurrentThumbnailRoot()` に変えれば OK。

### 📝 低優先
`MovieRecords` の "tabIndex → Thumbプロパティ" 対応表のドキュメント整備。
バグリスクは低いが、新しいタブを追加するときに見落としやすいので表にまとめておくと吉。

|---|---|---|
| サムネパス生成 | `ThumbnailPathResolver` で一元化済み ✅ | そのまま |

## 9. UI 詰まり防止の本命改善案 🔥

### 9-A. EverythingLiteProvider のブロッキング解消（最優先）

**問題箇所**: `EverythingLiteProvider.cs:249-252`

```csharp
// 🔥 ここが SYNC BLOCK！呼び出し元スレッドをそのままブロックする
_ = cacheEntry.Service.RebuildIndexAsync(null, CancellationToken.None)
    .GetAwaiter()
    .GetResult();
```

インデックス再構築は数万ファイルを舐めるため、ケースによっては**秒単位**でブロックする。
`EverythingLiteProvider.QueryIndexedItems` が `lock(SyncRoot)` 内で呼ばれているため
単純な `await` 化はデッドロックを引き起こす可能性がある。

**改善方針**:
```
Option A: CollectMoviePaths 自体を async に変えて await する（IFileIndexProvider の contract 変更が必要）
Option B: RebuildIndexAsync は Task.Run でプールに逃がし、結果だけ lock 外で受け取る
Option C: バックグラウンドで定期再構築（周期タイマ）し、CollectMoviePaths は常にキャッシュを見るだけにする ← 推奨
```

Option C が最もシンプルでUIに安全。「常に最新インデックス」とはならないが
3 秒クールダウン（`RebuildCooldown`）は既にあるため整合性は問題ない。

---

### 9-B. EverythingProvider (IPC) の呼び出し元スレッド確認

**現状**: `SearchClient.Search()` は同期呼び出し・タイムアウト 1500ms。
呼び出し元が既に `Task.Run` や bg-thread なら問題ない。
呼び出し元が UI スレッドなら **1.5sec フリーズ** になる。

**確認が必要**: `EverythingFolderSyncService` や `MainWindow.Watcher.cs` からの呼び出しが
`await Task.Run(...)` に包まれているかを確認する。

> 現状の `MainWindow.Watcher.cs` は async Task で設計されているが、
> `CollectMoviePathsWithFallback` の呼び出し部分を確認して
> `ConfigureAwait(false)` + bg thread で動いているか確認要。

---

### 9-C. `CollectThumbnailBodies` の Everything 活用（さらに高速化）

**現状 `EverythingLiteProvider.CollectThumbnailBodies`**:
```csharp
// EverythingLite なのに！同期 Directory.EnumerateFiles を使っている
foreach (string filePath in Directory.EnumerateFiles(thumbFolder, "*.jpg", ...))
```

一方 `EverythingProvider.CollectThumbnailBodies` は Everything IPC クエリを使っている。

**改善案**: EverythingLite でも同じく内部インデックスからサムネ収集する（既存の `CollectMoviePaths` と同じ方法でフィルタ）ことで Directory.Enumerate を省ける。

---

### 9-D. Everything 使用の整理まとめ

| 機能 | 現状 | 推奨 |
|---|---|---|
| 動画パス収集 | Everything IPC / EverythingLite | ✅ 想定通り |
| `EverythingLite` インデックス再構築 | 同期ブロック ⚠️ | BG タイマ定期更新（Option C） |
| サムネボディ収集（EverythingProvider） | Everything IPC ✅ | そのまま |
| サムネボディ収集（EverythingLiteProvider） | 同期 Directory.Enumerate ⚠️ | Lite インデックス参照に変更 |
| ファイルシステムフォールバック | 同期 I/O ⚠️ | 呼び出し元を必ず `Task.Run` or `await ConfigureAwait(false)` に |

## 10. 総合優先ロードマップ

| 優先 | 作業 | 効果 | コスト |
|---|---|---|---|
| 🔥 最高 | EverythingLite BG タイマ再構築化（9-A Option C） | UI ブロック根絶 | 中 |
| 🔥 高 | EverythingProvider / Watcher 呼び出し元の Task.Run 確認 | UI フリーズ防止 | 小 |
| ⚡ 高 | EverythingLite の CollectThumbnailBodies → インデックス参照（9-C） | I/O 削減 | 小 |
| 💡 中 | ThumbFolder 直引き → `ResolveCurrentThumbnailOutPath` 統合（7-高優先） | コード整理 | 小 |
| 📝 低 | tabIndex → Thumb プロパティ対応表のドキュメント整備 | メンテ性向上 | 極小 |
