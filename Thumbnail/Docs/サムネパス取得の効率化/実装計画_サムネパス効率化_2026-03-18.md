## 提案: 3 フェーズ段階的改善

### フェーズ 1: UI 詰まり根絶（最優先）⚡

> [!IMPORTANT]
> ここがユーザー体感テンポに直結する最重要改善

#### 1-A. EverythingLite BG タイマ再構築化

**問題**: `QueryIndexedItems` 内の `RebuildIndexAsync().GetAwaiter().GetResult()` が呼び出しスレッドを秒単位でブロック。

**解決方針（Option C: BG タイマ方式）**:
- インデックス再構築を呼び出し時ではなく、**バックグラウンドタイマ**（5〜10秒間隔）で定期実行
- `CollectMoviePaths` は常に**既存キャッシュを即座に返す**（ブロックなし）
- 既存の `RebuildCooldown`（3秒）との整合は自然に取れる

##### [MODIFY] EverythingLiteProvider.cs
- `QueryIndexedItems` から `RebuildIndexAsync().GetAwaiter().GetResult()` を除去
- `StartBackgroundRebuildTimer()` / `StopBackgroundRebuildTimer()` を追加
- タイマのコールバック内で `Task.Run(() => RebuildIndexAsync(...))` を非同期実行
- `lock(SyncRoot)` は読み取り時のスナップショット取得のみに限定

#### 1-B. EverythingLite CollectThumbnailBodies の改善

##### [MODIFY] EverythingLiteProvider.cs
- `CollectThumbnailBodies` で `Directory.EnumerateFiles` を使っている部分を、内部インデックス（LiteDB）からの照合に変更
- Everything を活かした **ゼロ I/O サムネボディ収集** を実現

### フェーズ 2: サムネ存在チェック専用キャッシュ DB 構想 💾

> [!NOTE]
> 新 DB による「パスが存在するか？」の爆速判定。`Directory.EnumerateFiles` と `Path.Exists` をまとめて置き換える。

#### コンセプト: ThumbnailIndexDb（SQLite ベースの軽量キャッシュ）

```
テーブル: thumb_index
  - thumb_path TEXT PRIMARY KEY   -- サムネのフルパス
  - body       TEXT               -- ファイル名のbody部分
  - tab_index  INTEGER            -- タブ番号
  - updated_at TEXT               -- 最終確認日時(UTC)

テーブル: movie_index
  - movie_path TEXT PRIMARY KEY   -- 動画のフルパス（正規化済み）
  - movie_id   INTEGER
  - hash       TEXT
  - updated_at TEXT
```

**役割**:
- `BuildThumbnailFileNameLookup` を `SELECT body FROM thumb_index WHERE tab_index = ?` に置換
- `CreateMovieRecordFromDataRow` 単発追加時の `Path.Exists` × 12 を `SELECT 1 FROM thumb_index WHERE thumb_path = ?` に置換
- バックグラウンドで定期的に `Directory.EnumerateFiles` → DB 書き込みしてキャッシュを更新
- Everything IPC / EverythingLite からの結果もここに流し込む

**メリット**:
- UI スレッドからは常に **SQLite の SELECT のみ**（μs オーダー）
- サムネが増減してもバックグラウンド更新で追従
- 複数箇所の `Directory.EnumerateFiles` / `Path.Exists` を統一窓口に

**注意点**:
- WhiteBrowser の既存 DB (*.wb) は変更しない（プロジェクトルール遵守）
- ThumbnailIndexDb はアプリ専用キャッシュとして `%LOCALAPPDATA%\IndigoMovieManager_fork_workthree\` に配置

##### [NEW] Models/ThumbnailIndexDb.cs
- SQLite キャッシュ DB の CRUD 操作
- `RebuildFromFileSystem(string outPath, int tabIndex)` — BG 用全件再構築
- `TryGetDisplayPath(string moviePath, string hash, int tabIndex, out string path)` — UI 用即座クエリ
- `Upsert(string thumbPath, string body, int tabIndex)` — サムネ生成完了時の即時登録

##### [NEW] Services/ThumbnailIndexSyncService.cs
- バックグラウンドタイマで `RebuildFromFileSystem` を定期実行
- Everything 結果の流し込み統合

##### [MODIFY] MainWindow.xaml.cs
- `BuildThumbnailFileNameLookup` を `ThumbnailIndexDb.GetBodiesForTab(tabIndex)` に段階的に置換
- `CreateMovieRecordFromDataRow` の `Path.Exists` 判定を DB クエリに

## 「新 DB vs Everything vs 現状」の適材適所

| 用途 | 最適解 | 理由 |
|---|---|---|
| 動画パス収集（Watcher） | Everything IPC (既存) | 既に `Task.Run` 保護済み。爆速 |
| サムネ存在チェック（UI表示） | **ThumbnailIndexDb (新)** | μs で応答。UI スレッド安全 |
| サムネボディ突合（Rescue） | **ThumbnailIndexDb (新)** | BG で更新済みキャッシュから即判定 |
| Everything 不在時フォールバック | ファイルシステム走査 → **IndexDb 経由** | BG タイマで事前キャッシュ |
| 動画 identity 照合 | WhiteBrowser DB (既存 *.wb) | 正規データソース。変更なし |
