# 調査結果: 動画パスとサムネイルパス取得の実情と一元化方針 (2026-03-18)

最終更新日: 2026-03-18

変更概要:
- 動画パスとサムネイルパスの取得経路を横断調査
- 取得元、変換、表示用フォールバック、例外経路を分離して整理
- 一元化できる範囲と、分けて残すべき範囲を提案

## 1. 結論

- 動画パスの正規化ルールは一部まとまっているが、取得と照合は UI / Watcher / Queue / DB 照会に分散している。
- サムネイルパスは「ファイル名生成」だけはかなり一元化できているが、「出力先決定」と「表示時の既存ファイル探索」が複数箇所に散っている。
- いきなり全部を 1 クラスへ押し込むのは危険で、以下の 2 本立てに分けて寄せるのが現実的。
  - 動画パス: 比較・DB照会をまとめる
  - サムネイルパス: 出力先決定・既存ファイル探索・表示用フォールバックをまとめる
- Bookmark 用画像は命名規則も用途も別物なので、通常サムネイル経路とは無理に統合しない方がよい。

## 2. 現状整理

### 2.1 動画パスの主な取得元

1. DB 読み出し
   - `movie.movie_path` を一覧構築時に読む
   - 起動時軽量ロードでも同じく `movie_path` を読む
2. Watcher
   - `FileSystemEventArgs.FullPath` から新規動画として取り込む
3. UI 選択
   - `MovieRecords.Movie_Path` をプレイヤー、手動サムネ、削除、救済に使う
4. Queue / Failure / Rescue
   - `QueueObj.MovieFullPath`
   - `ThumbnailFailureRecord.MoviePath`
   - `MoviePathKey`

### 2.2 サムネイルパスの主な決定要素

1. サムネ根
   - DB の `system.thum`
   - 未設定時は `ThumbRootResolver.ResolveRuntimeThumbRoot(...)`
2. タブごとの出力先
   - `ThumbnailLayoutProfileResolver.Resolve(...)`
   - `FolderName = "{Width}x{Height}x{Columns}x{Rows}"`
3. ファイル名
   - `ThumbnailPathResolver.BuildThumbnailFileName(...)`
   - 規則は `動画名本体.#hash.jpg`
4. エラーマーカー
   - `動画名本体.#ERROR.jpg`
5. 表示時フォールバック
   - 現行命名: `movie_path` ベース
   - 旧命名: `movie_name` または `movie_body` ベース
   - 見つからなければ `Images/error*.jpg`

## 4. 散っている部分

### 4.1 動画パス側

#### A. DB から動画 identity を引く処理が複数ある

- `Thumbnail/MainWindow.ThumbnailCreation.cs`
  - `TryResolveMovieIdentityFromDb(...)`
- `Watcher/MainWindow.Watcher.cs`
  - `TryAppendMovieToViewByPathAsync(...)`
  - `BuildExistingMovieSnapshotByPath(...)`

問題:
- `lower(movie_path) = lower(...)` の SQL が複数箇所にある
- 文字列連結で SQL を組んでいる
- UI 一覧から探す経路と DB から探す経路が別々に実装されている

#### B. パス比較ルールが 2 系統ある

- Queue / Failure 系
  - `QueueDbPathResolver.NormalizePathForCompare(...)`
- UI / Watcher 系
  - `StringComparison.OrdinalIgnoreCase`
  - `HashSet<string>(StringComparer.OrdinalIgnoreCase)`

問題:
- `\\?\` 接頭辞除去や `/` と `\` の吸収は Queue 系だけ
- 同一実体でも UI 側では別物扱いになる余地がある

### 4.2 サムネイルパス側

#### A. 出力先決定ロジックが複数実装

- `Thumbnail/MainWindow.ThumbnailPaths.cs`
  - `ResolveThumbnailOutPath(...)`
- `src/IndigoMovieManager.Thumbnail.Engine/ThumbnailMovieMetaResolver.cs`
  - `ResolveThumbnailOutPath(...)`
- `src/IndigoMovieManager.Thumbnail.RescueWorker/RescueWorkerApplication.cs`
  - `ResolveOutPath(...)`

問題:
- 同じ責務が app / engine / worker に重複
- 詳細タブモードや既定 thumb root の変更時にズレやすい

#### B. 表示用の既存サムネ探索が複数実装

- `Views/Main/MainWindow.xaml.cs`
  - `CreateMovieRecordFromDataRow(...)`
  - `ResolveThumbnailDisplayPath(...)`
- `Views/Main/MainWindow.Startup.cs`
  - `CreateStartupMovieRecordFromSource(...)`
- `BottomTabs/Extension/MainWindow.BottomTab.Extension.DetailThumbnail.cs`
  - `ResolveExistingExtensionDetailThumbnailPath(...)`
- `Watcher/MainWindow.ThumbnailFailedTab.cs`
  - `TryGetExistingThumbnailErrorMarkerPath(...)`

問題:
- 現行命名 / 旧命名 / プレースホルダ / ERROR マーカーの扱いが各所で少しずつ違う
- 一覧構築と詳細表示で判断基準が揃いにくい

#### C. リネーム・削除も個別処理

- `Views/Main/MainWindow.xaml.cs`
  - `RenameThumb(...)`
- `Views/Main/MainWindow.MenuActions.cs`
  - 動画削除時のサムネ削除

問題:
- ファイル名パターンを直接組んで全フォルダ走査している
- `ThumbnailPathResolver` の命名規則は使っているが、列挙と rename ルールは共通化されていない

#### D. Bookmark は独立ルール

- `BottomTabs/Bookmark/MainWindow.BottomTab.Bookmark.cs`

特徴:
- 通常サムネとは別フォルダ
- `movie_path` に疑似的な bookmark ファイル名を持たせている
- `ThumbDetail` のみ使う

判断:
- これは通常サムネとは別ドメインとして残す方が安全

## 5. 一元化の提案

### 5.1 動画パス用: `MoviePathService` 相当を作る

責務:
- パス正規化
- 比較用キー生成
- 同一動画判定
- `dbFullPath + moviePath` から `movie_id/hash` を引く

最低限持たせたい API:

```csharp
string Normalize(string path);
string CreateKey(string path);
bool IsSamePath(string left, string right);
bool TryGetMovieIdentity(string dbFullPath, string moviePath, out long movieId, out string hash);
```

狙い:
- `QueueDbPathResolver.NormalizePathForCompare(...)` を UI/Watcher 側でも使える形に寄せる
- `TryResolveMovieIdentityFromDb(...)` と Watcher 側 SQL を 1 か所へ寄せる

### 5.2 サムネイル用: `ThumbnailLocationService` 相当を作る

責務:
- thumb root 解決
- tabIndex から out path 解決
- 正常サムネ / ERROR マーカー / 旧命名候補の列挙
- 表示用の既存ファイル探索

最低限持たせたい API:

```csharp
string ResolveOutPath(string dbFullPath, string dbName, string thumbFolder, int tabIndex);
string BuildSuccessPath(string outPath, string moviePath, string hash);
string BuildErrorMarkerPath(string outPath, string moviePath);
bool TryResolveExistingDisplayPath(
    string outPath,
    string moviePath,
    string movieName,
    string hash,
    out string path);
```

狙い:
- `MainWindow.ThumbnailPaths`
- `ThumbnailMovieMetaResolver.ResolveThumbnailOutPath`
- RescueWorker の `ResolveOutPath`
- 一覧生成と詳細表示の探索ロジック

これらを同じ判断基準に揃える。

## 6. 実装順の提案

1. `MoviePath` の DB 照会を共通化する
   - 効果が大きい
   - 既存 UI 影響が小さい
2. `ResolveOutPath` を共通化する
   - app / engine / rescue worker の重複を減らせる
3. `TryResolveExistingDisplayPath` を共通化する
   - `MainWindow.xaml.cs` と `MainWindow.Startup.cs` の重複を潰せる
4. rename/delete 用に「対象動画の全サムネ候補列挙」を共通化する
   - 最後でよい
   - 影響範囲が広い

## 7. 一元化しない方がよい部分

- Bookmark サムネ
  - 通常サムネと命名規則も用途も違う
- WPF の `MovieRecords.ThumbPath*` 反映そのもの
  - これは UI モデル更新責務なので、path 解決サービスとは分けた方が読みやすい

## 8. 具体的な着手候補

### 8.1 小さく安全な第一歩

- `Thumbnail/MainWindow.ThumbnailCreation.cs` の `TryResolveMovieIdentityFromDb(...)` を外出し
- `Watcher/MainWindow.Watcher.cs` の path 指定 SQL を同じ helper に寄せる

### 8.2 次の一手

- `Views/Main/MainWindow.xaml.cs`
- `Views/Main/MainWindow.Startup.cs`

この 2 か所で共通に使う `TryResolveExistingDisplayPath(...)` を作る

### 8.3 その次

- `BottomTabs/Extension/MainWindow.BottomTab.Extension.DetailThumbnail.cs`
- `Watcher/MainWindow.ThumbnailFailedTab.cs`

ここも同じ helper へ寄せる

## 9. 調査で見えた判断

- 「動画パス」と「サムネイルパス」を 1 つの巨大サービスにまとめるのは悪手
- まとめるべき単位は次の 2 つ
  - 動画 identity
  - サムネイル location
- 現状のコードは完全に散乱しているわけではなく、基礎部品は既にある
- なので全置換より、既存部品の上に薄い共通 helper を足して呼び出し側を順次寄せるのが最短
