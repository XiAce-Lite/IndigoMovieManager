# 動画別サムネ処理追跡手法 2026-03-20

## 1. 目的

- 特定動画 1 本について、通常レーンから engine 実行、必要なら rescue worker までを時系列で追跡する。
- `debug-runtime.log` を肥大化させず、調査対象動画だけを専用ログへ分離する。
- 再調査時に、誰が見ても同じ手順で同じログを拾える状態にする。

## 2. 適用範囲

- `Debug` 実行時のみ
- 対象はサムネイル生成系
  - 通常レーン
  - rescue 要求
  - rescue worker
  - isolated child attempt

## 3. 使用する環境変数

### 3.1 最小構成

- `IMM_THUMB_MOVIE_TRACE=1`
- `IMM_THUMB_MOVIE_TRACE_FILTER=<対象動画名またはフルパス>`

例

```text
IMM_THUMB_MOVIE_TRACE=1
IMM_THUMB_MOVIE_TRACE_FILTER=sango72gb.mkv
```

### 3.2 任意

- `IMM_THUMB_MOVIE_TRACE_LOG_DIR=<出力先ディレクトリ>`

未指定時は既定のログディレクトリを使う。

## 4. 既定のログ出力先

- `%LOCALAPPDATA%\IndigoMovieManager_fork_workthree\logs\`

主に見るファイル

- `thumbnail-movie-trace.ndjson`
- `thumbnail-create-process.csv`
- `debug-runtime.log`
- 必要時だけ `thumbnail-rescue-trace.csv`

## 5. 基本手順

### 5.1 事前設定

- `Debug` プロファイルへ対象動画の環境変数を設定する
- 複数対象を追う時は `;` 区切りで並べる

例

```text
IMM_THUMB_MOVIE_TRACE=1
IMM_THUMB_MOVIE_TRACE_FILTER=sango72gb.mkv;another.mkv
```

### 5.2 実行

- アプリを `Debug` 起動する
- 対象動画のサムネイル処理を流す
- 必要なら missing thumb や rescue 導線も踏む

### 5.3 実行後の確認

- `thumbnail-movie-trace.ndjson` で対象動画の `trace_id` ごとの流れを見る
- `thumbnail-create-process.csv` で最終成功/失敗を確認する
- `debug-runtime.log` で周辺状況を補う

## 6. `thumbnail-movie-trace.ndjson` の読み方

### 6.1 主要項目

- `trace_id`
  - 1 回の生成束を識別する
- `source`
  - `main`
  - `engine`
  - `rescue-worker`
- `phase`
  - どの段階かを示す
- `result`
  - `started`
  - `ready`
  - `success`
  - `failed`
  - `retry`
  - `skipped`
- `engine`
  - 実行 engine
- `output_path`
  - 出力先 jpg
- `process_engine_id`
  - 最終結果として採用された engine

### 6.2 通常レーンで期待する代表フェーズ

- `main_create_dispatch`
- `workflow_prepared`
- `duration_resolved`
- `context_built`
- `engine_order_built`
- `engine_selected`
- `engine_retry_scheduled`
- `engine_succeeded`
- `engine_failed`
- `finalized`
- `main_create_succeeded`
- `main_create_failed`

### 6.3 rescue で期待する代表フェーズ

- `rescue_request_enqueued`
- worker 側の `action` / `phase` を引き継いだ行
- `trace_id` が同じなら通常レーンから追って連結できる

## 7. 実運用での抽出コマンド

PowerShell 7 系を前提とする。

### 7.1 対象動画だけをトレースから抽出

```powershell
$logDir = Join-Path $env:LOCALAPPDATA 'IndigoMovieManager_fork_workthree\logs'
Get-Content -Path (Join-Path $logDir 'thumbnail-movie-trace.ndjson') |
  Select-String 'sango72GB.mkv'
```

### 7.2 JSON として読む

```powershell
$logDir = Join-Path $env:LOCALAPPDATA 'IndigoMovieManager_fork_workthree\logs'
Get-Content -Path (Join-Path $logDir 'thumbnail-movie-trace.ndjson') |
  ForEach-Object { $_ | ConvertFrom-Json } |
  Where-Object { $_.movie_path -like '*sango72GB.mkv' } |
  Sort-Object ts_utc
```

### 7.3 最終結果を process log で確認

```powershell
$logDir = Join-Path $env:LOCALAPPDATA 'IndigoMovieManager_fork_workthree\logs'
Get-Content -Path (Join-Path $logDir 'thumbnail-create-process.csv') |
  Select-String 'sango72GB.mkv'
```

## 8. 判定の仕方

### 8.1 通常成功

- `thumbnail-movie-trace.ndjson`
  - `engine_succeeded`
  - `finalized`
  - `main_create_succeeded`
- `thumbnail-create-process.csv`
  - 対象行が `success`

### 8.2 途中失敗

- `engine_failed`
- `main_create_failed`
- `thumbnail-create-process.csv` の `failed`

### 8.3 rescue へ流れたか

- `rescue_request_enqueued`
- `thumbnail-rescue-trace.csv` の対象行
- 同じ対象動画で worker 側の trace 行が出る

## 9. 今回の設計上の前提

- 既定では完全無効
- 対象動画を絞って使う
- `debug-runtime.log` は要約用のまま維持する
- 詳細は `thumbnail-movie-trace.ndjson` に逃がす

## 10. 注意点

- `Release` ではこの追跡は出ない
- `trace_id` は 1 回の生成束単位なので、同じ動画でもタブごとや再試行ごとに変わる
- 動画全体を束で見たい時は `movie_path` でまとめて読む
- rescue の流れは `ExtraJson` ベースなので、既存 failure record を経由する

## 11. 推奨運用

- まず `IMM_THUMB_MOVIE_TRACE_FILTER` をファイル名 1 本に絞る
- 調査後は環境変数を外す
- 恒常監視には使わず、難動画や回帰調査専用に使う
