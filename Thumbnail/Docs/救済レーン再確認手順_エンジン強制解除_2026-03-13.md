# 救済レーン再確認手順 エンジン強制解除 2026-03-13

最終更新日: 2026-03-14

## 1. 目的

- 外部救済 worker の先頭試行が `ffmpeg1pass` から始まることを再確認する。
- 親プロセス側の `IMM_THUMB_ENGINE` 強制が残っていても、救済 worker の試行順が壊れていないことを切り分ける。

## 2. 背景

- 現在の救済は `FailureDb -> 外部救済 worker` へ完全分離されている。
- 救済 worker は自分のプロセス内で `IMM_THUMB_ENGINE` を試行ごとに上書きし、`ffmpeg1pass -> ffmediatoolkit -> autogen -> opencv` の順で回す。
- そのため親プロセスや `launchSettings.json` の `IMM_THUMB_ENGINE=autogen` は、通常系には影響し得るが、救済 worker の固定順をそのまま壊す設計ではない。
- 2026-03-14 以降は、worker の stdout / stderr が `thumbnail-rescue-worker` へ転送されるので、実動画で順序を追いやすくなった。

## 3. 事前確認

- 対象ファイル: `C:\Users\na6ce\source\repos\IndigoMovieManager_fork_workthree\Properties\launchSettings.json`
- 確認点:
  - `IMM_THUMB_ENGINE` が `autogen` になっていないか
  - Visual Studio / デバッグ起動プロファイルで同値が流し込まれていないか
- 補足:
  - 親プロセス側で `autogen` 強制が残っていても、救済 worker 側は試行ごとに上書きする
  - ただし「本当にそう動いているか」は実ログで確認する

## 4. 再確認手順

1. アプリを起動する。
2. 対象動画を 1 本だけ選ぶ。
3. 明示救済要求、または通常失敗からの `pending_rescue` を発生させる。
4. 次のログを確認する。
   - `thumbnail-rescue-request`
   - `thumbnail-rescue-worker`
5. `thumbnail-rescue-worker` で次の順を確認する。
   - `rescue worker launched`
   - `rescue worker stdout: rescue leased`
   - `rescue worker stdout: engine attempt start: ... engine=ffmpeg1pass`
   - 失敗時だけ後続の `ffmediatoolkit -> autogen -> opencv`

## 5. 合格条件

- 救済 worker の最初の `engine attempt start` が `ffmpeg1pass` になる
- 後続エンジンへの移行は失敗時だけ起きる
- 親プロセス側で `autogen` が選ばれていても、救済 worker 側の先頭順は壊れない

## 6. 失敗時の見方

- まだ先頭が `autogen` なら、次を疑う。
  - worker 側の試行順定数が崩れている
  - worker ログ転送が欠けており、親プロセス側ログだけを見ている
  - 実際には通常系ログを見ていて、worker 側ログと混同している
