# 設計メモ: 管理者権限テレメトリ劣化ログ方針（2026-03-07）

## 1. 目的
- `AdminTelemetry` 連携で発生する未接続、権限不足、タイムアウトを、同じ `fallback` 文字列へ潰さず区別して記録する。
- `LTH-023A` で扱う FileIndexProvider 側の可用性異常とは混ぜず、まずは管理者権限テレメトリ経路だけの分類を固定する。

## 2. 結論
- ログの主分類は `fallback_kind` とし、値は `none` / `unavailable` / `access-denied` / `timeout` に固定する。
- 詳細は `fallback_reason` に分離し、`capabilities`、`system-load`、`disk-thermal:UnauthorizedAccessException` のように「失敗した操作 + 例外型」を残す。
- `mode=internal-only` は運転モードであり、失敗種別そのものではない。失敗種別は `fallback_kind` で判定する。

## 3. 記録ルール
- 全体フォールバック
  - `admin_fallback_kind`
  - `admin_fallback_reason`
- Disk温度 signal 劣化
  - `admin_disk_thermal_fallback_kind`
  - `admin_disk_thermal_fallback_reason`
- UsnMft signal 劣化
  - `admin_usnmft_fallback_kind`
  - `admin_usnmft_fallback_reason`

## 4. 判定ルール
- `no-client`、接続不能、未実装、未対応
  - `unavailable`
- `UnauthorizedAccessException`、Win32 error 5、`access denied` を含む失敗
  - `access-denied`
- request timeout / health check timeout
  - `timeout`
- 正常取得
  - `none`

## 5. 境界
- `DiskThermal` と `UsnMft` の個別 signal が失敗しても、`SystemLoad` が取得できていれば `mode=service` を維持する。
- その場合は signal ごとの `*_fallback_kind` だけを更新し、全体 `admin_fallback_kind` は `none` のままにする。
- FileIndexProvider の `AdminRequired` や `availability_error:*` はこのログ軸へ混ぜない。

## 6. 受け入れ基準
- `unavailable` / `access-denied` / `timeout` がログ上で機械判定できる。
- `mode=internal-only` と失敗種別が分離されている。
- Disk温度と UsnMft の個別劣化が、全体フォールバックとは別に追跡できる。
