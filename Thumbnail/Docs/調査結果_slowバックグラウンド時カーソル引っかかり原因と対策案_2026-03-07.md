# 調査結果: slowバックグラウンド時カーソル引っかかり原因と対策案（2026-03-07）

## 1. 結論
- 現行の `slow` は「真のバックグラウンド実行」ではなく、「2並列のままポーリング間隔とバッチ間待機だけを少し増やしたモード」である。
- そのため、サムネイル生成中の CPU 使用、JPEG保存、DB更新、UI反映が通常優先度のまま走り、マウスカーソルやウィンドウ操作が引っかかる余地が残っている。
- 特に `slow` で体感が重い主因は、`slow` の負荷抑制が弱いことに加え、進捗UI更新と `Dispatcher.InvokeAsync` によるUIスレッド作業が処理完了ごとに発生している点である。

## 2. 実装確認で見えた原因

### 2.1 `slow` でも実効並列は2本
- `ThumbnailThreadPresetResolver.ResolveParallelism` は `slow` を `HardMinParallelism = 2` へ解決している。
- 「1スレッド相当」というUI表現に対して、実装は2並列で動く。

### 2.2 `slow` の追加抑制は弱い
- `slow` 固有の制御は以下のみ。
  - Queueポーリング間隔を2倍
  - バッチ完了後に750ms待機
- これは「処理していない間の待機」を増やしているだけで、実際に処理している最中の CPU / I/O / UI 負荷は下がっていない。

### 2.3 処理スレッド優先度の低下が無い
- `ThreadPriority` や `ProcessPriorityClass`、Windowsの background mode 相当の制御が実装されていない。
- つまりデコード、画像変換、JPEG保存は通常優先度でCPU時間を取りに行く。

### 2.4 UIスレッドへ仕事を投げ過ぎている
- ジョブ開始、完了、進捗更新、保存完了のたびに `RequestThumbnailProgressSnapshotRefresh()` が呼ばれている。
- 進捗反映は coalesce されているが、保存完了や状態変化が続くと `DispatcherPriority.Background` キューが継続的に埋まる。
- さらにジョブ完了後、各動画について `Dispatcher.InvokeAsync` で MovieRecs 更新と長さ補正を行っている。
- これらは「サムネ生成自体はバックグラウンド」でも、UIスレッドに継続的な仕事を発生させる。

### 2.5 画像保存・I/Oも通常のまま
- JPEG保存は `SemaphoreSlim` で保存並列数を絞っているが、優先度制御やI/O優先度低下は無い。
- そのため遅いストレージや大きいファイル群では、ExplorerやWPF描画側のI/O競合が起こり得る。

## 3. 根拠コード
- `slow` が2並列: `Thumbnail/ThumbnailThreadPresetResolver.cs`
- `slow` の追加抑制がポーリングとクールダウンのみ: `Thumbnail/ThumbnailThreadPresetResolver.cs`
- バッチ処理中はそのまま並列実行: `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailQueueProcessor.cs`
- ジョブ開始/完了/保存で進捗UI更新要求を出している: `Thumbnail/MainWindow.ThumbnailCreation.cs`
- 進捗UIを `DispatcherPriority.Background` で流している: `MainWindow.xaml.cs`
- 保存後に `Dispatcher.InvokeAsync` で MovieRecs を更新している: `Thumbnail/MainWindow.ThumbnailCreation.cs`
- 進捗ミニパネルは `PreviewRevision` 変化ごとに再評価される: `MainWindow.xaml`, `Converter/ThumbnailProgressPreviewConverter.cs`

## 4. 対策案

### 4.1 最優先: `slow` を本当に軽くする
- `slow` の実効並列を 2 ではなく 1 にする。
- ただし救済レーン確保が必要なら、「通常処理1 + 需要時だけRecovery/Slowを単発起動」に分ける。
- 少なくとも常時2本はやめる。

### 4.2 推奨: `idle` モードを新設する
- プリセットとして `idle` 実行モードを追加する。
- 位置づけは `slow` よりさらに体感優先。
- 想定仕様:
  - 通常レーン常時1本
  - Recovery/巨大動画は同時実行しない
  - 進捗UI更新を間引く
  - プレビュー生成を省略または遅延
  - MovieRecsの即時反映をバッチ化
  - Windows側の `Idle` 優先度相当を使う

### 4.3 Windowsの優先度を下げる
- 候補:
  - `slow`: サムネ生成ワーカーの `BelowNormal`
  - `idle`: サムネ生成ワーカーの `Idle` / `Lowest`
  - 可能なら process / thread の background mode を適用
- 注意:
  - UIスレッドへは適用しない
  - Queue本体ではなく、実際のデコード・保存処理スコープに限定する

### 4.4 UI更新を間引く
- `RequestThumbnailProgressSnapshotRefresh()` を毎イベント即要求ではなく、最短反映間隔を設ける。
- 例:
  - `idle`/`slow` 時は 250ms〜500ms に1回まで
  - 完了画像の反映は1件ごとではなく、一定間隔でまとめる
- `Dispatcher.InvokeAsync` での MovieRecs 更新も、1件ごとではなくキュー化してまとめて反映する。

### 4.5 `idle` 時はプレビューを切る
- ミニパネル用 `WriteableBitmap` 化と `PreviewRevision` 更新を止める。
- 背景実行中は「ファイル名と状態だけ」に落とすと、UI再描画コストをかなり減らせる。

### 4.6 I/O負荷をさらに落とす
- `idle` 時はバッチ間待機を750msより長くする。
- 保存並列 (`IMM_THUMB_JPEG_SAVE_PARALLEL`) を1へ落とす。
- DB更新やログ書き込みも、可能ならまとめて行う。

## 5. おすすめ実装順
1. `slow` の実効並列を見直す。まず2固定をやめる。
2. `slow/idle` 時のUI更新間引きを入れる。
3. `idle` 時はプレビュー生成を止める。
4. その上でWindows優先度低下を追加する。

## 6. 推奨方針
- 既存 `slow` をそのまま強化するより、意味を分けた方が運用が分かりやすい。
- 推奨は以下。
  - `slow`: 今より少し軽いが、進捗や救済レーンは維持する。優先度候補は `BelowNormal`
  - `idle`: 体感優先。プレビュー最小、UI更新最小、優先度 `Idle`、通常1本
- ユーザーが「触っていて邪魔にならない」を求めるなら、本命は `idle` モード追加である。

## 7. 次の実装候補
- 案A: 既存 `slow` を本当に軽くする最小修正
  - 実効並列1
  - UI更新間引き
  - プレビュー無効化
- 案B: `idle` モード新設
  - 既存 `slow` は互換維持
  - 新モードにだけ強い抑制を入れる
- 後方互換と説明のしやすさを考えると、案Bが最も安全。
