# 調査結果: サムネエンジン比較 `fork` 大粒度アーキ・リペア処理・並列管理の移植観点（2026-03-11）

最終更新日: 2026-03-19

## 1. 目的

この文書は、`IndigoMovieManager_fork\Thumbnail` 側で進んでいる

- 大粒度アーキテクチャ分離
- リペア処理分離
- 並列管理強化

を、`IndigoMovieManager_fork_workthree\Thumbnail` へ「速度を犠牲にせず移植するならどうするか」という観点で整理した比較資料である。

結論を先に言うと、`fork` の構成を丸ごと持ち込むのは重い。  
本線の速度を守るなら、`hot path` を崩さずに「判断ロジック」と「recovery 専用経路」から段階移植するのが安全である。

## 2. 先に結論

- `fork` の良さは、責務分離が進んでいて recovery と並列制御を拡張しやすいこと。
- `workthree` の良さは、通常動画の経路がまだ短く、速度劣化ポイントが少ないこと。
- 速度を落とさずに移植するなら、まず全部は移さない。
- 最初に持ち込むべきなのは以下の3つ。
  - 実行判断の集約 (`ThumbnailExecutionPolicy` 相当)
  - repair の分離 (`ThumbnailRepairWorkflowCoordinator` 相当)
  - 並列方針の分離 (`ThumbnailThreadPresetResolver` と lane 予約の一部)
- 後回しにすべきなのは以下。
  - Worker プロセス分離
  - IPC / 管理者テレメトリ
  - FailureDb の全面導入
  - SWF 系の分岐一式

## 3. 比較サマリ

### 3.1 構成の差

| 観点 | `fork` | `workthree` |
|---|---|---|
| `Thumbnail` 配下の `.cs` 数 | 82 | 37 |
| `src/IndigoMovieManager.Thumbnail.Queue` 配下の `.cs` 数 | 53 | 18 |
| `ThumbnailCreationService.cs` サイズ | 約 6.9 KB | 約 77.0 KB |
| `ThumbnailQueueProcessor.cs` サイズ | 約 123.4 KB | 約 48.9 KB |
| 方向性 | 責務を分割して coordinator 群へ寄せている | service / queue processor に責務が厚く残っている |

### 3.2 一言でいう違い

- `fork` は「変更しやすい構造」へ寄せている。
- `workthree` は「今の通常系を速く回す構造」に寄っている。
- したがって、`fork` の全移植は保守性には効くが、速度最優先の本線には載せ過ぎになりやすい。

## 4. 大粒度アーキの差

### 4.1 `fork` 側

`fork` では、`ThumbnailCreationService` は入口ラッパーに近く、主処理が以下へ分かれている。

- `ThumbnailCreationRuntime`
- `ThumbnailCreationFacade`
- `ThumbnailCreationOrchestrationCoordinator`
- `ThumbnailExecutionFlowCoordinator`
- `ThumbnailRepairExecutionCoordinator`
- `ThumbnailRepairWorkflowCoordinator`
- `ThumbnailResultCompletionCoordinator`
- `ThumbnailExecutionPolicy`

この構成により、

- preflight
- SWF 分岐
- repair 準備
- engine 実行
- forced repair 再実行
- result finalizer

が別責務として読める。

### 4.2 `workthree` 側

`workthree` では `ThumbnailCreationService` に以下がまとまっている。

- DRM 事前判定
- duration / bitrate / codec 解決
- engine 選択と fallback
- autogen retry
- placeholder 判定
- エラーマーカー生成
- JPEG 保存まわり

つまり今は「速く読むには1ファイルで追える」が、「条件追加が増えるほど肥大化しやすい」状態である。

### 4.3 速度観点での評価

- 通常動画の最短経路だけ見れば、`workthree` の方が分岐段数が少なく見える。
- ただし `fork` の coordinator 分離は、多くが pure な判断や組み立てであり、重いのは主に構造コストである。
- 実CPUコストよりも、回帰混入点と依存点が増えることの方がリスクになる。

## 5. リペア処理の差

### 5.1 `fork` 側

`fork` では repair 系が明確に分離されている。

- `Engines/IndexRepair/VideoIndexRepairService.cs`
- `ThumbnailRepairWorkflowCoordinator.cs`
- `ThumbnailRepairExecutionCoordinator.cs`
- `ThumbnailExecutionPolicy.cs`

主な特徴:

- recovery lane かどうかで事前 probe / repair を分けられる
- engine failure 後の forced repair 条件を policy 化している
- repair 一時ファイルの生成と cleanup の責務が分かれている
- index repair 対象拡張子を policy 側で一元管理している

### 5.2 `workthree` 側

`workthree` には明示的な repair coordinator 群が無い。

- `ThumbnailCreationService` に retry / fallback / placeholder が集まっている
- 難読動画対応を足すと、通常系と recovery 系の分岐が同じファイルへ増えていく
- 成功条件の一般化と、通常動画への副作用切り分けが難しくなる

### 5.3 速度を守るならどう移すか

repair は通常動画へ毎回効かせないことが重要である。  
そのため、次の条件でしか repair を起動しない構造にするのがよい。

- `AttemptCount > 0`
- repair 対象拡張子
- policy が「repair 候補」と判断した失敗だけ

この形なら、通常動画では追加コストはほぼ「軽い if 判定」で済む。

## 6. 並列管理の差

### 6.1 `fork` 側

`fork` の並列管理は `workthree` よりかなり進んでいる。

- `ThumbnailThreadPresetResolver.cs`
  - preset ごとの並列数
  - dynamic minimum
  - scale up 可否
  - queue poll 間隔
  - batch cooldown
- `ThumbnailLaneClassifier.cs`
  - `Priority / Normal / Slow`
- `ThumbnailQueueProcessor.cs`
  - Recovery 専用枠
  - Slow 専用枠
  - need-based lease 予約
  - lane ごとの gate
  - process priority 制御

### 6.2 `workthree` 側

`workthree` にも lane と queue processor はあるが、まだ簡素である。

- lane は実質 `Normal / Slow` 寄り
- queue processor は slow 抑制中心
- Recovery 専用枠の思想が弱い
- preset 解決専用クラスが無い
- process priority や worker role 分離は未導入

### 6.3 速度面のメリット

`fork` の並列管理を一部移すメリットは大きい。

- 巨大動画に通常動画が塞がれにくくなる
- recovery が飢餓しにくくなる
- preset の意味がコード上で固定される
- 先取り lease 過多や実行枠超過を抑えやすい

### 6.4 速度面のデメリット

ただし、全部入れると次の負担がある。

- queue processor の分岐増加
- ログ点増加
- 設定値の整合テスト増加
- `MainWindow` 側との接続変更

つまり、並列管理は「効く」が「一気に入れると重い」。

## 7. 移植のメリット

### 7.1 大粒度アーキ移植のメリット

- 難読動画条件を通常系から離して置ける
- recovery 専用修正を入れても通常動画の回帰を読みやすい
- `CreateThumbAsync` の責務が軽くなり、保守しやすい
- 将来の worker 分離や DLL 化へつなげやすい

### 7.2 repair 移植のメリット

- 「どの失敗で repair するか」を一般条件で固定しやすい
- 本線へ個別動画名依存の例外条件を入れずに済む
- repair の実行コストを recovery lane に閉じ込めやすい

### 7.3 並列管理移植のメリット

- 通常動画の体感速度を守りやすい
- 巨大動画や再試行で全体が鈍る事象を減らせる
- `slow` / 将来の `idle` の意味をコードで固定しやすい

## 8. 移植のデメリット

### 8.1 丸ごと移植のデメリット

- `workthree` の本線へ一度に入れるには変更面積が大きすぎる
- 現在の高速化施策と衝突しやすい
- UI、Watcher、Queue、Worker、FailureDb の複数箇所に波及する
- ベンチで差が出た時に、どの変更が原因か切り分けにくい

### 8.2 速度観点のデメリット

- hot path に不要な abstraction を増やすと、通常動画で利得がない
- logging / telemetry / failure recording を同期で入れると逆効果になる
- repair 判定を通常経路へ広く掛けると、動画数が多い時に初動が鈍る

### 8.3 チーム運用上のデメリット

- `fork` の概念を `workthree` へ持ち込むと、設計用語が一気に増える
- 移植途中は `old service logic` と `new coordinator logic` が共存しやすい
- テストの当たり先を整理しないと、変更コストだけ増える

## 9. 推奨方針

### 9.1 推奨: 段階移植

丸ごと移植ではなく、次の順で入れるのがよい。

1. `ThumbnailExecutionPolicy` 相当を導入する
2. repair を `ThumbnailRepairWorkflowCoordinator` 相当へ切り出す
3. `ThumbnailThreadPresetResolver` 相当を導入する
4. queue の Recovery / Slow 予約を入れる
5. 必要なら最後に FailureDb や worker 分離を検討する

### 9.2 今すぐ入れない方がよいもの

- `Worker\*ProcessManager.cs`
- `AdminTelemetry*`
- `FailureDb` 一式
- `Swf\*`
- `ThumbnailCoordinator*`
- health publish / IPC 外出し一式

これらは価値はあるが、「速度を犠牲にせず移植」という今回の目的には重い。

### 11.1 `sango72GB.mkv`

対象:

- `C:\Users\na6ce\Desktop\_読み取り困難ログ用\超巨大動画\72\sango72GB.mkv`

確認結果:

- `thumb-sec=10,30,60,120,300` は短時間成功
- `thumb-sec=600` は `180秒` timeout で未完了
- `thumb-sec=2159` は `autogen` / `ffmpeg1pass` / `opencv` とも短時間成功せず、既存ログでは `ffmediatoolkit` が `frame decode failed at sec=2159`
- 既存ベンチでは `autogen` が約 `18分57秒` で成功した記録あり

見立て:

- この動画は「巨大だから常に遅い」ではなく、秒位置依存が強い
- 本線へ重い処理順変更を一般化する根拠には弱い
- `thumb-sec` を短めに寄せるか、重い秒位置だけ rescue 側へ逃がす方が筋がよい

### 11.2 `60fps_scale_2x_prob-3(4)_scale_1x_prob-3(1).mov_NVEcqp16.mp4`

対象:

- `C:\Users\na6ce\Desktop\_読み取り困難ログ用\超巨大動画\234\60fps_scale_2x_prob-3(4)_scale_1x_prob-3(1).mov_NVEcqp16.mp4`

確認結果:

- `thumb-sec=10,30,60,120,300,600,1200,1800,2400` をすべて短時間成功
- 実行時間は概ね `8〜10秒台`、`1200` のみ約 `20秒`

見立て:

- 同じ「超巨大動画」でも、秒位置依存の強さは動画ごとにかなり違う
- したがって、本線の既定 engine 順や重い fallback 条件を「超巨大動画だから」で一括変更するのは荒い
- 本線で一般化するなら、まずはファイルサイズではなく「指定秒位置での進み方」や「短秒位置での成功可否」を見る方が安全である

- engine 順
- retry 条件
- repair 対象拡張子
- preset の意味

を `workthree` でも一箇所で定義する。

メリット:

- 実行フローは大きく変えずに、難読動画条件だけ一般化できる
- 通常動画の速度影響が最も小さい

### Phase 2: repair だけ分離する

対象:

- `Engines/IndexRepair/*`
- `ThumbnailRepairWorkflowCoordinator.cs`

狙い:

- repair の起動条件を recovery lane 限定にする
- 通常系に repair 実務を入れない

メリット:

- 難読動画救済を足しても通常動画の hot path を汚しにくい

### Phase 3: queue の枠予約を移す

対象:

- `ThumbnailLaneClassifier.cs`
- `ThumbnailQueueProcessor.cs` の Recovery / Slow 予約部

狙い:

- 通常動画を巨大動画から守る
- recovery の開始機会を保証する

注意:

- queue processor は差分が大きいので、丸コピーではなく該当ロジックだけ差し込む

### Phase 4: 観測と外出しを必要最小限で足す

対象候補:

- FailureDb
- worker role
- process priority

注意:

- ここは速さより運用観測の改善なので、最後でよい

## 11. 最終判断

### 11.1 採るべきもの

- policy 分離
- repair 分離
- lane / preset の判断分離
- queue の need-based 予約

### 11.2 採らない方がよいもの

- `fork` の構成をそのまま全部
- Worker / IPC / telemetry の全面導入
- FailureDb を含む重い観測基盤の同時導入

## 12. 結論

- `fork` の大粒度アーキ、repair、並列管理は方向として正しい。
- ただし `workthree` へ速度を落とさず持ち込むなら、構造を丸ごと移すのではなく、判断系から薄く移すべきである。
- 一番うまい移し方は、`ThumbnailCreationService` の hot path を保ったまま、
  - `policy`
  - `repair coordinator`
  - `preset / lane reservation`

  の順で分離すること。
- これなら通常動画のテンポを守りつつ、難読動画対応と並列制御だけを本線へ安全に持ち込める。
