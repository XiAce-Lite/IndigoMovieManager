# 調査結果: 低速Thread現状まとめ（2026-03-18）

## 1. 結論

- `低速Thread` は別workerや別エンジンではなく、`Slow lane` の進捗表示名である。
- `低速Thread` に入っても、実処理本体は通常キューと同じ `CreateThumbAsync(...)` を通る。
- 通常キューとの差は主に、`サイズ分類`、`同一bucket内の後ろ寄せ`、`並列抑制`、`FailureDb上の lane 記録` である。
- timeout は `低速Thread` だから無効になる実装ではない。ただし、`CancellationToken` を見ない同期処理に詰まると、見た目上 timeout が効かないことがある。

## 2. 低速Threadとは何か

- 進捗表示では、`Slow lane` に分類されたジョブだけ `低速Thread` と表示する。
- 表示名の切替は `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailProgressRuntime.cs` で行っている。
- `Thread n` と `低速Thread` は UI 上の見え方の違いであり、ここで別実装へ分岐しているわけではない。

## 3. 低速Threadへ入る条件

- lane 分類は `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailLaneClassifier.cs` で行う。
- `MovieSizeBytes >= ThumbnailSlowLaneMinGb` の時だけ `Slow` になる。
- コード上のフォールバック既定値は `3GB` だが、`Properties/Settings.settings` の既定値は `512GB` である。
- 実運用では、ユーザー設定の `ThumbnailSlowLaneMinGb` が優先される。

## 4. 通常キューとの差分

### 4.1 取得順

- lease 後の整列では、同一 `Priority` + 同一 `LeaseBucketRank` 内だけ `Normal` を先、`Slow` を後ろに寄せる。
- 並び替えは `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailLeaseAcquirer.cs` で行っている。
- これにより、巨大動画が先頭に貼り付いて通常動画の初動を潰すのを避けている。

### 4.2 並列抑制

- 実行時は `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailQueueBatchRunner.cs` で `Slow lane` だけ追加抑制を掛けている。
- 全体並列が `3` 以上の時だけ、`Slow` は `SemaphoreSlim(1, 1)` で 1 本ずつ流す。
- 逆に言うと、全体並列が `2` 以下の時は `Slow` 専用の追加抑制は掛からない。

### 4.3 失敗記録

- terminal failure は `src/IndigoMovieManager.Thumbnail.Queue/ThumbnailFailureRecorder.cs` で `normal` / `slow` の2値へ落として FailureDb に書く。
- `Slow lane` の失敗も rescue 対象に含まれる。

## 5. 変わらない部分

### 5.1 実処理入口

- `低速Thread` でも通常キューと同じく `Thumbnail/MainWindow.ThumbnailCreation.cs` の `CreateThumbAsync(queueObj, false, token)` を使う。
- `low speed 専用の CreateThumbAsync` や `low speed 専用 worker` はない。

### 5.2 timeout 判定

- 通常キューの timeout 有効判定は lane を見ていない。
- `manual` でなく、`disableNormalLaneTimeout` でもなければ timeout を掛ける。
- そのため、`低速Thread` だけ timeout を外す分岐は現状存在しない。

## 6. エンジン選択との関係

- エンジン選択は `Thumbnail/Engines/ThumbnailEngineRouter.cs` で行う。
- `Slow lane` だから `ffmpeg1pass` になる、という実装ではない。
- 現在の差分は次の通り。
  - 通常は `autogen` 先頭
  - `超巨大動画` 判定時は lane に関係なく `ffmpeg1pass` 先頭
- `超巨大動画` 判定の既定閾値は `IMM_THUMB_ULTRA_LARGE_FILE_GB=32` 相当である。

## 7. timeout が効いていないように見える理由

### 7.1 timeout の実体

- 通常キュー timeout は `Thumbnail/MainWindow.ThumbnailCreation.cs` で `CancellationTokenSource(normalLaneTimeout)` を作り、linked token を `CreateThumbAsync(...)` へ渡す形で実装している。
- つまり hard kill ではなく、基本は cooperative cancel である。

### 7.2 効きやすい場所

- `Thumbnail/Engines/FfmpegOnePassThumbnailGenerationEngine.cs` は cancel 時に `process.Kill(entireProcessTree: true)` まで入る。
- そのため、`ffmpeg1pass` の外部プロセス実行中まで入っていれば timeout は比較的効きやすい。

### 7.3 効きにくい場所

- engine 実行前の同期処理は token 非対応の箇所がある。
  - `Thumbnail/Adapters/AppVideoMetadataProvider.cs`
  - `src/IndigoMovieManager.Thumbnail.Engine/ThumbnailShellDurationResolver.cs`
  - `src/IndigoMovieManager.Thumbnail.Engine/ThumbnailPrecheckCoordinator.cs`
- ここで `MovieInfo` や Shell COM の取得が長く詰まると、10秒を過ぎても即座には抜けない。
- したがって、実機で「低速Threadなのに timeout が効かない」と見える時は、`Slow lane` 固有問題というより、`timeout の効く範囲が engine 実行以降寄り` である可能性が高い。

## 8. 2026-03-18 時点の理解

- `低速Thread` は「巨大動画を通常動画より少し後ろへ回し、必要時だけ実行数を絞る」ための lane である。
- `低速Thread` 自体は救済workerではない。
- `低速Thread` 自体は別timeout戦略でもない。
- 体感上の論点は今も `timeout 秒数` より、`timeout 前にどこで詰まっているか` の切り分けが重要である。
