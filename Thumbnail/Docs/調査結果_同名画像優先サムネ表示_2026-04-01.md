# 調査結果: 同名画像優先サムネ表示 (2026-04-01)

最終更新日: 2026-04-01

変更概要:
- WhiteBrowser 互換要望の「動画と同名の画像を表紙として優先表示」を要件化
- 現行の表示経路、生成経路、Watcher 経路の差し込み点を整理
- 「表示優先」と「生成スキップ」を分けて段階導入する方針を提案

## 1. 結論

- この要望は実現可能で、ユーザー満足度に対してコスパが高い。
- ただし要件は 2 段ある。
  - A. **表示優先**: 同名画像があれば、その画像パスを UI に出す
  - B. **生成スキップ**: 同名画像があれば、通常サムネ生成や ERROR マーカー作成も抑止する
- A は比較的安全に入れやすい。
- B までやるなら、Watcher の「未作成判定」も同名画像を成功扱いに寄せる必要がある。
- 本ブランチの最優先は体感テンポなので、**まず A を正本要件**として固め、B は第 2 段で入れるのが安全。

## 2. 背景

WhiteBrowser ユーザー文脈では、動画ファイルと同じ名前の画像ファイルを同じフォルダに置き、
それを一覧の表紙として見せたい需要が強い。

例:

- `俺の神映画.mp4`
- `俺の神映画.jpg`

この時、動画から切り出したランダムなフレームではなく、`俺の神映画.jpg` を一覧・詳細の表紙として優先表示したい。

## 3. 要件定義

### 3.1 正本要件（Phase 1: 表示優先）

1. 対象は「動画ファイルと同じフォルダにある、同じベース名の画像ファイル」とする。
2. 優先対象拡張子は、まず `jpg` / `jpeg` / `png` とする。
3. 拡張子比較は Windows 前提で大文字小文字を無視する。
4. 同名画像が存在し、かつ読み取り可能なら、上側タブと詳細表示ではその画像を優先表示する。
5. 同名画像が無い、壊れている、読めない場合は、現行どおり生成済みサムネイル jpg を使う。
6. 生成済みサムネイルも無い場合は、現行どおり placeholder を使う。
7. 同名画像は「表示専用」とし、DB へ新列追加はしない。
8. 同名画像は thumb フォルダへコピーしない。
9. 既存の手動サムネ生成、救済 worker、bookmark サムネの保存仕様は変えない。

### 3.2 推奨要件（Phase 2: 生成スキップ）

1. 同名画像が存在する動画は、通常サムネ生成キューへ積まない。
2. 同名画像が存在する動画には `.#ERROR.jpg` を作らない。
3. Watcher / 起動時未作成判定でも、同名画像を「既に表紙あり」と扱う。

### 3.3 非スコープ

- Bookmark 用画像ルールの統合
- 外部画像の自動リネーム
- 外部画像の自動削除
- WebP / AVIF など追加拡張子の初回対応
- DB スキーマ変更

## 4. 受け入れ条件

1. `movie.mp4` と `movie.jpg` が同居している時、Small / Big / Grid / List / Big10 / Detail で `movie.jpg` が表示される。
2. その状態で生成済みサムネイル jpg が存在していても、UI は同名画像を優先する。
3. 同名画像が削除された後に再読込すると、生成済みサムネイル jpg または placeholder に戻る。
4. 壊れた同名画像があっても UI が例外で落ちず、従来のフォールバックに戻る。
5. 起動時一括ロードと、Watcher/単発追加のどちらでも同じ表示規則になる。
6. 手動生成や救済成功直後でも、同名画像がある場合は generated jpg に見た目が上書きされない。

## 5. 現行実装の調査結果

### 5.1 一覧の初回表示

- `Views/Main/MainWindow.xaml.cs` の `CreateMovieRecordFromDataRow(...)` と `ResolveThumbnailDisplayPath(...)` は、
  **thumb 出力フォルダ内の生成 jpg** だけを見ている。
- `Views/Main/MainWindow.Startup.cs` の `CreateStartupMovieRecordFromSource(...)` も同じ判定を使っている。
- つまり現状では、動画の隣にある `movie.jpg` / `movie.png` は参照されない。

### 5.2 詳細表示

- `BottomTabs/Extension/MainWindow.BottomTab.Extension.DetailThumbnail.cs` の
  `ResolveExistingExtensionDetailThumbnailPath(...)` も、詳細タブ用 outPath 群の jpg 探索に閉じている。
- ここも同名画像優先の追加が必要。

### 5.3 生成成功後の UI 反映

- `Thumbnail/MainWindow.ThumbnailCreation.cs` は生成成功後、
  `Thumbnail/MainWindow.ThumbnailFailureSync.cs` の `TryApplyThumbnailPathToMovieRecord(...)` を通して
  `ThumbPath*` / `ThumbDetail` へ **生成 jpg のパスをそのまま入れる**。
- このため、初回表示で同名画像を出せても、後から生成成功が走ると generated jpg へ戻される。
- 実装時は「成功反映前に同名画像を再評価する」か、「UI 反映を preferred display path へ寄せる」必要がある。

### 5.4 生成スキップ / ERROR マーカー

- `Thumbnail/MainWindow.ThumbnailQueue.cs` の
  `ShouldCreateErrorMarkerForSkippedMovie(...)` は、
  `Thumbnail/ThumbnailPathResolver.cs` の `TryFindExistingSuccessThumbnailPath(...)` を使い、
  **thumb フォルダ内の成功 jpg** だけを既存成功として扱う。
- ここに同名画像は含まれていない。
- そのため Phase 2 をやるなら、この成功判定の意味を広げる必要がある。

### 5.5 Watcher の未作成判定

- `Watcher/EverythingLiteProvider.cs`
- `Watcher/EverythingProvider.cs`
- `Watcher/EverythingFolderSyncService.cs`

これらは `CollectThumbnailBodies(...)` / `TryCollectThumbnailBodies(...)` で
**thumb フォルダ内の jpg body 集合** だけを集めて差分判定に使っている。

つまり、

- 「同名画像があるから generated jpg を作らない」

まで踏み込むと、Watcher 側ではその動画を今後も「サムネ未作成」と誤認しうる。

## 6. 実装上の論点

### 6.1 速度面の注意

起動時全件ロードは、いま `BuildMovieRecordBulkBuildCache(...)` で
thumb フォルダの jpg 一覧を先に HashSet 化し、1件ずつの同期 I/O を減らしている。

ここへ単純に

- `Path.ChangeExtension(moviePath, ".jpg")`
- `File.Exists(...)`
- `Path.ChangeExtension(moviePath, ".png")`

を全件分入れると、数千件 DB で起動テンポを落とす危険がある。

したがって Phase 1 でも、起動時経路は次のどちらかが望ましい。

1. ディレクトリ単位の同名画像キャッシュを持つ
2. 初回は現状維持で出し、可視範囲だけ後追いで同名画像へ差し替える

本ブランチ方針を踏まえると、**naive な per-row `File.Exists` の直足しは避けるべき**。

### 6.2 UI 表示自体は可能

- `Infrastructure/Converter/NoLockImageConverter.cs` は渡されたファイルパスを
  `BitmapDecoder` / `BitmapImage` で開いている。
- このため UI 側は「thumb フォルダ配下の jpg だけ」しか表示できないわけではない。
- 少なくとも `jpg` / `jpeg` / `png` を直接指す実装は成立しやすい。

### 6.3 rename 契約は別論点

- `Watcher/MainWindow.WatcherRenameBridge.cs` は、UI が今握っている既存サムネパスの差し替えを行う。
- ただし対象は主に既存サムネ資産の rename 契約であり、動画フォルダ内の同名画像を自動追従させる責務とは少し違う。
- 初回スコープでは「再読込で追従する」を正とし、自動 rename 追従は別件に分けた方が安全。

## 7. 推奨実装方針

### 7.1 Phase 1: 表示優先

共通 helper を 1 つ追加し、少なくとも次を共通化する。

想定責務:

- 動画パスから同名画像候補を列挙する
- 優先拡張子順で最初の有効画像を返す
- 画像が無ければ従来の generated jpg 探索へフォールバックする

差し込み候補:

1. `Views/Main/MainWindow.xaml.cs`
   - `ResolveThumbnailDisplayPath(...)`
2. `Views/Main/MainWindow.Startup.cs`
   - 起動 feed の表示 path 決定
3. `BottomTabs/Extension/MainWindow.BottomTab.Extension.DetailThumbnail.cs`
   - `ResolveExistingExtensionDetailThumbnailPath(...)`
4. `Thumbnail/MainWindow.ThumbnailFailureSync.cs`
   - `TryApplyThumbnailPathToMovieRecord(...)` の前段
   - もしくは別メソッドで preferred display path を書く

### 7.2 Phase 2: 生成スキップ

次の広がりが必要になる。

1. `Thumbnail/MainWindow.ThumbnailQueue.cs`
   - enqueue 前に同名画像存在チェック
   - `ShouldCreateErrorMarkerForSkippedMovie(...)` の意味拡張
2. Watcher 系
   - `CollectThumbnailBodies(...)` 相当だけでは足りない
   - 「動画フォルダの同名画像」も未作成判定へ織り込む必要がある

この段階は表示優先より影響範囲が広い。

## 8. 最小実装案

最小で効果が高いのは次の形である。

1. 一覧/詳細表示の path 解決に「同名画像優先」を追加
2. 生成成功/救済成功で generated jpg を UI へ反映する直前に、同名画像があればそちらを維持
3. 生成キューや ERROR マーカーの契約は最初は変えない

この形なら:

- ユーザーが見たい見た目は先に満たせる
- 既存の rescue / marker / watcher 契約を大きく壊しにくい
- 体感テンポ悪化の原因も切り分けやすい

## 10. 実装判断

この要望は採る価値が高い。

ただし正しい切り方は、

- **先に表示優先**
- **後で生成スキップ**

である。

ここを混ぜると、Watcher の未作成判定と ERROR 契約まで一気に広がる。
本線の体感テンポ最優先方針を守るなら、まずは UI 表示の正しさを薄く載せるのがよい。

## 11. 2026-04-01 実装メモ

- 今回は B を「同名画像を管理サムネへ取り込む」方向で着手した。
- 実装の主筋は次の 2 点。
  1. engine の precheck で、同名 `jpg/jpeg/png` があれば通常デコード前に管理サムネ jpg へ取り込み成功扱いにする
  2. queue の 0KB 早期 reject は、同名画像がある時だけ precheck 取り込みへ通す
- これにより、通常生成エンジン実行と `#ERROR` マーカー作成をかなり手前で抑止できる。
- あわせて UI fallback も追加した。
  1. 一覧/起動時ロードの `ResolveThumbnailDisplayPath(...)` は、管理サムネ未発見時に同名画像を直接返す
  2. エクステンション詳細の既存サムネ探索でも、最後に同名画像を直接返す
- これにより、既存 DB に対して後から cover 画像を置いたケースでも、キュー取り込み完了を待たず次回表示更新で表紙が見える。
