# 絵文字パス問題 症状と対策のまとめ

## 1. 症状
- **動画読み込みの失敗:** ファイルやディレクトリのパスに絵文字などのUnicode特有文字（WindowsのANSI変換不可能な文字）が含まれていると、動画を正しく開けずサムネイル生成に失敗する。
- **画像保存の失敗:** サムネイル生成後の画像を保存する際、出力先パスに絵文字が含まれていると書き込みエラーや例外がスローされる。
- **大容量ファイルの処理遅延:** パス問題を回避するためにファイルを丸ごと安全なパスへコピーすると、3GBを超えるような大容量動画の場合、処理が著しく遅延しディスク容量も圧迫する。

## 2. 絵文字が読めないライブラリやシステム
- **OpenCV (`OpenCvSharp4` など):**
  - 内部のC/C++処理やOS連携においてマルチバイト文字体系（Windows環境のCodePage 932など）を前提とする箇所がある。そのため、絵文字を含むUnicodeパスを渡すと `VideoCapture`（動画オープン）や `Cv2.ImWrite`（画像保存）が文字化け・パス認識エラーを起こして失敗する。
- **古いシステム・一部の外部プロセス:**
  - 厳密な文字エンコーディングの制約環境全般（※`ffmpeg`は比較的絵文字パスに強いが確実とは限らない）。

### 2.1 調査結果：ffmpeg および .NET (C#) の安全性について
- **ffmpegへの引数渡し:**
  - `ProcessStartInfo.ArgumentList` を用いて引数を追加しているため、コマンドプロンプトの文字列結合によるエスケープ不備や文字コード制限（Shift_JISなど）を受けず、安全にUnicode文字列のままプロセスへパスが渡される。ffmpeg自体も絵文字を含むUTF-8/Unicodeパスを解釈できるため、引数渡しにおける問題は生じない。
- **.NETネイティブのファイル操作:**
  - .NET側の `System.IO` API（`File.Copy`、`File.Move`、`File.Delete`、`Directory.Exists` など）は、内部で Unicode（UTF-16）として Windows API のワイド文字版（`*W` 系関数）を直接呼び出す。OSのファイルシステム自体がUnicodeをフルサポートしているため、ファイルの移動・削除や存在チェックなど、.NET標準のファイルI/Oのレイヤーで絵文字パスによる文字化けやエラーは発生しない。

## 3. 対策
ファイルコピーという最終手段をなるべく避け、ネイティブなOS機能を利用した**「4段階の入力パス対策」**と**「保存時のフォールバック処理」**を採用している。

### 3.1 入力パスの4段階対策（OpenCV / ffmpeg共通）
1. **生パス (Raw):** 最初にそのままのパスで試行。
2. **短縮名 (ShortPath):** `GetShortPathName` APIを使用し、絵文字を含まない8.3形式の短いパスを取得して試行。
3. **別名 (Junction / HardLink):**
   - パスのうち、ディレクトリ部分に絵文字がある場合は、ASCII文字のみの一時フォルダから**ジャンクション (Directory Junction)** を作成してアクセス。パス生成（1,2失敗時のみ実行）。
4. **一時コピー (Copy):**
   - 最終手段として、安全なASCIIパスにファイルを丸ごとコピーする（ffmpegの最終フォールバック時のみ）。
   - **大容量対策:** 3GBを超える場合は処理をブロックさせないため、すぐにはコピーせずキューを「後回し（Deferred）」状態にし、既存処理の完了後、ユーザー確認を経てから実行する。

### 3.2 保存時のフォールバック処理
- **ANSI変換判定と安全な保存:**
  `Encoding.GetEncoding(932)` の厳密な例外フォールバックを用いて、パスに変換不可な文字（絵文字等）が含まれるか判定する。安全な場合はそのまま `Cv2.ImWrite` で保存。
- **一時ASCII保存と移動:**
  例外が発生した、または絵文字が含まれていると判定された場合は、**一時的なASCII名称のパスへ保存**する。その後、Unicodeパスへ対応している `.NET の File.Move` を活用して最終的な保存先へ移動させる。
  ※CodePage 932の判定を動作させるため、起動時に `Encoding.RegisterProvider(CodePagesEncodingProvider.Instance)` の登録を行っている。

### 3.3 クリーンアップとロギング
- **確実な削除:** 一時的に作成したジャンクションやコピーした一時ファイルは、処理終了後の `finally` ブロックで必ず削除（クリーンアップ）する。ジャンクション削除の際は、実体まで消さないよう注意する。
- **実行ログ:** どの段階のパス入力が試行・採用されたか、どの処理（OpenCV/ffmpeg/保存）で失敗したかをログ（`thumb opencv input selected` など）に明確に出力し、原因箇所を追跡可能にしている。
