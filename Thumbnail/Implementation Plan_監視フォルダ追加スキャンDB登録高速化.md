# Implementation Plan（監視フォルダ追加時のスキャン・DB登録高速化）

## 1. 目的
- 監視フォルダ追加時の「初回キュー投入までの待ち時間」と「全件登録完了時間」を短縮する。
- WhiteBrowser互換を維持し、既存DBスキーマは変更しない。
- UI表示件数が `20件以下` の場合は、1件ごとにDB登録・サムネイル作成投入・表示反映を行う。
- 段階導入し、各フェーズでロールバック可能にする。

## 2. 現状ベースライン（2026-02-24）
- 軽負荷ケース: `scanned=240, new=0, elapsed_ms=916`
- 重負荷ケース: `scanned=2034, new=2034, elapsed_ms=388714`（約6分29秒）
- 重負荷ケースの初回投入遅延: スキャン開始から初回100件投入まで `33.104秒`
- 重負荷ケースの実効スループット: `約5.23件/秒`
- QueueDB Upsertは追従しており、主な律速はスキャン後段（MovieInfo作成・movie登録・UI反映）

## 3. ボトルネック整理
- 1件ごとに `MovieInfo` を生成し、動画解析（OpenCV）とハッシュ計算を実行している。
- 1件ごとに `InsertMovieTable` が接続・採番・INSERTを行う。
- `InsertMovieTable` 末尾の `await Task.Delay(5)` が件数分積み上がる。
- 1件ごとに `select * from movie order by movie_id desc` を実行してUI反映している。
- フォルダ走査が拡張子ごと再帰列挙で、拡張子数分だけ探索コストが増える。

## 4. 目標値（受け入れ条件）
- 初回キュー投入遅延: `33秒 -> 5秒以内`
- 重負荷ケース総時間（2034件）: `388秒 -> 150秒以内`
- 実効スループット: `5.23件/秒 -> 15件/秒以上`
- 小規模ケース（新規20件以下）: 1件ずつUIに増え、同時にサムネイル作成が進行すること
- 既存機能退行なし
  - 重複登録しない
  - UIが固まらない
  - 監視・サムネイル作成が継続する

## 5. 処理モード方針（追加条件反映）
- 判定基準: `scanResult.NewMoviePaths.Count`
- Smallモード（`<= 20`）
  - 1件ごとに処理する（DB登録 -> サムネキュー投入 -> UI反映）
  - ただしUI反映は「全件SELECT」ではなく、対象1件だけを使って反映する
  - ユーザー体感を優先し、逐次表示を維持する
- Bulkモード（`>= 21`）
  - DB登録とUI反映をバッチ寄りにする
  - サムネキューは100件単位で投入し、走査中から並行生成させる
  - 1件ごとのUI再クエリは禁止する

## 6. 実装フェーズ

## Phase 1（低リスク・即効）
- [x] `CheckFolderAsync` にモード分岐を追加する（`<=20` はSmall、`>=21` はBulk）。
- [x] Smallモードは「1件ごと処理」を維持しつつ、UI反映を全件再クエリなしで実施する。
- [x] Bulkモードは「1件ごとのUI再クエリ/反映」を停止し、フォルダ単位でまとめて反映する。
- [x] `InsertMovieTable` の `await Task.Delay(5)` を削除する。
- [x] 計測ログを追加する（走査時間・MovieInfo生成時間・DB挿入時間・UI反映時間）。
- [x] 影響範囲を `MainWindow.Watcher.cs` と `DB/SQLite.cs` に限定する。

期待効果:
- Smallモードは体感を維持し、BulkモードはUIクエリ多発を除去して遅延を下げる。

## Phase 2（中リスク・DB挿入最適化）
- [ ] movie登録のバッチAPIを追加する（接続/トランザクションを件数分でなくバッチ単位へ）。
- [ ] 採番処理（`max(movie_id)`）を1件ごとではなくバッチ先頭1回へ集約する。
- [ ] 100件単位でINSERTし、失敗時は該当バッチのみ再試行する。
- [ ] 既存単件APIは残し、`Smallモード=単件API` / `Bulkモード=バッチAPI` を切り替える。

期待効果:
- DB書き込みオーバーヘッドを大幅に削減する。

## Phase 3（中リスク・走査/解析最適化）
- [ ] 走査を「拡張子ごとの再帰列挙」から「単一列挙 + 拡張子判定」へ変更する。
- [ ] `MovieInfo` 生成を限定並列化する（例: 2〜4並列、上限を設定値化）。
- [ ] 解析ワーカーとDB書き込みを分離し、DB側は単一ライターで順序制御する。

期待効果:
- ファイルI/Oと動画解析の並列度を適正化し、CPU待ち時間を圧縮する。

## Phase 4（オプション・高効果）
- [ ] 監視スキャン時は「軽量登録（ハッシュ/詳細メタ後回し）」モードを選択可能にする。
- [ ] 詳細メタ情報更新はバックグラウンド補完ジョブに分離する。
- [ ] 既定は互換優先（従来動作）にし、設定で有効化できるようにする。

期待効果:
- 大規模投入時の初動を最大化する。

## 9. ロールバック方針
- フェーズごとにコミットを分離し、切り戻し可能にする。
- バッチAPI導入後も単件APIは保持し、切替フラグで戻せる状態を維持する。
- オプション機能は既定OFFで導入する。

## 10. 同期方式の最終決定（2026-02-24）
- ローカルNTFSドライブ:
  - Everything SDK（Everythingインデックス）を使用した差分同期を使用する（高速経路）。
- NAS / 非NTFSドライブ:
  - 既存の `FileSystemWatcher + 起動時スキャン` を使用する（フォールバック経路）。
- 監視フォルダ単位で同期戦略を自動判定し、混在構成（NTFS + NAS）を許容する。

## 11. 実装方針（Everything SDK + フォールバック）
- 戦略インターフェース:
  - `IPathSyncStrategy`（`InitialScanAsync` / `CollectDeltaAsync`）を追加し、Everything版と既存版を差し替え可能にする。
- 経路判定:
  - `DriveInfo` とパス属性で「ローカル固定ドライブか」「NTFSか」「UNCか」を判定する。
  - 判定結果をログ出力し、想定外判定を追跡できるようにする。
- Everything版:
  - Everything SDKラッパー（例: `EverythingSearchClient` / `EverythingNet`）で、監視フォルダ配下の差分候補を取得する。
  - 設定は `Everything連携モード`（`OFF / AUTO / ON`）で制御する。
  - `AUTO` は実行時にEverything可用性を検知し、利用可能時のみ高速経路を有効化する。
  - `ON` は常にEverything経路を試行し、失敗時は通常監視へフォールバックする。
  - `OFF` はEverything経路を使用しない。
  - Everythingクエリ側で動画拡張子（`ext:`）を先に絞り、動画以外の大量ヒットを抑制する。
  - 検索結果が `TotalItems > NumItems`（上限打ち切り）なら不完全結果を採用せず、既存走査へフォールバックする。
  - `SearchClient` は呼び出しごとに生成し、インスタンス共有による競合を避ける。
  - SDKに変更プッシュ通知はないため、短周期ポーリング（例: 1〜3秒）で差分を回収する。
  - 差分結果を `Added/Updated/Deleted` の共通DTOへ変換して既存DB反映へ渡す。
  - 初回同期基準時刻を保持し、`recently changed` 系条件で増分同期する。
- 既存版（フォールバック）:
  - 現行の `FileSystemWatcher + 起動時スキャン` をそのまま利用する。
  - 例外時は監視を止めず、指数バックオフで再開する。
- 自動フォールバック:
  - Everything未導入、Everythingサービス停止、SDK問い合わせ失敗時は即時に既存版へ切替える。
  - クエリ打ち切り検知（`everything_result_truncated`）時も即時に既存版へ切替える。
  - 切替理由をUIログに1行表示し、原因調査可能にする。
- 実行制御:
  - `CheckFolderAsync` は単一実行とし、再要求は後続1回へ圧縮（coalescing）して多重起動を防ぐ。
  - 圧縮時は `Manual > Watch > Auto` の優先度で強い要求を採用する。

## 12. 追加受け入れ条件（Everything導入分）
- [ ] NTFS監視フォルダでは初回同期後の差分反映が `FileSystemWatcher` 単独時より短時間で完了する。
- [ ] NAS/非NTFS監視フォルダでは既存挙動（監視・起動時スキャン）が退行しない。
- [ ] NTFSとNASの混在時に、フォルダごとに適切な戦略で動作する。
- [ ] Everything利用不可時に自動フォールバックし、監視自体は継続する。
- [ ] Everything結果が打ち切られた場合（`TotalItems > NumItems`）は、不完全結果を採用せず既存走査へフォールバックする。
- [ ] `CheckFolderAsync` 多重要求時に同時実行されず、要求が圧縮処理される。
- [ ] 削除イベントを含む差分反映でDB整合性が維持される（重複登録なし）。
