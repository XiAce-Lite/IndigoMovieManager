# サムネイルキュー 手動再試行運用手順（Failed -> Pending）

## 1. 目的
- `Failed` で停止したサムネイルジョブを、再実行対象（`Pending`）へ戻す。
- 再起動せずに現在のConsumerループで再消化させる。

## 2. 前提
- 対象DBがアプリで開かれていること。
- 失敗原因（ファイル消失、権限不足、容量不足など）を先に解消していること。

## 3. 推奨手順（アプリ内メソッド利用）
1. 対象DBを開いた状態で、`MainWindow.ResetFailedThumbnailJobsForCurrentDb()` を実行する。
2. 戻り値 `resetCount` が `1` 以上なら、`Failed -> Pending` への戻しが完了。
3. Consumerが次ポーリングで `Pending` をリースし、再処理を開始する。
4. ログ `queue-ops` で実行結果を確認する。

## 4. SQLで直接戻す場合（運用補助手段）
```sql
UPDATE ThumbnailQueue
SET
    Status = 0,
    AttemptCount = 0,
    LastError = '',
    OwnerInstanceId = '',
    LeaseUntilUtc = '',
    UpdatedAtUtc = strftime('%Y-%m-%dT%H:%M:%fZ','now')
WHERE MainDbPathHash = @MainDbPathHash
  AND Status = 3;
```

## 5. 確認ポイント
- `queue-consumer` ログにリース取得件数が出ていること。
- `queue-consumer` / `thumb queue summary` で失敗件数が増え続けていないこと。
- 再実行後に同一ジョブが `Done` へ遷移すること。

## 6. サムネイルjpg削除時の自動復旧確認（Watch運用）
1. 対象DBを開き、対象動画が表示されていることを確認する。
2. 現在タブの対象サムネイルjpgを手動で削除する。
3. Everything監視ポーリング（既定3秒）を待つ。
4. `watch-check` ログに以下のいずれかが出ることを確認する。
   - `missing-thumb rescue start`
   - `missing-thumb rescue throttled`（直近60秒以内で再実行抑止）
   - `missing-thumb rescue skipped: queue busy`（高負荷時の抑止）
5. 低負荷状態で `missing-thumb rescue start` が出た後、`thumbnail-create-process.csv` に再生成結果（`success`）が記録されることを確認する。
6. すぐ再投入したい場合は `手動更新` またはメニューの `監視フォルダ更新チェック` を実行する。どちらも `CheckMode.Manual` の欠損救済として扱われるので、`missing-thumb rescue start: mode=Manual` が出ることを確認する。

## 6.1 実フォルダとDB/表示がずれた時の自動補正確認（Watch運用）
1. 現在タブで開いている監視フォルダに、実ファイルは存在するが DB/表示へ未反映の動画を用意する。
2. Everything監視ポーリングを待ち、`watch-check` ログに以下のいずれかが出ることを確認する。
   - `scan reconcile start: folder='...' reason=watch_zero_diff`
   - `scan reconcile throttled: folder='...' next_in_sec=...`
3. `scan reconcile end: ... new=1` 以上が出た後、対象動画が表示へ反映されることを確認する。
4. DB登録済みだがサムネイルだけ欠損している場合は、続いて `missing-thumb rescue start` または `enqueue by missing-tab-thumb` が出ることを確認する。
5. 監視だけで補正を待ちたくない場合は `手動更新` またはメニューの `監視フォルダ更新チェック` を実行して即時補正してよい。

## 7. 注意点
- 欠損救済はDB+タブ単位で最短60秒間隔に制限される。
- Queue高負荷時（active件数が閾値以上）は `Watch` の自動救済を後回しにする。
- 明示的な `Manual` 実行では、高負荷中でも欠損救済を走らせる。
- 連続削除テスト時は、抑止ログが出ても異常ではない。
- `Watch + Everything + 差分0件` の時だけ、現在タブの監視フォルダに対して最短60秒ごとのフル再突合が入る。
