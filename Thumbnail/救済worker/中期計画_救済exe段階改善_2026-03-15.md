- 各段で「アルゴ更新」と「テストで失敗洗い出し」を交互に回す進め方を固定した
- `p6 -> p5` の反復条件を明文化した
- `p1` の初手として、fixed 順でも親行 `ExtraJson` に `RouteId=fixed / CurrentPhase / CurrentEngine` を書く方針を反映した
- `p2` の入口として、`attempt_failed` を engine / kind / reason で束ねる失敗洗い出しスクリプトを追加した

## 1. 目的

- 本体は `autogen` 単発で早期見切りまで到達した
- 残る主戦場は救済exeである
- ただし救済exeを一気に作り替えると、何が効いたかと何が壊れたかが追いにくい
- そのため中期計画は、次の 6 段で進める

## 2. 基本方針

### 2.1 先に順序を固め、その後で分岐へ上げる

- いきなり症状分岐型へ飛ばない
- まず fixed 順の各処理を詰めて、基準線を安定させる
- その後で症状分岐型へ上げる

### 2.2 各段の直後に必ず失敗洗い出しを入れる

- アルゴ更新だけを連続で積まない
- 各段の直後にテストと実測で失敗を洗い出す
- そこで見えた失敗を次段へ返す

### 2.3 本体へ重さを戻さない

- 改善対象は救済exe側に閉じる
- 本体の `autogen` 単発方針は維持する

## 4. 各フェーズの中身

### 4.1 `p1` 救済exeの fixed 順各処理のアルゴ更新

対象:
- `ffmpeg1pass`
- `ffmediatoolkit`
- `autogen`
- `opencv`
- `repair.probe`
- `repair.remux`

ここでやること:
- engine ごとの timeout 方針見直し
- `No frames decoded` 周りの文言吸収
- `repair gate` 語彙の拡張
- `FailureDb` 試行ログの質向上
- `CurrentEngine` / `CurrentPhase` の観測性整理

完了条件:
- fixed 順のままでも、各 engine の失敗理由が比較可能
- 既知の timeout / repair 入口 / 出力欠損が観測で追える

### 4.2 `p2` テストで失敗洗い出し

ここでやること:
- 単体テストで engine ごとの失敗を洗う
- 実動画で代表ケースを再投入する
- `FailureDb` とログから失敗束を作る
- `Thumbnail\救済worker失敗束サマリ_2026-03-15.ps1` を使って、`attempt_failed` の偏りを先に掴む

見る観点:
- timeout 不達
- `gave_up` が早すぎる
- `repair` へ入るべき個体が漏れる
- `opencv` の居座り
- `near-black` と `no-frame` の混同

成果物:
- 失敗一覧
- 再現個体
- 追加するべき分類キー

### 4.3 `p3` fixed 順から症状分岐型

ここでやること:
- `ClassifyRescueSymptom(...)` を追加する
- `BuildRescuePlan(...)` を route 別に持つ
- parent 行 `ExtraJson` に
  - `SymptomClass`
  - `RouteId`
  - `CurrentEngine`
  - `CurrentPhase`
  を載せる

初版 route:
- `long-no-frames`
- `ultra-short-no-frames`
- `near-black-or-old-frame`
- `corrupt-or-partial`

完了条件:
- fixed 順を直書きせず、route から engine 列が決まる
- 親行だけ見て「今どの route で止まっているか」が分かる

### 4.4 `p4` テストで失敗洗い出し

ここでやること:
- route ごとの単体テストを足す
- 誤分類ケースを洗う
- route には入ったが順序が悪いケースを洗う

見る観点:
- `ultra-short` と `long` の誤分類
- `corrupt-or-partial` の過検出
- `near-black` の取りこぼし
- route によって成功率が下がったケース
- 新 route を足す前に、既存 route へ寄せられる束が残っていないか

成果物:
- 誤分類一覧
- route 順序の再調整候補
- 追加すべき軽量判定項目

### 4.5 `p5` 症状分岐型各処理のアルゴ更新

ここでやること:
- 新 route 追加より先に、既存 route の分類語彙を強める
- 新 route 追加より先に、repair gate を route ごとに強める
- route ごとに engine 順を詰める
- `repair` を使う条件を route ごとに絞る
- `opencv` を route ごとに使うかどうかを見直す
- `FailureDb` への progress snapshot を整理する
- `IMM_THUMB_RESCUE_TRACE=1` 時だけ、改善処理専用 CSV で route / phase / engine / terminal を追えるようにする

主対象:
- `R-A long-no-frames`
- `R-B ultra-short-no-frames`
- `R-C near-black-or-old-frame`
- `R-D corrupt-or-partial`

完了条件:
- route ごとに「先に軽い手、後ろに重い手」が守られている
- 失敗時に次の一手が reason から読める
- `attempt_failed.kind` が `Unknown` に寄りすぎず、FailureDb 束だけで主因が読める
- `fixed / unclassified` 束が、語彙追加だけで寄せられる範囲まで減っている

### 4.6 `p6` テストで失敗洗い出し

ここでやること:
- route ごとの代表ケースを再実測する
- 未解決束を作る
- 未解決を `p5` のどの route へ返すか整理する

見る観点:
- まだ `gave_up` へ落ちる個体
- route 末尾まで使っても取れない個体
- watchdog が必要な個体
- 新しい分類が必要な個体

成果物:
- route 別未解決束
- `p5` へ戻す改善課題

## 5. 反復ルール

### 5.1 基本ループ

1. `p1`
2. `p2`
3. `p3`
4. `p4`
5. `p5`
6. `p6`
7. `p6` の結果を持って `p5` へ戻る

### 5.2 `p6 -> p5` へ戻る条件

- route の分類は正しいが、順序や gate が弱い
- `repair` の前後関係が悪い
- `opencv` を使うべき route と不要な route が見えてきた
- `near-black` / `old-frame` / `no-frame` がまだ粗い

### 5.3 `p6` で止めてよい条件

- 新しい未解決が watchdog 系だけに絞れた
- route 追加より engine hard timeout の方が先と判断できた
- 代表ケースの成功率が十分に上がった

## 6. 直近の着手順

1. `p1`

   - `opencv` nominal timeout 分離を入れる
