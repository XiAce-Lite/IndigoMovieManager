## 2. 初期観測

- `ffprobe`
  - `codec_name=av1`
  - `pix_fmt=yuv420p`
  - `3840x2160`
  - `duration=4318.655 sec`
  - `size=73763711551 byte`
- `video stream` は存在する
- `NoVideoStream` でも `CODEC NG` でもない

## 3. 実測結果

### 3.1 単発1枚抜き

- `ffmpeg -ss 300 -frames:v 1`
  - 一度は `sec-300.jpg` を出せた
- ただし再試行では
  - `20 sec timeout` で `30 / 300 / 900 sec` 全滅
  - `60 sec timeout` でも `300 sec` が全滅

## 4. 現時点の判断

- 本丸は `near-black` ではない
- 本丸は **超巨大 4K AV1 のシーク重さ**
- つまり、この個体は
  - 明るい場所を探す段階
  - route を増やす段階
  ではなく、
  - **まず1枚抜けるだけの安定したシーク経路を作る段階**

## 5. いま採らない判断

- 通常 rescue route へ特殊分岐を入れない
- `near-black` 扱いへ寄せない
- `CODEC NG` 扱いにも寄せない

## 6. 次の切り分け候補

1. `ffmpeg1pass` の timeout を `20 / 60 / 120 sec` で比較する
2. `-analyzeduration` / `-probesize` を厚めにした時に改善するかを見る
3. `-ss` の位置をより前寄りへ固定して、keyframe 依存の差を見る
4. remux や proxy 的な前処理を入れずに済むかを先に見極める
5. 一般方針は `超巨大AV1専用seek戦略_2026-03-20.md` へ分離して追う

## 7. 補助スクリプト

- `Invoke-ManualHugeAv1SeekProbe_2026-03-20.ps1`
  - 指定秒ごとの 1 枚抜きと timeout を、手動で比較するための切り分け用

## 8. 運用メモ

- この個体は `難読.wb` 本線とは別に扱う
- rescue worker 本体へ入れる前に、まず manual 切り分けで
  - timeout
  - seek 点
  - option 差
  を確定する
- 2026-03-20 実験として、同系統の seek 戦略を rescue worker の **最終救済** に仮組み込みした
- ただし本個体を即救う前提ではなく、
  - 通常 route では落ちる
  - しかし前進 decode なら拾える
  個体が他にも混ざるかを広く観測する段階とする
