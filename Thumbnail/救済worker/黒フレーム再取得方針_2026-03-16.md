# 黒フレーム再取得方針 2026-03-16

最終更新日: 2026-03-16

## 1. 目的

- 生成そのものは成功するのに、代表フレームが黒で見栄えが壊れる個体を救う
- route を増やさず、既存勝ち筋の中で「同じ engine を別時刻で撮り直す」だけに絞る
- `C:\WhiteBrowser\難読.wb` の黒残り個体を、個体別チューニング前に一段まとめて減らす

## 2. 発火条件

- rescue worker で `CreateThumbAsync(...)` が成功した
- ただし出力 jpg が near-black 判定で reject された

この時だけ `black_retry` へ入る。

## 3. 初版の固定ルール

- 対象:
  - rescue worker のみ
  - 本exe の通常本線には入れない
- 再取得回数:
  - 追加で最大 4 回
- 再取得位置:
  - 動画長の `10%`
  - 動画長の `35%`
  - 動画長の `65%`
  - 動画長の `85%`
- 実装制約:
  - 現在の `ThumbInfo` は整数秒しか持てない
  - そのため初版は「割合 -> 整数秒へ丸めた候補」で再取得する

## 4. 実行順

1. 初回の engine 試行
2. near-black reject
3. 同じ engine で `black_retry`
4. まだ near-black なら次候補時刻へ進む
5. 全候補が尽きたら、その engine は失敗として次 engine へ進む

## 5. 例外と制限

- 動画長が `1秒未満` で整数秒候補を作れない個体は、救済worker 限定で小数秒 ffmpeg 多点抜きへ逃がす
  - `10%`
  - `25%`
  - `50%`
  - `75%`
  - `90%`
  の小数秒候補を使う
  - 先に通った 1 枚で即決せず、全部抜いてから
    - near-black を除外
    - 明るさ
    - 彩度
    - コントラスト
    で採点し、上位フレームを panel 数ぶん並べる
  - 目的は `Grid` 表示を救うことなので、末尾メタ秒数は `0秒` で固定する
- それでも near-black なら既存の次 engine へ進む
- `opencv` は child process 隔離のまま維持し、`ThumbSec` override も child へ引き継ぐ

## 6. 観測方法

- runtime log
  - `black_retry`
- progress snapshot
  - `direct_black_retry`
  - `repair_black_retry`
- trace
  - `thumbnail-rescue-trace.csv`
  - `action=black_retry`

## 7. 期待する効果

- `【ライブ配信】神回scale_2x_prob-3`
- `インデックス破壊-093-2-4K`
- `「ラ・ラ・ランド」は少女漫画か！？ 1_2`
- `「ラ・ラ・ランド」は少女漫画か！？ 2_2`

のような「取れているが黒い」個体を、route 追加なしで減らす。
- 特に `【ライブ配信】神回scale_2x_prob-3` のような `0.16秒級` では、小数秒 fallback が本命になる。
