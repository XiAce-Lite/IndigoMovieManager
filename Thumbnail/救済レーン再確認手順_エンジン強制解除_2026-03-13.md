# 救済レーン再確認手順 エンジン強制解除 2026-03-13

## 1. 目的

- 救済レーンで `ffmpeg1pass` 先頭選択が実際に効いているかを再確認する。
- 直近ログでは rescue 中も `engine selected: id=autogen` が出ていたため、環境変数強制の影響を切り分ける。

## 2. 背景

- `Thumbnail/Engines/ThumbnailEngineRouter.cs` では、`QueueObj.IsRescueRequest == true` の時に `ffmpeg1pass` を先頭選択する。
- しかし実ログでは rescue 中の `CreateThumbAsync` でも `engine selected: id=autogen` が出ていた。
- `Properties/launchSettings.json` には `IMM_THUMB_ENGINE=autogen` があり、これが有効な起動では rescue 用ルーティングより強制指定が優先される。

## 3. 事前確認

- 対象ファイル: `C:\Users\na6ce\source\repos\IndigoMovieManager_fork_workthree\Properties\launchSettings.json`
- 確認点:
  - `IMM_THUMB_ENGINE` が `autogen` になっていないか
  - Visual Studio / デバッグ起動プロファイルで同値が流し込まれていないか

### 3.1 起動方法ごとの見方

| 起動方法 | `IMM_THUMB_ENGINE` の入り方 | 今回の見立て |
|---|---|---|
| Visual Studio / Debug の `Project` 起動 | `launchSettings.json` のプロファイルが反映される | `autogen` 強制が入る可能性が高い |
| `bin\\x64\\Debug\\net8.0-windows\\IndigoMovieManager_fork_workthree.exe` を直接起動 | `launchSettings.json` は通常反映されない | 強制が無ければ rescue ルーティングを見やすい |
| PowerShell から手動起動 | 現在のシェル環境変数だけを引き継ぐ | 事前に `Remove-Item Env:IMM_THUMB_ENGINE` で明示解除できる |

補足:
- リポジトリ内では `launchSettings.json` 以外に、本番コードが `IMM_THUMB_ENGINE` を常時設定する箇所は見当たらない。
- このシェルでは `IMM_THUMB_ENGINE` は空だった。

## 4. 再確認手順

1. アプリを停止する。
2. `IMM_THUMB_ENGINE` 強制を無効化する。
3. アプリを再起動する。
4. 対象動画を 1 本だけ選ぶ。
   - `「ラ・ラ・ランド」は少女漫画か！？ 2_2.mp4`
   - `インデックス破壊-093-2-4K.mp4`
5. 通常レーン失敗から rescue へ handoff させる。
6. 次のログを確認する。
   - `thumbnail-recovery`
   - `thumbnail-rescue`
   - `thumbnail-repair`
   - `thumbnail`

## 5. 合格条件

- rescue 開始後の最初のエンジン選択ログが `engine selected: id=ffmpeg1pass` になる
- 必要時だけ `engine fallback` で後続へ流れる
- `autogen` 強制時と比べて、rescue の経路差が説明できる

## 6. 失敗時の見方

- まだ `engine selected: id=autogen` のままなら、次を疑う。
  - 実行時環境変数が別経路で残っている
  - rescue 投入時に `IsRescueRequest` が落ちている
  - ログ地点とルーティング地点の認識がずれている
