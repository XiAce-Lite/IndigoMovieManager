# 設計メモ: 管理者権限サービス責務境界（2026-03-07）

## 1. 目的
- サムネイル高負荷検知で追加する管理者権限サービスの責務を、`AdminUsnMft` 既存境界と衝突しない形で固定する。
- Disk温度取得、UsnMft状態取得、権限境界、オーケストレータ責務の切り分けを先に確定する。

## 2. 結論
- 管理者権限サービスは **特権が必要な情報取得だけ** を持つ。
- 並列度の縮退・復帰・レーン予約の最終判断は **オーケストレータ側** に残す。
- `Watcher` 側の `AdminUsnMft` と、サムネイル側の Disk温度取得は、将来的に **同じ昇格サービス基盤** に寄せる前提で扱う。
- ただしこの段階では「単一サービスへ統合する詳細実装」までは決めず、共通基盤へ載せられる責務境界だけを固定する。

## 3. サービスに入れる責務
- `GetSystemLoadSnapshot`
  - CPU、I/O待ち傾向、メモリ圧迫など、プロセス外または特権付きで取ると安定する全体負荷の取得。
- `GetDiskThermalSnapshot`
  - 物理ディスク温度、警告閾値、危険閾値、取得不能状態の返却。
- `GetUsnMftStatus`
  - ボリューム単位の UsnMft 利用可否、走査状態、直近遅延、未処理量の返却。
- `GetCapabilities`
  - そのサービスが `DiskThermal` と `UsnMft` のどちらに対応しているか、昇格が必要かを返す。

## 4. サービスに入れない責務
- `HighLoadScore` の合成
- 並列度の縮退・復帰判断
- `slow / normal / ballence / fast / max / custum` のプリセット解釈
- レーン予約判定
- QueueDB やジョブ再投入の制御
- UI通知文言の組み立て
- ログカテゴリの最終決定

## 5. 権限境界
- 昇格が必要な API 呼び出し、デバイス情報取得、UsnMft状態取得は管理者権限サービス側へ閉じ込める。
- 本体UI、オーケストレータ、サムネイルエンジンは管理者権限を前提にしない。
- サービス未接続や権限不足は、オーケストレータ側で `internal-only` の縮退運転へ落とす。
- `Watcher` の通常監視や `StandardFileSystem` 経路は、管理者権限サービスに依存させない。

## 6. オーケストレータとの境界
- オーケストレータが持つもの
  - DTOの解釈
  - `HighLoadScore` の計算
  - しきい値比較
  - 縮退と復帰
  - ログ出力
  - UI表示更新
- 管理者権限サービスが持つもの
  - 特権付き情報取得
  - サービス能力の開示
  - 取得失敗理由の返却

## 7. Watcher との境界
- `Watcher` は `FileIndexProviderFacade` を維持し、検索プロバイダ選択責務を持ち続ける。
- 管理者権限サービスは `Watcher` のUI責務やプロバイダ選択責務を持たない。
- `AdminUsnMft` 由来の状態取得は、将来的に管理者権限サービスの `GetUsnMftStatus` へ統合可能な粒度に揃える。
- `availability_error:AdminRequired` のような既存 `Watcher` 側 reason 文字列は、そのままサムネイル高負荷ログへ流用しない。

## 8. 契約型
- `src/IndigoMovieManager.Thumbnail.Queue/Ipc/AdminTelemetryContracts.cs`
  - `AdminTelemetryRequestContext`
  - `AdminTelemetryServiceCapabilities`
  - `IAdminTelemetryClient`
- 取得契約は `Task<...>` で統一し、サービスが判断主体にならないよう `SetThrottle...` のような逆方向APIは持たせない。

## 9. 受け入れ基準
- Disk温度取得、UsnMft状態取得、昇格境界の責務が1ページで説明できる。
- サービスが縮退判断主体ではないことが明記されている。
- `Watcher` とサムネイルが別々の昇格経路を増やさない方針と矛盾しない。
- 将来 `LTH-019A` で共通基盤化しても、責務の再分割が不要である。
