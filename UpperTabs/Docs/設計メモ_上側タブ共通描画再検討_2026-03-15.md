# 設計メモ: 上側タブ共通描画再検討 (2026-03-15)

## 1. 結論

- 上側タブ全体を 1 つの「共通描画クラス」に寄せる案は採用しない。
- 採用するのは「共通描画基盤」の導入であり、共通化の対象は以下に限定する。
  - アクティブタブ判定
  - 可視範囲の追跡
  - 可視範囲優先のサムネ要求スケジューリング
  - タブ別 decode サイズ定義
  - 画像キャッシュ戦略
- `Small / Big / Grid / Big10` は同じ基盤に乗せる。
- `List` は同じ優先制御には乗せるが、描画面は別扱いにする。

要するに、共通化するのは「速くするための基盤」であって、「見た目まで 1 クラスに押し込むこと」ではない。

## 2. 目的

- 目的は高速化。
- 最優先は「今表示されている部分」。
- サムネ作成も「今表示されている部分」を最優先にする。
- 1 万件規模でもスクロールを引っ掛けない。
- 上側タブを将来タブ単位で分担保守しやすくする。

## 3. 現状整理

### 3.1 上側タブの実体

- `Small / Big / Grid / Big10` は、同じ `FilteredMovieRecs` を `ListView + VirtualizingWrapPanel` で別テンプレート描画している。
- `List` は同じ `FilteredMovieRecs` を `DataGrid` で描画している。
- つまり、データ源は共通だが、描画コンテナと1件あたりの visual tree が違う。

### 3.2 すでに入っている最適化

- 上側タブの切替時に `CurrentTabIndex` を更新し、通常キューのリース順は「現在タブ優先」になっている。
- `ListView` 側は `VirtualizingWrapPanel`、`VirtualizationMode=Recycling` が入っている。
- 下部タブでは「非アクティブ時は更新しない」方針を導入済み。

### 3.3 足りていない点

- 優先制御は「現在タブ」までで、可視範囲までは見ていない。
- サムネ要求は visible-first ではなく、基本はキュー投入順と FIFO に近い。
- 上側タブの画像読込は、ほとんどが `IsExists` しか converter parameter に渡しておらず、tab ごとの `DecodePixelHeight` が効いていない。
- `NoLockImageConverter` のメモリキャッシュは 256 件固定で、1 万件前提では小さい。
- `FilteredMovieRecs` は `ReplaceFilteredMovieRecs` で全件差し替えする構造で、一覧全体志向がまだ強い。

## 4. 単一の共通描画クラスを採用しない理由

### 4.1 レイアウト差がそのまま性能差になる

- `Small / Big / Grid / Big10` は全部サムネ主役だが、1件の大きさ、詳細量、タグ量が違う。
- `List` は `DataGrid` で、列仮想化やテキスト列の比重が高い。
- ここを 1 つの base class や巨大 `UserControl` に押し込むと、性能差の調整点まで隠れてしまう。

### 4.2 WPF では visual tree の違いを消しすぎると逆に遅くなる

- 画像サイズ
- テンプレートの深さ
- タグ領域の有無
- `DataGrid` と `ListView` のスクロール特性

この差を「共通描画クラス」内部の分岐で吸収すると、保守対象は 1 つに見えても、実際は最適化ポイントが読みにくくなる。

### 4.3 visible-first 要件は view 共通化だけでは満たせない

- 速くしたい本丸は「見えている範囲だけを先に描く」「見えている範囲のサムネだけを先に作る」こと。
- これは view テンプレート共通化ではなく、viewport と queue priority の設計課題。

## 5. 採用する構成

### 5.1 方向性

- 描画はタブごとに持つ。
- 制御は共通基盤へ寄せる。

### 5.2 新しい責務分割

## 6. 共通基盤で持つべきもの

### 6.1 `UpperTabActivationGate`

- 上側タブのうち、今アクティブな 1 タブだけを更新対象にする。
- 下部タブと同じく、非アクティブ時は dirty だけ立てる。
- ただし上側は、これだけでは不十分で、さらに可視範囲 gate が必要。

### 6.2 `UpperTabViewportTracker`

- アクティブタブの visible item range を取る。
- 最低限必要なのは以下。
  - 先頭可視 index
  - 末尾可視 index
  - overscan 前後件数
- `ScrollChanged` をそのまま多発反映せず、16ms から 33ms 程度で間引く。

### 6.3 `UpperTabThumbnailPriorityScheduler`

- `Visible`
- `NearVisible`
- `Background`

の 3 段程度で優先度を分ける。

- `Visible` は今見えている範囲。
- `NearVisible` はその前後の少しだけ。
- `Background` は残り全部。

ここで重要なのは、現在タブ優先のさらに内側へ、visible-first を入れること。

### 6.4 `UpperTabDecodeProfile`

タブごとに decode 目安を固定する。

- `Small`
- `Big`
- `Grid`
- `Big10`
- `List`

これを `NoLockImageConverter` の parameter へ渡す。

今の構造では、上側タブの多くが decode サイズ未指定で原寸に近い読み込みへ流れている可能性がある。ここは速度に直結する。

### 6.5 `UpperTabImageCachePolicy`

- cache key は path だけでなく decode profile を含める。
- visible 領域で触る decode 結果は優先保持する。
- 256 件固定は見直し候補。
- ただし無制限キャッシュは採らず、メモリ上限を持つ。

## 7. タブごとに残すべきもの

### 7.1 `Small / Big / Grid / Big10`

- item template
- 1件あたりの詳細量
- クリック操作
- 選択追従
- タグ表示量

これらはタブ固有のままでよい。

### 7.2 `List`

- `DataGrid` のまま扱う。
- visible-first の queue 優先制御には乗せる。
- ただし描画共通クラスには無理に乗せない。

`List` まで同じ描画 host に寄せると、かえって `DataGrid` の調整点が埋もれる。

## 8. サムネ優先制御の目標形

### 8.1 望ましい流れ

1. タブがアクティブになる
2. viewport tracker が visible range を取る
3. visible range の item だけ先に UI 更新対象へ入れる
4. 同じ range の item だけ先にサムネ生成要求へ入れる
5. 近傍 range をその次に入れる
6. 背景残件は最後に回す

### 8.2 今の通常キューとの差

今は `CurrentTabIndex` までは効いているが、同じタブ内の「どの item が今見えているか」は見ていない。

この差が、1 万件規模での体感差になる。

## 9. queue 側の設計判断

- 現在の `TabIndex` 優先は残す。
- その上で、同一タブ内の visible-first を表せる priority を追加する。
- 既存の `ThumbPanelPos` は本来の意味を崩すので、viewport 優先の流用先にはしない方が安全。
- 追加するなら以下のような独立値がよい。
  - `PriorityClass`
  - `ViewportOrdinal`
  - `ViewportSnapshotId`

要するに、visible-first は「今見えている順序」の情報として別に持つべきで、手動サムネ位置の意味を混ぜない。

## 10. スクロール最適化で重視する点

- スクロール中は visible item の画像以外を触りすぎない。
- スクロール中に一覧全体 `Refresh()` を走らせない。
- 画像の decode サイズを view サイズへ寄せる。
- タグや詳細の重い領域は、必要ならサムネ表示後に遅延反映する余地を残す。

まず最初に効くのは、

- visible range の把握
- decode サイズ指定
- visible-first enqueue

の 3 点である。

## 11. 導入順

### Phase 1: 計測と軽量化

- 上側タブ用 `UpperTabDecodeProfile` を導入
- `NoLockImageConverter` に tab ごとの decode 指示を通す
- アクティブ上側タブ以外の更新を止める

### Phase 2: 可視範囲追跡

- `UpperTabViewportTracker` を追加
- active tab の visible / near-visible range を取る
- dirty 管理を visible range 単位へ広げる

### Phase 3: サムネ優先制御

- `UpperTabThumbnailPriorityScheduler` を追加
- queue request に visible-first priority を載せる
- consumer の lease 順へ反映する

### Phase 4: タブ分割

- `UpperTabs/Common` を追加
- `Small / Big / Grid / Big10 / List` を host 化
- `MainWindow` は shell に寄せる

## 12. 成功条件

- 非アクティブな上側タブは更新しない
- アクティブタブでも、可視範囲を最優先に更新する
- サムネ生成も、可視範囲を最優先に消化する
- 上側タブの画像 decode は tab ごとのサイズに寄る
- 1 万件規模でも、スクロール中に「見えていない項目のための仕事」で引っ掛からない

## 13. 今回の判断

- 「共通描画クラス」は採らない
- 「共通描画基盤」は採る
- 先にやるべきは XAML の統一ではなく、visible-first を成立させる制御層の共通化

この順序なら、`workthree` の体感テンポ最優先という方針と噛み合う。
