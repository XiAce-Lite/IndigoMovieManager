# Implementation Plan: 重複動画タブ 2026-03-20

最終更新日: 2026-03-20

変更概要:
- 上側に新規 `重複動画` タブを追加する前提を整理
- 初期要件を `同一ハッシュ検出` に限定し、高速化優先で段階導入する
- 画面構成を「左右2ペイン」に変更し、`prob情報` は右詳細だけで扱う方針へ更新

## 1. 目的

`重複動画` タブは、通常一覧の代替ではなく、重複検出を一時的に見るための分析タブとして追加する。

優先順位は以下で固定する。

1. 通常一覧のテンポを落とさない
2. ボタン押下で必要な時だけ重複検索する
3. 初期版は「同一ハッシュ」を正とする
4. 左で重複束を選び、右で比較対象を最小 UI で並べる

## 2. 初期要件

- タブ名: `重複動画`
- ヘッダーパネルに `検出` ボタンを置く
- 検出対象:
  - `movie.hash` が同じ
  - `hash` が空でない
  - 件数が 2 件以上
- 画面は左右 2 つの表で構成する
- 左表:
  - `Grid` アイコン
  - 代表動画名
  - 重複数
- 左クリックで、右表に選択した重複動画群の詳細を出す
- 右表:
  - サムネイル
  - ファイル名
  - `prob情報`
  - サイズ
  - ファイル日付
  - パス
- `prob情報` は右表だけでよい
- サイズ比較がすぐ分かる

## 3. 初期版でやらないこと

- 類似動画判定
- 画質推定ベースの自動優劣判定
- 音声違い、尺違いの自動クラスタリング
- 常時自動更新
- Watcher 連動の自動再検索

## 4. 置き場所と固定ID方針

既存の上側固定IDは崩さない。

- `Small = 0`
- `Big = 1`
- `Grid = 2`
- `List = 3`
- `5x2 = 4`
- `救済 = 5`
- `重複動画 = 6` を新設

このタブは通常サムネ生成の visible-first 制御対象には入れない。
理由は、重複動画タブは通常一覧ではなく「調査用の別ビュー」だからである。

## 5. データ取得方針

初期版は DB 直読みに寄せる。

理由:

- `FilteredMovieRecs` 全件走査に乗せる必要がない
- 検出ボタン押下時だけ SQL を打てばよい
- 1 万件規模でも SQLite の `GROUP BY hash HAVING COUNT(*) > 1` の方が素直

### 5.1 取得の2段構え

1. 左表用の重複ハッシュ群を取る

```sql
SELECT
    hash,
    COUNT(*) AS item_count,
    MIN(movie_size) AS min_size,
    MAX(movie_size) AS max_size
FROM movie
WHERE COALESCE(hash, '') <> ''
GROUP BY hash
HAVING COUNT(*) >= 2
ORDER BY COUNT(*) DESC, MAX(movie_size) DESC;
```

2. 右表用に、選択したハッシュ 1 件ぶんの行を取る

```sql
SELECT
    movie_id,
    movie_name,
    movie_path,
    movie_size,
    file_date,
    movie_length,
    score,
    hash
FROM movie
WHERE hash = @Hash
ORDER BY movie_size DESC, file_date DESC, movie_id DESC;
```

### 5.2 `prob情報` の扱い

`prob情報` は現状 DB 列に無い。
そのため初期版は `Movie_Name` または `Movie_Body` から抽出する。

対象はまず `prob` 系だけに限定する。

例:

- `_prob3`
- `_prob-3`
- `prob3`
- `prob-3`

必要なら次段で `ghq5` や `scale_2x` なども追加するが、初期版では広げすぎない。

## 6. UI 方針

## 6.1 ヘッダーパネル

- `検出` ボタン
- `重複ハッシュ数` 表示
- `選択中ハッシュ` 表示
- `選択中重複数` 表示
- `検索中` の簡易表示

必要なら次段で追加:

- `サイズ差ありのみ`
- `probありのみ`

## 6.2 左表

左表は軽量 `DataGrid` か `ListView` のどちらでもよいが、初期版は `DataGrid` を第一候補とする。

理由:

- 入れ子を減らせる
- 行仮想化が素直
- 代表列だけに絞れる

左表の列はこれで固定する。

- `Grid` アイコン
- 代表動画名
- 重複数

左表の代表動画名は、同一ハッシュ群の中で以下の優先順で選ぶ。

1. サイズ最大
2. `file_date` が新しい
3. `movie_id` が大きい

## 6.3 右表

右表は選択中のハッシュ群だけを表示する軽量 `DataGrid` にする。

理由:

- 左クリック時に必要な群だけ読めばよい
- `prob情報` 抽出は選択群に対してだけ行えばよい
- 群単位の比較に集中できる

ただし、完全に `List` タブの列数までは増やさない。
列は最小限に絞る。

推奨列:

- サムネイル
- `prob情報`
- サイズ
- サイズ差分
- ファイル日付
- ファイル名
- パス

右表はグループ化しない。
左表で選んだ 1 グループだけを、そのままサイズ順で並べる。

## 7. サムネイル表示方針

初期版は `Grid` 相当サイズを使う。

- 幅: 160
- 高さ: 120
- 使用パス: `ThumbPathGrid`

理由:

- 情報量と密度のバランスが良い
- 既存キャッシュを流用しやすい
- `Small` は横長すぎ、`Big` は分析タブには重い

## 8. モデル設計

新規 ViewModel をタブ専用で作る。

候補:

- `UpperTabs/DuplicateVideos/UpperTabDuplicateGroupViewModel.cs`
- `UpperTabs/DuplicateVideos/UpperTabDuplicateItemViewModel.cs`

左表モデルの最低限の持ち物:

- `Hash`
- `RepresentativeMovieRecord`
- `RepresentativeThumbnailPath`
- `RepresentativeMovieName`
- `DuplicateCount`
- `MaxMovieSize`
- `MinMovieSize`

右表モデルの最低限の持ち物:

- `MovieRecord`
- `ThumbnailPath`
- `MovieName`
- `MoviePath`
- `MovieSizeText`
- `FileDateText`
- `ProbText`
- `SizeCompareText`

## 9. 実装フェーズ

### Phase 1

- タブ追加
- ヘッダーパネル追加
- 左表の重複グループ一覧を DB から読む
- 左クリックで右表を切り替える
- 右表は選択グループだけを読む

完了条件:

- 左に重複束が並ぶ
- 左クリックで右に詳細が出る
- 通常一覧のスクロールやサムネ生成テンポに影響しない

### Phase 2

- `prob情報` 抽出を右表へ追加
- サイズ差分表示
- 左表の代表選定を固定

完了条件:

- 「同じ hash だがサイズ差あり」が右表で一目で分かる
- `prob` の有無が右表で一目で分かる

### Phase 3

- フィルタ追加
- 保持候補の優先表示
- 右クリック操作の追加

候補:

- 片方を残して他方を選択
- 親フォルダを開く
- 比較再生

## 10. 性能上の注意

- タブを開いただけでは何もしない
- 検出ボタン押下時だけ左表を読む
- 右表は選択時だけ 1 ハッシュぶん読む
- UI へ渡す前にバックグラウンドで整形する
- 深い `ItemsControl` 入れ子は避ける
- サムネイルは既存 `ThumbPathGrid` を読むだけにする
- このタブのために新規サムネ生成は走らせない

## 11. 懸念点

### 11.1 空 hash

空 hash 行は初期版では除外する。
ここを混ぜると誤検出が一気に増える。

### 11.2 `prob情報` の定義

専用列が無いので、初期版は「ファイル名由来の補助情報」で割り切る。
検索キーにはしない。

### 11.3 大量群

同一 hash が非常に多い場合でも UI を固めないよう、1 回の描画件数はログで観測できるようにする。

## 12. 最初の実装順

1. `UpperTabs/DuplicateVideos` フォルダ新設
2. `重複動画` タブ host 追加
3. 左表用の DB 読み取り service 追加
4. 右表用の DB 読み取り service 追加
5. 左表 group model / 右表 item model 追加
6. 検出ボタンで左表反映
7. 左クリックで右表反映
8. `prob情報` とサイズ差分を追加

## 13. 結論

`重複動画` タブは、通常一覧の派生表示として作るより、上側の「手動分析タブ」として切り出す方が正しい。

初期版は `同一ハッシュ` に限定し、

- DB 直読みに寄せる
- `Grid` サイズの既存サムネを流用する
- 左右 2 ペインにする
- 左は `Grid` アイコン、代表動画名、重複数だけに絞る
- 右は選択グループの詳細だけを出す
- `prob情報` は右だけで扱う

この4点で入るのが一番安い。
