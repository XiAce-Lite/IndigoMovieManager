# Implementation Plan: UI詰まり通知 別スレッドHWNDオーバーレイ (2026-03-20)

最終更新日: 2026-03-20

変更概要:
- UI処理詰まり通知の第1実装を `別スレッド HWND オーバーレイ` に固定する
- WPF のメインUIスレッドが詰まっても通知だけは出せる構成を定義する
- 実装を重くしないため、通知責務を最小化する
- 背景 heartbeat から `MainWindow` へ直接触れず、UI スレッドで取得した HWND/最小化状態のスナップショットを使う方針を追記

## 1. 結論

第1実装は以下で固定する。

1. 通知は WPF メインUIツリーに載せない
2. 専用 STA スレッドで通知用ネイティブ HWND を 1 個だけ持つ
3. UI詰まり検知はバックグラウンド側 heartbeat で行う
4. 閾値超過時だけオーバーレイを表示する
5. 復帰したら自動で閉じる

この方式なら、WPF 本体が重くても通知経路を分離できる。

## 2. 採用理由

- 同一 WPF UI スレッド依存の通知は、詰まり時に一緒に止まる
- `InvalidateVisual` や `UpdateLayout` では本質的に解決しない
- 別プロセスより軽く、同一プロセス内では十分強い
- 通知専用に責務を絞れば保守しやすい

## 3. 非採用案

### 3.1 WPF ヘッダー直表示だけで済ませる案

- 軽いが、本当に詰まった時は一緒に止まる
- 補助表示としては残せるが、本命通知にはしない

### 3.2 MessageBox / 別WPF Window

- 重い
- フォーカスや操作を邪魔しやすい
- 同じUI詰まりの影響を受けやすい

### 3.3 別プロセス watchdog

- 強いが重い
- IPC、起動順、終了処理、誤検知制御が必要
- 第1実装としては過剰

## 4. 構成

構成要素は 3 つに分ける。

- `UiHangHeartbeatMonitor`
  - UI詰まりを検知する
- `NativeOverlayHost`
  - 専用 STA スレッドを起動し、通知用 HWND を管理する
- `UiHangNotificationCoordinator`
  - 検知結果を通知表示へ変換する

## 5. 監視方式

heartbeat 監視はバックグラウンドで行う。

方式:

1. 一定間隔で `Dispatcher.BeginInvoke` を投げる
2. 投げた時刻と UI 実行時刻の差を測る
3. 差が閾値超過なら「UI応答低下」と判定する

観測値は以下だけに絞る。

- `delay_ms`
- `level`
- `is_recovered`

## 6. 通知UI仕様

通知は小さな横長バナーに固定する。

表示内容:

- 状態色
- 短文1行

文言候補:

- `UI応答低下を検知`
- `重い処理を継続中`
- `しばらくお待ちください`

表示位置候補:

- 右上固定

第1実装では右上固定でよい。

## 7. HWND 方針

通知用ウィンドウは以下を目標にする。

- 非アクティブ表示
- 最前面
- タスクバー非表示
- サイズ固定
- アニメーションなし

第1実装の考え方:

- `WS_EX_TOPMOST`
- `WS_EX_TOOLWINDOW`
- `WS_EX_NOACTIVATE`

描画はシンプルにする。

- 背景色塗り
- テキスト1行
- アイコンは任意

## 8. 通知ルール

誤検知を減らすため、連続超過でだけ出す。

- 250ms超: 注意
- 1000ms超: 応答低下
- 3000ms超: 重度応答低下

表示条件:

- 2回連続超過で表示

解除条件:

- 3回連続正常で解除

## 9. API イメージ

通知側APIは最小にする。

- `Start()`
- `Stop()`
- `Show(level, message)`
- `Hide()`
- `Update(level, message)`

メイン側からは同期呼び出ししない。
通知要求は queue 経由またはスレッド安全なポストだけにする。

## 10. 実装メモ

### 10.1 別スレッド

- 専用 STA スレッドを1本起動する
- そのスレッドでメッセージループを回す
- HWND の生成と破棄もそのスレッド内で完結させる

### 10.2 更新頻度

- 表示状態が変わった時だけ更新する
- 毎 heartbeat ごとに再描画しない

### 10.3 ログ

既存 `ui-tempo` へ状態遷移だけ出す。

- 検知開始
- 重度化
- 復帰

## 11. リスク

- 同一プロセスなので、プロセス全体クラッシュには勝てない
- DPI やマルチモニタ対応を後回しにすると表示位置がぶれやすい
- 最前面ウィンドウ競合の調整が必要になる可能性がある
- 通知文言が多すぎると運用がぶれる

## 12. 第1実装でやらないこと

- 進捗率表示
- 詳細原因一覧
- スタックトレース表示
- アニメーション
- 別プロセス化
- 入力受付

## 13. 成功条件

- WPF メインUIが重い時でも通知が見える
- 通知そのものが新たな詰まり要因にならない
- 復帰後に自動で消える
- ログで検知開始と復帰が追える

## 14. 実装順

1. heartbeat 監視の最小実装
2. 別スレッド HWND の固定バナー生成
3. `Show/Hide/Update` の接続
4. 閾値と復帰条件の調整
5. 文言整理

## 15. 2026-03-20 骨格実装反映

今回の反映では、以下の最小骨格まで入れた。

- `UiHangHeartbeatMonitor`
  - `Dispatcher.BeginInvoke` 往復と pending 経過時間の観測だけを持つ
- `NativeOverlayHost`
  - 専用 STA スレッド
  - `HwndSource` ベースの固定バナー HWND
  - `Show / Hide / Update` のポスト経路
- `UiHangNotificationCoordinator`
  - 閾値判定
  - 2回連続超過で表示
  - 3回連続正常で解除
  - `ui-tempo` の状態遷移ログ
  - 実行中の代表処理種別に応じた文言切替
  - 危険扱いは `pending 長時間継続` または `hung window` を優先して `Critical` へ上げる
- `MainWindow`
  - `Loaded` で開始
  - `Closing` で停止
  - 背景通知がクロススレッド例外を起こさないよう、`SourceInitialized` / `StateChanged` / `Activated` / `Deactivated` で HWND と最小化状態を更新し、判定側はそのスナップショットだけを見る

文言切替の第1段階では、以下の代表種別を使う。

- `監視処理`
- `DB処理`
- `サムネイル処理`
- `起動処理`

この段階では、まだ以下は後続とする。

- DPI / マルチモニタ補正
- 文言や色の運用調整
- 表示位置の詳細微調整
- 実動画 / 実負荷での閾値微調整

表示位置は第1反映として、画面右上固定ではなく `MainWindow` のヘッダー帯へ追従する形へ寄せた。
