# 詳細設計: 大DB起動段階ロード化 (2026-03-17)

最終更新日: 2026-03-18

変更概要:
- 大DB起動時の全件展開をやめるための詳細設計を定義する
- 現行コードの責務分解点と、新コンポーネントの役割を固定する
- そのまま実装へ落とせるタスクリストを付ける
- Phase 0 / Phase 1 / Phase 2 の初期骨格実装の到達点を反映する
- 起動時 full feed を廃止し、viewport 近傍でだけ page append する現仕様を反映する
- メインヘッダー件数を `表示 / 検出 / 登録` に分離し、`登録` は DB 総数を後追い確定する仕様を反映する

## 0. 現在地

2026-03-17 時点で、以下の骨格実装を反映済み。

- 起動時の `FilterAndSort(...)` 直列実行を外し、first-page 起動経路へ切り替えた
- `StartupLoadCoordinator` を新設し、起動要求の revision / cancel 管理を入れた
- `ReloadBookmarkTabData()`、`QueueCheckFolderAsync(...)`、`CreateWatcher()` を first-page 後へ後ろ倒しした
- `CheckThumbAsync(...)`、queue persister、Everything poll supervisor も first-page 後の起動へ寄せた
- さらに startup の deferred を light / heavy に分離し、heavy 側は first-page 後の短い遅延起動へ寄せた
- `StartupDbPageReader` を新設し、起動時の DB fallback を `SELECT *` ではなく必要列のみの reader 読取へ切り替えた
- 先頭 200 件表示と、viewport 近傍での 300 件ずつ on-demand append を追加した
- startup partial 中は `watch-check` の既存DB動画 view 補修を抑止し、意図した partial view を不整合扱いしないようにした

まだ未完了のもの:

- `MovieRecords` 依存マップの棚卸し
- `MovieRecordShellFactory` の独立クラス化
- `Movie_Id` 基準の選択再解決
- sidecar Catalog 全般

## 1. 目的

目的は 1 つだけである。

- 大DBでも、起動直後に一覧の先頭を操作可能にする

このために、起動時の処理を以下へ作り替える。

1. ウィンドウと最低限の表示設定だけ先に出す
2. 一覧は先頭 200 件だけ出す
3. 残りは scroll / viewport 近傍で必要になった時だけ追加する
4. Watcher、キュー、存在確認、件数確定は後ろへ逃がす

## 2. 現行構造と問題点

現行の起動主要経路は以下である。

1. `MainWindow_ContentRendered`
2. `TrySwitchMainDb(...)`
3. `OpenDatafile(...)`
4. `BootNewDb(...)`
5. `FilterAndSort(...)` / `FilterAndSortAsync(...)`
6. `ReloadBookmarkTabData()`
7. `QueueCheckFolderAsync(...)`
8. `CreateWatcher()`

ボトルネックは `BootNewDb(...)` の中に「UI 表示に必要な仕事」と「起動後でよい仕事」が同居していること。

特に重いのは以下。

- `GetData(db, "SELECT * FROM movie ...")`
- `SetRecordsToSource(...)` による全件 `MovieRecords` 生成
- `ReplaceFilteredMovieRecs(...)` による全件差し替え
- `Refresh()` による一覧再描画

`VirtualizingWrapPanel` は UI 要素化の抑制には効いているが、
一覧へ渡す前の全件展開には効かない。

## 3. 設計方針

### 3.1 起動経路と通常経路を分ける

起動時は、従来の「DB を開いたらそのまま全件一覧を構築する」経路を使わない。

分ける経路:

- 起動経路
  - 先頭ページを早く出すことが目的
- 通常経路
  - 検索、ソート、再読込、監視追従を安定して回すことが目的

### 3.2 UI 互換を優先し、段階的に軽量化する

最終形では `MovieListItem` / `MovieDetails` の 2 層へ分ける。
ただし Phase 2 までは UI 破壊を避けるため、`MovieRecords` を即時廃止しない。

初期段階の方針:

- データ源は軽量 DTO 化する
- UI には互換シェルとして軽い `MovieRecords` を流す
- 重い情報は後から hydrate する

### 3.3 sidecar は高速化の補助であって正本ではない

正本はあくまで `*.wb`。
sidecar は壊れてもよい前提で設計する。

### 3.4 workthree 方針に合わせる

優先順位は固定する。

1. 最初の 200 件が見えること
2. クリックできること
3. 正確さは後追いでもよいこと

## 4. 目標構成

## 4.1 新規コンポーネント

### A. `StartupLoadCoordinator`

役割:

- 起動時のロード全体を調停する
- warm start / cold start / cancel / stale request を管理する
- first-page 表示後に deferred task を起動する

責務:

- 起動要求の採番
- 現在要求以外の結果を捨てる
- UI へ first-page を渡す
- 後続ロードを順番に起動する

### B. `StartupFeedRequest`

役割:

- 起動時の読み込み条件を 1 つにまとめる

主な内容:

- `DbPath`
- `SortId`
- `SearchKeyword`
- `Skin`
- `InitialTabIndex`
- `PageSize`

### C. `StartupFeedPage`

役割:

- 一覧へ 1 ページ分を渡すための DTO

主な内容:

- `Items`
- `ApproximateTotalCount`
- `HasMore`
- `SourceKind`
  - `CatalogWarm`
  - `CatalogPaged`
  - `DbFallback`

### D. `StartupCatalogService`

役割:

- sidecar カタログの読取、整合性判定、再構築を担当する

主な責務:

- Catalog の open
- DB 識別情報の照合
- warm start ページ取得
- 主要ソートの先頭ページ取得
- 再構築要求の受付

### E. `StartupDbPageReader`

役割:

- sidecar が使えない時の DB フォールバック読取を担当する

主な責務:

- 本家 DB からページ 1 を読む
- 主要ソート外のフォールバックを吸収する
- 必要なら `LIMIT/OFFSET` か keyset 方式を切り替える

### F. `StartupDeferredServiceCoordinator`

役割:

- first-page 後に起動する後段処理を管理する

主な責務:

- Bookmark 再読込
- Watcher 起動
- queue 常駐開始
- rescue worker 起動
- 正確件数更新
- 存在確認更新

### G. `MovieListItemSource`

役割:

- 一覧用の最小データを表す軽量 DTO

主なフィールド候補:

- `MovieId`
- `MoviePath`
- `MovieName`
- `MovieBody`
- `LastDate`
- `FileDate`
- `RegistDate`
- `Score`
- `ViewCount`
- `Hash`
- `ThumbKey`
- `Ext`

### H. `MovieRecordShellFactory`

役割:

- `MovieListItemSource` から、UI 互換の軽量 `MovieRecords` を作る

方針:

- 起動時点で不要な重いフィールドは空または遅延値にする
- XAML をすぐ全面差し替えずに済ませる

### I. `MovieDetailsHydrator`

役割:

- 選択された 1 件だけ重い情報を埋める

主な責務:

- `movie_id` または `movie_path` で本家 DB を読む
- タグ、コメント、メディア詳細、存在確認を反映する
- 選択が変わったら古い要求を捨てる

## 4.2 既存メソッドの再配置

### `OpenDatafile(...)`

残す責務:

- スキーマ検証
- 旧 DB の shutdown
- 起動要求の開始

外へ出す責務:

- 一覧構築
- Bookmark 再読込
- Watcher 起動
- queue 常駐開始

### `BootNewDb(...)`

この名前のままでは責務が広すぎるため、最終的に分解する。

分解先:

- `LoadDbSystemState(...)`
- `ApplyDbVisualState(...)`
- `BeginStartupFeedOpen(...)`
- `ScheduleDeferredDbServices(...)`

### `FilterAndSortAsync(...)`

通常経路では残すが、起動直後の first-page 取得には使わない。

理由:

- 全件配列前提が強い
- `MovieRecords[]` と `FilteredMovieRecs` 全体志向が前提になっている

## 5. データ設計

## 5.1 UI 表示モデル

### Phase 2 まで

- UI の `ItemsSource` は `ObservableCollection<MovieRecords>` を維持する
- ただし中身は `MovieRecordShellFactory` が作る軽量シェルにする

### Phase 4 以降

- 詳細表示まわりを `MovieDetails` 分離へ寄せる
- 必要なら `MovieRecords` は adapter 専用へ縮退する

## 5.2 sidecar カタログ

初期は `StartupCatalog.sqlite` 1 ファイルに統合する。

最小テーブル案:

- `catalog_meta`
  - `db_path`
  - `db_last_write_utc`
  - `db_file_size`
  - `movie_count`
  - `catalog_version`
  - `app_version`
  - `generated_utc`
- `warm_pages`
  - `sort_id`
  - `search_key`
  - `page_no`
  - `ordinal`
  - `movie_id`
  - 一覧表示に必要な軽量列
- `page_index`
  - 主要ソートの追加ページ管理用

初期はやりすぎない。

- 主要ソートの先頭ページ
- 必要なら 2 ページ目まで

だけでもよい。

## 5.3 整合性判定

整合性判定は保守的に置く。

一致判定候補:

- DB パス一致
- DB 最終更新時刻一致
- DB ファイルサイズ一致
- Catalog バージョン一致

不一致時:

- warm page 使用を中止
- DB fallback に切り替え
- 背景で Catalog 再生成

## 6. シーケンス設計

## 6.1 Warm start

1. `MainWindow_ContentRendered`
2. `TrySwitchMainDb(...)`
3. `OpenDatafile(...)`
4. `ShutdownCurrentDb()`
5. `LoadDbSystemState(...)`
6. `ApplyDbVisualState(...)`
7. `StartupLoadCoordinator.OpenAsync(...)`
8. `StartupCatalogService.TryLoadWarmPage(...)`
9. `MovieRecordShellFactory.CreateRange(...)`
10. `FilteredMovieRecs` へ first-page 反映
11. `StartupDeferredServiceCoordinator.StartAsync(...)`

## 6.2 Cold start

1. `StartupCatalogService` が Catalog 無し / 不整合を返す
2. `StartupDbPageReader` が DB fallback で先頭ページを読む
3. first-page を表示する
4. 背景で Catalog 再構築を開始する
5. 完了後、次回起動から warm start を使う

## 6.3 ソート変更

1. 現在のページを破棄する
2. `StartupFeedRequest` を新しい `SortId` で作り直す
3. Catalog 先頭ページがあれば使う
4. なければ DB fallback
5. 一覧をページ 1 へ戻す

## 6.4 検索変更

1. debounce
2. 最新要求だけ残す
3. ページ 1 を再取得
4. 追加ページ状態をリセットする

## 6.5 項目選択

1. 一覧選択
2. `MovieDetailsHydrator` が選択中アイテムを特定
3. 重い情報だけ読み込む
4. まだ同じ項目が選択中なら UI へ反映

## 7. 依存整理

## 7.1 `MovieRecords` 依存棚卸し方針

Phase 2 着手前に、`MovieRecords` の利用先を分類する。

分類:

- 起動直後に必須
- 選択時で十分
- 編集時で十分
- 起動経路から切り離してよい

最初に最低限確認する場所:

- `Views/Main/MainWindow.xaml`
- `Views/Main/MainWindow.Selection.cs`
- `ViewModels/MainWindowViewModel.cs`
- タグ編集系
- ブックマーク系
- 下側タブ系

## 7.2 選択状態の扱い

ページング導入後は index 基準の選択を信用しない。

基準キー:

- 第一候補: `Movie_Id`
- 第二候補: `Movie_Path`

選択再解決のルール:

- 追加ページ時は現在キーで再解決する
- ソート変更時は選択解除を基本とする
- 検索変更時は結果内に同一キーがある時だけ再選択する

## 8. ログ設計

カテゴリは既存に寄せる。

- `ui-tempo`
  - `startup window shown`
  - `startup first-page shown`
  - `startup input ready`
  - `startup deferred services started`
  - `startup page append`
- `db`
  - `catalog hit`
  - `catalog miss`
  - `catalog stale`
  - `catalog rebuild start/end`

必須計測点:

- `ContentRendered -> first-page shown`
- `first-page shown -> input ready`
- `input ready -> watcher started`
- `page append elapsed_ms`

- sidecar 不整合時に DB fallback へ落ちる

- 起動直後に Bookmark/Watcher の遅延起動が体感を壊さない
