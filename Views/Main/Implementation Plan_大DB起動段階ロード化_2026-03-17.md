# Implementation Plan: 大DB起動段階ロード化 (2026-03-17)

最終更新日: 2026-03-18

変更概要:
- 大DB起動時の「全件読込してから一覧を出す」方式をやめる
- 起動専用カタログと段階ロードを本命案として定義する
- WhiteBrowser DB を変更せず、外部キャッシュで起動テンポを作る方針を明記する
- Phase 0 / Phase 1 / Phase 2 の初期実装として、DB fallback の first-page / append 骨格が入ったことを反映する
- 起動直後の bulk scan とサムネ常駐再開は、first-page 後の短い遅延起動へ送る方針を反映する
- 起動時 full feed を廃止し、viewport 近傍でだけ page append する方針を反映する
- メインヘッダー件数表示を `表示 / 検出 / 登録` に分け、`登録` は DB 総数で後追い確定する方針を反映する

## 1. 結論

本命は **起動専用カタログ + 段階ロード** である。

起動時は以下に固定する。

1. ウィンドウだけ先に出す
2. 前回状態に対応する先頭 200 件だけを出す
3. それ以外は scroll / viewport 近傍で必要になった時だけ追加する
4. `exists` / サムネ有無 / 全件件数 / Watcher 常駐は初回表示後へ逃がす

現行の「起動時に `movie` 全件を読んで `MovieRecords` へ全展開し、その後に絞り込み・ソート・一覧差し替えを行う」構造は、
大DBでは上限が高すぎる。
ここは局所最適化ではなく、起動経路そのものを分ける。

## 2. なぜこの案が本命か

### 2.1 今の詰まりは一覧前段が支配している

観測済みログでは、起動時の重さは主に以下へ集中している。

- DB 再読込
- `DataRow -> MovieRecords` 全件変換
- `FilteredMovieRecs` 全件差し替え

外側の `VirtualizingWrapPanel` は既に効いているため、
「見えていない項目まで UI 要素化している」のが本命ではない。
本命は、**一覧へ渡す前に全件を抱え込んでいること** である。

### 2.2 WhiteBrowser DB を変えられない

このプロジェクトでは `*.wb` を変更しない。
つまり、起動高速化のために本家 DB へ index を追加する方針は採れない。

そのため、

- `SELECT * FROM movie ORDER BY ...`
- `SELECT ... LIMIT 200`

だけに頼ると、ソート条件によっては内部で重い全件処理が残る。

### 2.3 だから外部の読み取り最適化層が要る

起動を速くしたいなら、起動用のデータ面を分離する必要がある。

採るべき方針は以下。

- 本家 DB は正とする
- 起動と一覧は外部カタログを読む
- カタログは LocalAppData 配下へ持つ
- 変更追従は Watcher / バックグラウンド同期で行う

## 3. 非採用案

### 3.1 現行構造のまま async 化だけを進める案

これは延命には効くが、上限はあまり下がらない。

- 全件読込
- 全件変換
- 全件差し替え

の思想が残る限り、大DBではまた詰まる。

### 3.2 起動時に本家 DB へ `LIMIT 200` を投げるだけの案

これは冷スタート初回の応急処置にはなるが、本命にはならない。
本家 DB 側に十分な index が無い前提では、`ORDER BY ... LIMIT 200` でも速い保証がない。

### 3.3 スプラッシュやローディング画面で隠す案

これは解決ではなく隠蔽である。
workthree の方針である「ユーザー体感テンポ最優先」とも合わない。

## 4. 目標アーキテクチャ

## 4.1 起動専用カタログ

`%LOCALAPPDATA%\\IndigoMovieManager_fork_workthree\\` 配下に、
本家 DB ごとの外部カタログを持つ。

候補:

- `StartupCatalog.sqlite`

役割は分ける。

- `StartupCatalog`
  - ソート、検索、ページ追加に使う読み取り最適化データ
  - warm start 用の先頭ページ保持もここへ含める

初期案では `Snapshot` と `Catalog` を分ける選択肢もあるが、
**初回実装では 1 ファイルへ統合する**。

理由:

- 整合性管理対象を 1 つに減らせる
- warm start 用の先頭 200 件は、Catalog の 1 view / 1 table として保持できる
- 2 ファイル運用にすると「どちらが新しいか」の管理コストが増える

したがって、Phase 3 の本命は以下。

- `StartupCatalog.sqlite`
  - DB 識別情報
  - カタログ世代
  - warm start 用先頭ページ
  - 主要ソートの先頭ページ
  - 必要なら追加ページ用の軽量一覧データ

`Snapshot` 分離は、実測で SQLite 起動コストすら惜しいと分かった時だけ再検討する。

## 4.2 カタログ同期と整合性保証

sidecar 導入の最大リスクは、速度ではなく整合性管理である。
ここは Phase 3 着手前に必ず仕様を固定する。

最低限の検知情報:

- 本家 DB のフルパス
- 本家 DB の最終更新時刻
- 本家 DB のファイルサイズ
- 可能なら `movie` 件数
- カタログ生成時刻
- カタログ生成元のアプリバージョン

初期実装の整合性方針は保守的に置く。

- 一致:
  - Catalog をそのまま使う
- 軽微不一致:
  - warm start の先頭 200 件だけ仮表示しつつ、背景で再同期する
- 重度不一致または破損:
  - Catalog を使わず cold start にフォールバックする
  - 背景で Catalog を再構築する

不整合時の安全ネット:

1. まず UI は出す
2. Catalog が古い/壊れていれば起動専用ページだけ無効化する
3. 本家 DB からページ 1 を読む
4. 背景で Catalog を再生成する

重要:

- sidecar が壊れても起動不能にしない
- sidecar は高速化の補助であり、正本ではない

## 4.3 一覧モデルを 2 層へ分ける

今の `MovieRecords` 1 個に何でも詰める形は、起動には重い。

分ける。

- `MovieListItem`
  - 一覧に最初から要る最小データだけ持つ
  - 例: `movie_id`, `movie_path`, `movie_name`, `last_date`, `score`, `rate`, 軽い表示用サムネキー
- `MovieDetails`
  - 選択時や詳細表示で要る重い情報
  - 例: タグ配列、コメント群、メディア詳細、存在確認結果

起動では `MovieListItem` だけを扱う。

ただし、`MovieRecords` の依存は広い。
そのため Phase 2 の前提として、依存棚卸しを先に行う。

最低限洗う対象:

- 選択
- タグ編集
- 検索
- ブックマーク
- 詳細表示
- コンテキストメニュー
- 下側タブ連携

最初に決めるべきこと:

- 起動直後の一覧に本当に必要なフィールドは何か
- 選択時ロードへ逃がせるフィールドは何か
- `MovieRecords` 互換アダプタで吸収するか、新型へ寄せるか

## 4.4 一覧供給をページング化する

`FilteredMovieRecs` を「全件集合」ではなく、
「今読み込んだ範囲」の集合へ変える。

初期値:

- 先頭 200 件

その後:

- アイドル時に次 300 件
- スクロール終端近辺で追加 300 件
- ソート変更時はページ 1 へ戻す
- 検索変更時もページ 1 へ戻す

ページング導入時の注意:

- 選択状態は index 基準で持たない
- `movie_id` または `movie_path` 基準で再解決する
- ページ追加時に選択がずれない設計へ変える

## 4.5 起動後タスクは段階起動する

初回描画より後に回す。

- `exists` 実確認
- サムネ存在確認
- 正確な件数取得
- Watcher 作成
- rescue worker / queue 常駐

## 5. 起動シーケンスの完成形

## 5.1 Warm start

1. ウィンドウを出す
2. 設定と前回 DB を読む
3. `StartupCatalog` の warm start ページから先頭 200 件を即表示する
4. 裏で `StartupCatalog` の整合性を確認し、差分があれば置き換える
5. 初回表示完了後に各常駐タスクを起動する
6. アイドル時に次ページを読む

## 5.2 Cold start

1. ウィンドウを出す
2. `StartupCatalog` が無ければ空の一覧で開始する
3. `StartupCatalog` が無ければバックグラウンドで構築を始める
4. 先頭ページができ次第、即表示する
5. その後に次ページと常駐タスクを起動する

重要:

- cold start では「最初の一覧が出るまで何も触れない」状態を作らない
- 最悪でも「シェルは出ていて、今は初回カタログ生成中」と分かる状態にする

## 6. フェーズ案

## Phase 0: 観測固定

目的:
- 起動改善を ms 単位で比較できるようにする

追加ログ:

- `ui-tempo start-window-shown`
- `ui-tempo first-page-shown`
- `ui-tempo input-ready`
- `ui-tempo watcher-started`
- `ui-tempo page-load count=... elapsed_ms=...`

既存ログ体系へ寄せる。

- ログカテゴリは原則 `ui-tempo`
- DB 起因の補助情報は `db`
- 起動ライフサイクル補助は既存カテゴリと衝突しない範囲で追加する

完了条件:
- 大DBで「どこまでを起動完了とみなすか」がログで固定される

## Phase 1a: 起動経路の最小分離

目的:
- `ContentRendered -> OpenDatafile -> FilterAndSort 全件読込` の依存だけ先に壊す

実施内容:

- 起動専用の `StartupLoadCoordinator` を作る
- 既存 `OpenDatafile` とは別に `OpenStartupFeed` を作る
- 起動では `FilterAndSortAsync` を直接呼ばない
- 起動時は空一覧 or warm start ページから始める

この段階では、以下はまだ据え置きでもよい。

- Bookmark タブ再読込
- Watcher 作成
- queue 走査

完了条件:
- 起動で全件 `FilterAndSort` が直列に走らない

## Phase 1b: 常駐処理の後ろ倒し

目的:
- 起動主経路から一覧と無関係な重い処理を外す

実施内容:

- Bookmark タブ再読込を first-page 後へ移す
- Watcher 作成を first-page 後へ移す
- queue 走査と rescue worker 起動を first-page 後へ移す

完了条件:
- 起動主経路に一覧表示と無関係な常駐処理が残らない

## Phase 2: 先頭 200 件ロード

目的:
- 起動時の見え方を最速化する

実施内容:

- `MovieListItem` を新設する
- `MovieRecords` 依存マップを作る
- `ItemsSource` をページ追加型へ変える
- 先頭 200 件だけを出す
- `exists` / サムネ存在確認は後追いにする

完了条件:
- 起動後すぐに 200 件だけ見える
- 全件件数が未確定でも操作できる

## Phase 3: 起動専用カタログ導入

目的:
- 本家 DB の重いソート依存を起動経路から外す

実施内容:

- LocalAppData 配下に sidecar カタログを作る
- DB パス、更新日時、件数、先頭ページ群を持つ
- 主要ソートの先頭ページを事前保持する
- 必要なら検索用の軽量インデックスも持つ

事前に決めること:

- 事前保持する主要ソートの対象一覧
- 主要ソート外のフォールバック
  - 本家 DB への `LIMIT` 付き問い合わせ
  - または Catalog 内簡易ソート

初期方針:

- よく使う主要ソートだけ Catalog 最適化対象へ入れる
- 低頻度ソートはフォールバックを許す
- 全ソートを最初から sidecar 最適化しようとしない

完了条件:
- warm start で本家 DB を全件触らずに先頭ページが出る

## Phase 4: 追加ページと lazy detail

目的:
- 初回表示以外の機能も段階化する

実施内容:

- 終端近辺での次ページ追加
- 選択時だけ詳細情報を hydrate
- タグ配列や重い文字列整形は選択時または詳細表示時へ逃がす

完了条件:
- 起動後に重い詳細情報が一括で流れ込まない

## Phase 5: 常駐処理の後ろ倒し

目的:
- 起動の主経路から常駐タスクを外す

実施内容:

- Watcher 起動を first-200 後へ移す
- queue 常駐を first-200 後へ移す
- rescue worker も first-200 後へ移す

完了条件:
- 起動直後に一覧と無関係なバックグラウンド仕事で詰まらない

## 7. 具体方針

## 7.1 起動時に絶対やらないこと

- `movie` 全件を `DataTable` へ読む
- 全件を `MovieRecords` へ変換する
- 全件へ `Path.Exists` を打つ
- 全件件数確定待ちで UI を止める
- Watcher 完了待ちで UI を止める

## 7.2 起動時にやってよいこと

- `StartupCatalog` の warm start ページ即表示
- 先頭 200 件だけの表示
- 未確定情報をプレースホルダで出す
- 背景タスクのキャンセル付き段階起動

## 7.3 検索とソート

検索とソートは「全件配列に LINQ を当てる」のをやめる。

方向:

- ソート変更
  - データ源へ再問い合わせしてページ 1 を出す
- 検索変更
  - データ源へ再問い合わせしてページ 1 を出す
- 連続入力中
  - debounce して最新要求だけ生かす

## 7.4 件数表示

起動時は件数確定を必須にしない。

現仕様:

- `表示` は `DbInfo.SearchCount`
- `検出` は `MovieRecs.Count`
- `登録` は `movie` テーブルの `COUNT(*)`
- `登録` はバックグラウンドで後追い確定し、first-page 表示を止めない

この割り切りは体感を守りつつ、件数の意味を UI 上で分離できる。

## 8. 本命案と代替案の比較

### A. 現行改善継続

長所:
- 差分が小さい

短所:
- 上限が高いまま
- 大DBではまた詰まる

判定:
- 本命ではない

### B. 本家 DB 直 paging

長所:
- 実装は比較的軽い

短所:
- 本家 DB 側の並び替え性能に依存する
- `*.wb` を変えない前提では伸び代に限界がある

判定:
- Phase 1, 2 のつなぎとしては有効

### C. 起動専用カタログ + 段階ロード

長所:
- warm start を最速化できる
- 起動と常用一覧の両方に効く
- `*.wb` 非変更ルールを守れる

短所:
- 実装規模は大きい
- sidecar の整合性管理が要る

判定:
- 本命

## 9. リスク

- sidecar の整合性管理コスト
- `MovieRecords` 依存が広く、一覧軽量化の影響範囲が大きい
- Phase 1 は `BootNewDb` の責務分解が必要で、単独でも膨らみやすい
- ページ追加時に選択状態やスクロール位置の扱いを誤ると体感が悪化する
- 主要ソートの対象選定を誤ると、期待した warm start 効果が出ない

## 10. 成功条件

- warm start の first-200 表示が 1 秒台に入る
- cold start でもウィンドウ表示は即時
- 起動時に全件 `MovieRecords` 生成をしない
- 起動時に全件 `Path.Exists` をしない
- 起動直後に操作不能時間を作らない
- ソート変更と検索変更でも「まず先頭ページ」が先に見える

## 11. 最初の実装順

最初の 1 本は以下に固定する。

1. Phase 0 の計測固定
2. Phase 1a の起動経路最小分離
3. Phase 2 の先頭 200 件ロード
4. Phase 1b の常駐処理後ろ倒し
5. Phase 3 の sidecar 導入

理由:

- まず「起動で全件を触らない」骨格を作る
- その後に sidecar を入れて warm start を完成させる

sidecar から入ると設計は綺麗でも着手が重い。
先に起動経路を分離し、ページング前提の UI へ変えてから sidecar を入れる方が安全で、体感改善も早い。
