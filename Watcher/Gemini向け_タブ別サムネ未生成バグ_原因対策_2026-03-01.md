# Gemini向け: タブ別サムネ未生成バグ 原因・対策・再発防止（2026-03-01）

## 0. 先に結論
- 今回の不具合は、**タブ別サムネイルの有無判定をしていない**ことが本質。
- `tab0` など他タブの画像が1枚でもあると、`tab2/tab3/tab4` の生成候補が走査段階で落とされる。
- その結果、UIでは初期画像のまま大量に残る。

## 1. 症状
- 監視ログは動いているが、`already exists in db` で追加がスキップされる。
- `tab2`（`160x120x1x1`）や `tab3`（`56x42x5x1`）が未生成なのに、キューへ入らない。
- QueueDB集計で `TabIndex=0` は多いが、`TabIndex=2/3/4` が極端に少ない（または0）。

## 2. 根本原因（コード上）
- 対象: `Watcher\MainWindow.Watcher.cs`
- 問題箇所:
  - `ScanFolderWithStrategyInBackground(...)` 内
  - `ScanFolderInBackground(...)` 内
- 現在の判定:
  - `existingThumbBodies.Contains(fileBody)` で候補除外している。
  - `existingThumbBodies` は「サムネ画像の本体名（body）」だけの集合。
- 何がダメか:
  - `body` にはタブ情報が無い。
  - 例えば `tab0` の `xxx.#hash.jpg` があるだけで、`tab2` 欠損でも `xxx` が既存扱いになる。
  - 本来は `TabInfo` ごとの出力先とファイル名で判定すべき。

## 3. 直す方針（最小変更）
1. 走査段階の `existingThumbBodies` 除外を外す（または無効化）。
2. 既存DB行ごとの処理で、**現在タブの期待サムネイルパス**だけを見てキュー投入可否を決める。
3. つまり判定は必ず以下で統一:
   - `TabInfo tbi = new(snapshotTabIndex, snapshotDbName, snapshotThumbFolder);`
   - `saveThumbFileName = Path.Combine(tbi.OutPath, $"{fileBody}.#{hash}.jpg");`
   - `Path.Exists(saveThumbFileName)` が false なら投入。

## 4. 実装タスク（Gemini実装用）
- [ ] `ScanFolderWithStrategyInBackground` の `existingThumbBodies.Contains(fileBody)` を撤去
- [ ] `ScanFolderInBackground` の `existingThumbBodies.Contains(fileBody)` を撤去
- [ ] `CheckFolderAsync` の `foreach (movieFullPath in scanResult.NewMoviePaths)` で、現行の `saveThumbFileName` 判定を唯一の投入判定にする
- [ ] `skip db insert (already exists in db)` の分岐でも、`movie_id` を補完したうえで同じ判定を通して投入可能にする
- [ ] ログを追加:
  - `enqueue by missing-tab-thumb: tab={snapshotTabIndex}, movie='{path}'`
  - `skip enqueue by existing-tab-thumb: tab={snapshotTabIndex}, movie='{path}'`

## 5. 既存データ救済（埋め戻し）
- [ ] `tab2` 欠損を既存DB全件で再キューするワンショットを用意
- [ ] 100件単位で `TryEnqueueThumbnailJob` してUI固まりを避ける
- [ ] 完了後にQueueDBで `TabIndex=2` の `Status=2` が増えることを確認

## 6. 回帰テスト観点
- [ ] 同一動画で `tab0` のみ存在、`tab2` 欠損の状態を作る
- [ ] 監視走査1回で `tab2` がキュー投入されること
- [ ] `tab2` 生成後は同じ走査で再投入されないこと
- [ ] `tab3/tab4` でも同じ挙動であること

## 7. 次にやらかさないための小言（重要）
- 「重複排除キー」に仕様にない次元（今回だとタブ）を落とすな。
- 先に最適化（Everything連携、body集合）を入れる前に、正しさの軸を固定しろ。
- 判定に使うキーは「最終出力単位」に合わせる。今回は `tab + body + hash + outPath` が正。
- ロジック追加時は最低1本、**欠損タブ再生成の回帰テスト**を先に書け。

## 8. 実装完了条件
- `tab0` が既にあっても `tab2/tab3/tab4` 欠損はキュー投入される。
- 監視ログに「タブ別欠損で投入」の行が出る。
- 初期画像のまま残っていた件数が減る（対象フォルダで目視確認）。
