# 提案書: WebView2スキン完全互換向け サムネ表示層と生成層の境界 2026-04-01

最終更新日: 2026-04-01

変更概要:
- WebView2 による WhiteBrowser スキン完全互換計画に対し、サムネサイズ自由化の観点から境界案を提示する
- 「表示層」は WebView2 / HTML / CSS 側、「生成層」は既存サムネ生成・保存・解決側と定義した
- 生成層の責務は本線側で持ち、I/F は WebView2 側との共同管理で固定する前提へ更新した
- `thumbRevision`、`dbIdentity + movieId`、寸法 DTO 必須化を正式契約案へ昇格した
- `wb.getInfo / wb.getInfos / onUpdateThum` へ持ち込むべき v1 契約案を整理した

## 1. この提案書の目的

本書は、`WebView2によるWhiteBrowserスキン完全互換` 計画に対して、
サムネサイズ自由化の影響を踏まえた **境界案** を提示するための提案書である。

結論を先に書く。

- **表示サイズの正本は WebView2 側に置く**
- **サムネ生成・保存・実体解決の正本は既存本線側に置く**
- **I/F の正本は WebView2 側単独ではなく共同管理で固定する**

本書は、そのための叩き台である。

## 2. 前提

### 2.1 すでに決まっていること

- 外部 WhiteBrowser スキンは WebView2 側で描画する方向で進んでいる
- `WhiteBrowserSkinConfig` は `thum-width / thum-height / thum-column / thum-row` を読める
- 既存本線側には、サムネ生成・Queue・FailureSync・同名画像優先表示 / 取り込みの仕組みがある

### 2.2 今回ぶつかる論点

WhiteBrowser の `thum-*` は本来 **表示サイズ** の話である。
一方、現行 IndigoMovieManager ではサムネサイズが次へ結びついている。

1. 生成サイズ
2. 保存先フォルダ名
3. 既存サムネ探索
4. Queue / FailureSync / Watcher の成功判定

このため、WebView2 側がサイズ自由化を取り込む時に、
生成層まで一緒に動かす設計にすると手戻りが大きい。

## 3. 境界の提案

## 3.1 表示層

表示層は **WebView2 / HTML / CSS / JS** 側と定義する。

ここが持つ責務は次である。

- `thum-width / thum-height / thum-column / thum-row` の解釈
- 何pxで見せるか
- 何列何行で並べるか
- タイトルや余白の見え方
- DOM 更新
- 遅延表示、可視範囲ロードなどの表示最適化

つまり、**「どう見せるか」** は WebView2 側の責務に寄せる。

## 3.2 生成層

生成層は、既存本線のサムネ管理側と定義する。

ここが持つ責務は次である。

- 既存 generated jpg の生成
- 同名画像の source-image-import
- 同名画像 UI fallback を含む実体解決
- ERROR / placeholder / nofile の扱い
- 保存先フォルダと metadata の整合
- Queue / FailureSync / Watcher / Rescue との整合

つまり、**「何を資産として持つか」** は生成層の責務に寄せる。

## 3.3 やってはいけない混線

次は非推奨である。

- WebView2 側が生成フォルダ規約を自前で決める
- WebView2 側が `movie path -> thumb path` を独自合成する
- `thum-width` を受けて即 custom 管理サムネ生成へ進む

ここをやると、表示層が生成層の責務を持ち始めて破綻する。

## 4. 推奨 I/F 契約案

## 4.1 基本方針

WebView2 側へは **「表示に使ってよい解決済みサムネ情報」** を渡す。
WebView2 側で `movie path` から自力でサムネパスを導出させない。

## 4.2 正式 DTO v1 案

`wb.getInfo / wb.getInfos` が、各動画について次の情報を返す案を推奨する。

### 必須

- `dbIdentity`
- `movieId`
- `recordKey`
- `moviePath`
- `movieName`
- `thumbUrl`
- `thumbRevision`
- `thumbSourceKind`
- `thumbNaturalWidth`
- `thumbNaturalHeight`
- `thumbSheetColumns`
- `thumbSheetRows`
- `exists`

### 推奨

- `thumbUpdatedUtc`
- `thumbMimeType`

### `thumbSourceKind` 候補

- `managed-thumbnail`
- `source-image-direct`
- `source-image-imported`
- `error-placeholder`
- `missing-file-placeholder`

### 各項目の意図

- `dbIdentity`
  - 現在の DB を一意に識別するキー
- `movieId`
  - DB 内での動画 ID
- `recordKey`
  - 更新通知で使う主キー。正式に `dbIdentity + movieId` の組を表す
- `thumbUrl`
  - WebView2 側がそのまま `<img src>` に使う URL
- `thumbRevision`
  - キャッシュ無効化用の改訂番号
- `thumbNaturalWidth / thumbNaturalHeight`
  - 実画像のピクセルサイズ
- `thumbSheetColumns / thumbSheetRows`
  - WB互換シートの分割数

### キャッシュ無効化の正式方針

`thumbUrl` 更新契約は、次のどちらかではなく **両方を持つ** 形を正式案とする。

1. DTO に `thumbRevision` を必須で持つ
2. `thumbUrl` は `?rev={thumbRevision}` を付与した最終 URL として返す

この形にしておけば、

- WebView2 側は URL だけ見ても再読込できる
- JS 側 state 比較では `thumbRevision` を明示比較できる
- 将来 URL 生成方式が変わっても revision 比較を残せる

### 更新キーの正式方針

更新通知の主キーは `movieId` 単独ではなく、**`recordKey = dbIdentity + movieId`** を正式採用する。

`movieId` は DB をまたぐと衝突しうるため、`onUpdateThum` や差分更新の対象キーに単独採用してはいけない。

本提案では、

- DTO には `dbIdentity` と `movieId` を個別に持たせる
- 更新通知と DOM 再利用の主キーは `recordKey` を使う

を推奨する。

`dbIdentity` の具体実装は WebView2 計画と本線側で最終決定するが、少なくとも
「DB 切替や再登録をまたいでも別レコードへ刺さらない」ことを満たす必要がある。

この案の利点は、WebView2 側が

- `thumbUrl` をそのまま `<img>` に入れればよい
- `thumbSourceKind` で CSS や表示分岐を変えられる
- `thumbRevision` と `recordKey` で安全に差分更新できる
- 生成層の内部規約を知らなくてよい

ことである。

## 4.3 寸法 DTO を必須にする理由

`thumbNaturalWidth / Height` と `thumbSheetColumns / Rows` は任意ではなく、**v1 から正式必須** を提案する。

理由は次のとおり。

- WB スキン互換では「何px の何分割資産か」を後段で参照したくなる
- CSS や JS 側で aspect 比や sheet 前提の描画調整をしたくなる
- 後付けオプションにすると、初期 DTO と更新 DTO で契約差が出やすい

したがって本提案では、

- `thumbNaturalWidth`
- `thumbNaturalHeight`
- `thumbSheetColumns`
- `thumbSheetRows`

を **初期 DTO から正式項目として固定** する。

## 4.4 非推奨案 B

旧 WhiteBrowser 互換に寄せて、`wb.getThumDir()` だけ返し、
スキン側がサムネ相対パスを自前計算する案は非推奨とする。

理由は次のとおり。

- 同名画像 direct 表示を表現しにくい
- placeholder と managed thumb の区別が弱い
- 今後 custom 生成や detail 差分が入るとすぐ破綻する

`wb.getThumDir()` 自体は互換用途で残してよいが、
**主経路の画像解決は `thumbUrl` 直返し** を推奨する。

## 5. onUpdateThum 系の提案

サムネ更新通知は、少なくとも次を渡せる形が望ましい。

- `dbIdentity`
- `movieId`
- `recordKey`
- `moviePath`
- `thumbUrl`
- `thumbRevision`
- `thumbSourceKind`
- `thumbNaturalWidth`
- `thumbNaturalHeight`
- `thumbSheetColumns`
- `thumbSheetRows`

推奨するキーは `recordKey` である。
`moviePath` は rename で揺れうるため、同一性の主キーには弱い。

### 更新通知の推奨形

`onUpdateThum(recordKey, thumbUrl, thumbRevision, thumbSourceKind, sizeInfo)` 相当を基準形とする。

`thumbUrl` だけの更新通知では、古い画像キャッシュを安全に追い出せない。
このため、`thumbRevision` は更新通知にも必須で含める。

## 6. こちらが持つ責務

本線側、つまり「生成層」はこちらで持つ。

具体的には次を担当する前提でよい。

1. 既存サムネ生成
2. 同名画像優先の資産化
3. 同名画像 direct fallback
4. placeholder / error marker / success 判定
5. WebView2 側へ返す `thumbUrl` 候補の最終解決

つまり WebView2 側へは、
**「解決済みの表示候補」** を返すところまでをこちらの責務に寄せる案である。

## 7. 契約管理の提案

I/F の最終決定権を WebView2 側単独へ寄せすぎるのは避けたい。
この契約は表示層だけでなく、生成層の責務と実装も縛るからである。

したがって本提案では、I/F 正本を次の共同管理とする案を出す。

1. PM 管理の正本契約として `skin/` 配下の計画書へ固定する
2. WebView2 側と生成層側の両方がレビューして確定する
3. 片側だけで破壊的変更を決めない

### 共同管理で固定すべき論点

1. `wb.getInfo / wb.getInfos` に `thumbUrl` を正式採用するか
2. `thumbRevision` を必須採用するか
3. `recordKey = dbIdentity + movieId` を正式主キーにするか
4. 寸法 DTO を v1 から正式採用するか
5. `wb.getThumDir()` を主経路ではなく互換用途へ落とすか

本書としての推奨は次である。

- `thumbUrl`: 採用
- `thumbRevision`: 採用
- `thumbSourceKind`: 採用
- 更新キー: `recordKey` 正式採用
- 寸法 DTO: v1 から必須採用
- `getThumDir`: 互換用途へ縮小

## 8. フェーズ提案

## Phase A: WebView2 側が表示サイズだけを正本化

- `thum-*` は WebView2 側でそのまま解釈
- サムネ実体は既存 generated / source-image / placeholder を再利用
- custom 生成には入らない

## Phase B: `thumbUrl` 直返し I/F を確立

- `wb.getInfo / wb.getInfos` へ `thumbUrl` を載せる
- `thumbRevision` と寸法 DTO を同時に載せる
- WebView2 側は path 合成をやめる

## Phase C: 更新通知の薄い同期

- `onUpdateThum(recordKey, thumbUrl, thumbRevision, thumbSourceKind, sizeInfo)` 相当を整える
- まずは再読込でもよく、差分更新は後段

## 9. この提案の利点

1. WebView2 側は表示自由化に集中できる
2. 生成層の複雑さを表示層へ漏らさずに済む
3. `thumbRevision` と `recordKey` で更新事故を抑えられる
4. 同名画像 direct / imported / placeholder を一貫して扱える
5. 後から custom 生成が必要になっても I/F 破壊を抑えやすい

## 10. 今回の提案結論

本線側からの提案は、次の一文に尽きる。

**WebView2 側は「表示サイズと見せ方」を正本にし、生成層は「表示に使ってよい解決済みサムネ情報」を返す。**

そのうえで、

- 生成層の実装は本線側で持つ
- I/F の正本は WebView2 側単独ではなく共同管理で固定する

という役割分担を提案する。
