# 調査結果 skin切り替え重さの原因 2026-04-12

最終更新日: 2026-04-13

変更概要:
- `skin` 切り替え時の重さについて、現在コードから読める処理経路を整理した
- 外部 skin と標準 skin を分けて、どこで重い仕事が走るかを優先度順に棚卸しした
- `FilterAndSort` のような全件再読込は、今回の skin 切り替え経路には入っていないことを明記した
- 原因調査に対して、そのまま着手できる改善プランを優先度付きで追記した
- `WebView2` が UI スレッド前提であり、別プロセス実行と混同しやすい点を補足した
- `SQLite` の別スレッド化余地と効果を、現行コード再確認ベースで追記した
- DB 分離の詳細実装計画を別紙 `Implementation Plan_skin切り替え高速化_DB保存分離先行_2026-04-13.md` として作成した

## 1. 目的

ユーザー体感として「skin 切り替え時に重い」原因を、推測ではなく現在コードの実処理から説明できる形にする。

今回の調査対象は次である。

- built-in skin 切り替え
- 外部 skin 切り替え
- 外部 skin から built-in への復帰

## 2. 結論

重さの本命は 2 系統ある。

1. 外部 skin
   - WebView2 host の再準備
   - `handleSkinLeave` と `NavigateToString` を含む再 navigate
   - skin カタログの再走査
   - `system` / `profile` への同期 SQLite I/O
2. built-in skin
   - タブ切り替えに伴う先頭選択・詳細更新
   - アクティブタブ側の画像 decode
   - 詳細ペインの詳細サムネ探索

一方で、今回の skin 切り替え経路には `FilterAndSort()` による movie 全件再読込は入っていない。

## 3. 切り替え入口

skin 切り替えの入口は `WhiteBrowserSkin/MainWindow.Skin.cs:61` の `ApplySkinByName(...)` である。

ここから実際には次が走る。

1. `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:86`
   - skin 名解決
   - `MainVM.DbInfo.Skin` 更新
   - 表示タブ決定
   - 必要なら DB 永続化
2. `Views/Main/MainWindow.WebViewSkin.cs:63`
   - `DbInfo.Skin` の `PropertyChanged` を受けて external skin refresh を queue
3. `WhiteBrowserSkin/MainWindow.Skin.cs:71`
   - 成功時に `QueueExternalSkinHostRefresh("apply-skin")` を追加で queue

この時点で、skin 切り替え 1 回に対して refresh 要求が 2 系統積まれる構造になっている。

## 4. 主原因

### 4.1 外部 skin は毎回ほぼフル navigate している

外部 skin 側の重さの本命はここである。

- `Views/Main/MainWindow.WebViewSkin.cs:320`
- `WhiteBrowserSkin/Host/WhiteBrowserSkinHostControl.xaml.cs:48`
- `WhiteBrowserSkin/Runtime/WhiteBrowserSkinRuntimeBridge.cs:24`

外部 skin 表示では `TryPrepareExternalSkinHostAsync(...)` から `hostControl.TryNavigateAsync(...)` が呼ばれ、内部で次を毎回実行している。

- `TryEnsureAttachedAsync(...)`
- `HandleSkinLeaveAsync()`
- `BuildInitialDocument(...)`
- `NavigateToStringAsync(...)`

特に初回 attach 時は `EnsureCoreWebView2Async(...)` まで入るため、初回切り替えは重くなりやすい。

さらに built-in へ戻す fallback 側でも `WhiteBrowserSkin/Host/WhiteBrowserSkinHostControl.xaml.cs:83` の `ClearAsync()` が走り、空 HTML への遷移を待つ。

つまり外部 skin 切り替えは、

- 旧 skin の leave
- 新 skin HTML の再構築
- WebView2 への再 navigate

を毎回ほぼフルでやっている。

### 4.2 refresh の世代破棄が「重い準備の後」なので無駄仕事が残る

- `Views/Main/ExternalSkinHostRefreshScheduler.cs:34`
- `Views/Main/ExternalSkinHostRefreshScheduler.cs:61`
- `Views/Main/MainWindow.WebViewSkin.cs:151`

scheduler は queue を畳んでいるが、古い世代を捨てる判定は `RefreshExternalSkinHostPresentationAsync(...)` の後半にある。

つまり、古い refresh が

- definition 解決
- host 準備
- navigate

まで進んだ後で「古いので捨てる」ことがありうる。

連打時や `PropertyChanged` と明示 queue が重なった時は、この無駄仕事が体感を悪くする。

### 4.3 skin 切り替え 1 回で refresh が二重に積まれる

- `WhiteBrowserSkin/MainWindow.Skin.cs:33`
- `DB/DBInfo.cs:51`
- `Views/Main/MainWindow.WebViewSkin.cs:63`
- `WhiteBrowserSkin/MainWindow.Skin.cs:71`

`WhiteBrowserSkinOrchestrator.ApplySkinByName(...)` は `setCurrentSkinNameToViewModel(...)` で `MainVM.DbInfo.Skin` を更新する。
`DBInfo.Skin` setter は `PropertyChanged(nameof(Skin))` を発火する。

その結果 `MainWindow.WebViewSkin` 側で `QueueExternalSkinHostRefresh("dbinfo-Skin")` が積まれる。

その直後に `MainWindow.Skin.ApplySkinByName(...)` がさらに `QueueExternalSkinHostRefresh("apply-skin")` を積んでいる。

scheduler が直列化はしてくれるが、

- generation が 2 回進む
- 先に走った refresh が stale になりやすい
- 前項 4.2 の無駄仕事を起こしやすい

という構造になっている。

### 4.4 skin カタログを毎回再走査している

- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:70`
- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:168`
- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:177`
- `WhiteBrowserSkin/WhiteBrowserSkinCatalogService.cs:34`
- `WhiteBrowserSkin/WhiteBrowserSkinCatalogService.cs:51`
- `WhiteBrowserSkin/WhiteBrowserSkinCatalogService.cs:151`
- `WhiteBrowserSkin/WhiteBrowserSkinCatalogService.cs:189`

skin 名解決や一覧取得のたびに `ReloadAvailableSkinDefinitions()` が走り、`WhiteBrowserSkinCatalogService.Load(...)` が毎回ディレクトリ列挙と HTML 読み込みを行う。

Load の中では各外部 skin について、

- `Directory.EnumerateDirectories(...)`
- skin HTML の探索
- `File.ReadAllBytes(...)`
- config 解析

まで実行している。

外部 skin 数が多いほど、切り替え前段のコストが増える。

### 4.5 現在の最小ヘッダードロップダウンも外部 skin 一覧を毎回取り直す

- `Views/Main/MainWindow.WebViewSkin.Chrome.cs:43`
- `Views/Main/MainWindow.WebViewSkin.Chrome.cs:52`
- `Views/Main/MainWindow.WebViewSkin.Chrome.cs:65`
- `WhiteBrowserSkin/MainWindow.Skin.cs:51`
- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:70`

現在の minimal chrome は表示同期時に `SyncExternalSkinMinimalSkinSelector(...)` を呼び、その中で `GetAvailableSkinDefinitions()` を叩いている。

つまり外部 skin 表示の更新時には、

- refresh 本体で definition 解決
- ヘッダー同期で外部 skin 一覧再取得

の 2 回目の catalog 走査が入りうる。

今回のドロップダウン化以後は、この分のコストも上乗せされている。

### 4.6 SQLite の読み書きが UI 経路上で同期実行される

- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:98`
- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:101`
- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:218`
- `DB/SQLite.cs:952`
- `DB/SQLite.cs:860`
- `DB/SQLite.cs:919`

skin 切り替えでは次の DB I/O が同期で入る。

- 外部 skin の前回タブ復元用 `SelectProfileValue(...)`
- 保存時の `UpsertSystemTable(...)`
- 保存時の `UpsertProfileTable(...)`

特に `UpsertSystemTable(...)` は

1. 存在確認の SELECT
2. その後の UPDATE または INSERT

と接続をまたいで 2 段で走る。

通常は小さい I/O だが、DB が遅い場所にある時やロック競合時は UI 体感へ乗りやすい。

### 4.7 built-in skin でもタブ切り替えに伴う描画仕事が走る

- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:99`
- `UpperTabs/Common/MainWindow.UpperTabs.TabIdentity.cs:120`
- `UpperTabs/Common/MainWindow.UpperTabs.SelectionFlow.cs:272`

built-in skin は「名前だけ変える」ではなく、対応する上側タブへ切り替えている。

この時に走るのは次である。

- `CurrentTabIndex` 更新
- 先頭選択
- 詳細ペイン更新
- visible range 更新

そのため built-in skin でも、表示先タブが変わればそれなりに仕事量がある。

### 4.8 タブ切り替え直後の画像 decode が重い

- `UpperTabs/Common/UpperTabImageSourceConverter.cs:25`
- `UpperTabs/Common/UpperTabImageSourceConverter.cs:32`
- `Infrastructure/Converter/NoLockImageConverter.cs:74`
- `Infrastructure/Converter/NoLockImageConverter.cs:107`
- `Infrastructure/Converter/NoLockImageConverter.cs:483`
- `Infrastructure/Converter/NoLockImageConverter.cs:498`

アクティブになったタブでは `UpperTabImageSourceConverter` が有効化され、画像ファイルの decode が走る。

`NoLockImageConverter` ではキャッシュが効けば軽いが、未キャッシュ画像は実ファイルを開いて `BitmapImage` を作る。

一覧件数が多い時や、別タブで decode サイズが変わる時は、この部分が built-in skin 切り替えの体感に出る。

### 4.9 詳細ペインの詳細サムネ探索が追加で乗る

- `UpperTabs/Common/MainWindow.UpperTabs.SelectionFlow.cs:23`
- `BottomTabs/Extension/MainWindow.BottomTab.Extension.cs:88`
- `BottomTabs/Extension/MainWindow.BottomTab.Extension.cs:111`
- `BottomTabs/Extension/MainWindow.BottomTab.Extension.DetailThumbnail.cs:148`
- `BottomTabs/Extension/MainWindow.BottomTab.Extension.DetailThumbnail.cs:236`
- `BottomTabs/Extension/MainWindow.BottomTab.Extension.DetailThumbnail.cs:459`

タブ切り替え後は選択中レコードの詳細ペイン更新も走る。

詳細ペインが有効な時は `EnsureActiveExtensionDetailThumbnail(...)` が動き、

- 既存詳細サムネ探索
- layout 候補フォルダの列挙
- 条件によっては `Directory.EnumerateFiles(outPath, "*.jpg")` による body scan
- 必要なら詳細サムネ生成キュー投入

まで進む。

一覧本体よりは小さいが、切り替え直後の追加コストとして無視できない。

## 5. 今回の切り替え経路で「起きていないこと」

### 5.1 `FilterAndSort()` による全件再読込は今回の skin 切り替え経路には見えていない

- `Views/Main/MainWindow.xaml.cs:1461`
- `Views/Main/MainWindow.xaml.cs:1467`
- `Views/Main/MainWindow.xaml.cs:1512`
- `Views/Main/MainWindow.xaml.cs:1933`

`FilterAndSortAsync(...)` は確かに重い処理で、

- DB 再取得
- `SetRecordsToSource(...)`
- サムネ出力先 lookup 構築
- 表示コレクション差し替え

まで含む。

ただし、今回追った `ApplySkinByName(...)` の経路からは `FilterAndSort()` 呼び出しはつながっていない。

したがって、skin 切り替えの重さを説明する主因はまず `FilterAndSort` ではない。

## 6. 体感別の見立て

### 6.1 外部 skin だけ明確に重い場合

本命は次の組み合わせである。

- WebView2 再準備
- `handleSkinLeave` + `NavigateToString`
- catalog 再走査
- 同期 SQLite I/O
- 二重 refresh queue

### 6.2 built-in skin でも重い場合

本命は次の組み合わせである。

- タブ切り替え後の画像 decode
- 詳細ペイン更新
- 詳細サムネ探索

## 7. 優先度付きまとめ

優先度順に並べると次である。

1. 外部 skin の WebView2 再 navigate が重い
2. refresh 要求が二重に積まれ、stale 破棄が後段なので無駄仕事が出る
3. skin カタログを毎回再走査している
4. 同期 SQLite I/O が UI 経路上にある
5. built-in skin でもタブ切り替え後の画像 decode が重い
6. 詳細ペインの詳細サムネ探索が追加負荷になる

## 8. 次に見るべき点

改善前に最低限ログを足すなら、次の 4 点が効果的である。

1. `TryPrepareExternalSkinHostAsync(...)` の総時間
2. `WhiteBrowserSkinCatalogService.Load(...)` の総時間
3. `PersistCurrentSkinState(...)` と `SelectProfileValue(...)` の時間
4. `HandleUpperTabSelectionChangedCore()` の時間

これで「外部 skin 側が支配的か」「標準タブ側 decode が支配的か」を定量で切り分けやすくなる。

## 9. 改善プラン

今回の調査結果から、改善は「まず効くもの」と「構造的に効くもの」を分けて進めるのが安全である。

特に外部 skin は、いきなり大きく作り替えるより、

- 無駄な二重 refresh を止める
- catalog の再走査を減らす
- 計測を入れて支配要因を固定する

の順で進めた方が、体感改善と回帰防止を両立しやすい。

### 9.1 第1段階 すぐ効く改善

最優先は次の 3 点である。

1. refresh の二重 queue をやめる
2. stale 判定をできるだけ前段へ寄せる
3. skin catalog をメモリキャッシュする

具体的には次を行う。

- `DbInfo.Skin` の `PropertyChanged` 経由と `ApplySkinByName(...)` 明示 queue のどちらを正本にするか決め、外部 skin refresh 起点を 1 本化する
- `RefreshExternalSkinHostPresentationAsync(...)` の前半で generation を再確認し、古い要求の host 準備や navigate をなるべく始めない
- `WhiteBrowserSkinCatalogService.Load(...)` の結果をキャッシュし、同一プロセス内では毎回ディレクトリと HTML を総なめしない
- minimal chrome の skin ドロップダウンも同じ cache を参照し、一覧同期のたびに再走査しない

この段階の狙いは、設計を大きく変えずに「切り替え 1 回あたりの無駄仕事」を減らすことである。

### 9.2 第2段階 計測と保存処理の整理

第1段階と並行して、切り替え経路に短い時間計測ログを入れる。

対象は次である。

1. `TryPrepareExternalSkinHostAsync(...)`
2. `WhiteBrowserSkinCatalogService.Load(...)`
3. `PersistCurrentSkinState(...)`
4. `SelectProfileValue(...)`
5. `HandleUpperTabSelectionChangedCore()`

この計測で支配要因を確認したうえで、保存処理は次の方針で軽くする。

- `system` / `profile` への保存は、表示成立後に遅延させられるものを切り分ける
- 少なくとも「表示に必須でない保存」は UI 直列経路から外す
- `UpsertSystemTable(...)` の存在確認と更新処理は、必要ならまとめて見直す

この段階の狙いは、体感に効いている I/O を「見える化」し、外せる同期処理を減らすことである。

### 9.3 第3段階 外部 skin を HTML 差し替え中心へ寄せる

外部 skin 同士の切り替えについては、「WebView2 を使い回しつつ新しい document を流し込む」方向へ寄せる余地がある。

ただし、ここで言う HTML 差し替えは `document.body.innerHTML` のような雑な差し替えではなく、次を前提にする。

- 既存 `CoreWebView2` は再生成しない
- skin ごとの初期 document は組み直す
- 差し替えは `NavigateToString` 系で新しい document を適用する
- `handleSkinLeave` / 初期化順 / base path 互換は維持する

つまり目標は「ブラウザの再生成を避けつつ、外部 skin 切り替えを document 差し替え中心へ寄せる」である。

built-in skin と外部 skin の相互切り替えは表示基盤自体が異なるため、この方式だけでは統一できない。

### 9.4 built-in skin 側の改善方針

built-in skin で体感が残る場合は、主戦場は外部 skin host ではなくタブ切り替え側である。

優先度は次である。

1. タブ切り替え直後の画像 decode の見直し
2. 詳細ペイン更新の軽量化
3. 詳細サムネ探索の条件厳格化

必要であれば、

- 非表示タブでは重い decode を遅延する
- 詳細ペインが見えていない時は詳細サムネ探索を始めない
- 切り替え直後の先頭選択処理で不要な再描画を抑える

の順で効く可能性が高い。

### 9.5 実施順

実施順は次を推奨する。

1. 計測ログ追加
2. refresh 起点の一本化
3. stale 判定の前倒し
4. skin catalog cache 導入
5. SQLite 保存経路の軽量化
6. 外部 skin の document 差し替え中心化
7. built-in skin 側 decode と詳細更新の軽量化

### 9.6 完了判定

改善完了は、少なくとも次を満たした時点とする。

1. 外部 skin 切り替え 1 回で refresh が実質 1 回だけ動く
2. catalog 再走査が常時発生しない
3. 連続切り替え時に stale な prepare / navigate が大幅に減る
4. built-in skin と外部 skin のどちらで重いかをログで説明できる
5. 既存の skin 切り替え互換が維持される

## 10. 補足 WebView2 は別スレッド実行か

結論として、このアプリで扱っている `WebView2` は「別ワーカースレッドへ投げて実行している」のではない。

整理すると次の 2 層で考える必要がある。

1. アプリ側の `WebView2` control 操作
   - 基本は WPF の UI スレッド
2. `WebView2 Runtime` の内部実行
   - browser process / renderer process / GPU process などの別プロセス

つまり、

- C# から `EnsureCoreWebView2Async(...)`
- `HandleSkinLeaveAsync()`
- `NavigateToString(...)`
- `NavigationCompleted` 待ち

を行う経路は UI スレッド前提であり、
その先の HTML 解釈や描画実行は Runtime 側のプロセス群が担う。

### 10.1 このアプリ側の実装

現在の外部 skin refresh は `Dispatcher` で直列化されている。

- `Views/Main/ExternalSkinHostRefreshScheduler.cs:45`
  - `dispatcher.BeginInvoke(...)`
- `Views/Main/MainWindow.WebViewSkin.cs:118`
  - `RefreshExternalSkinHostPresentationAsync(...)`
- `Views/Main/MainWindow.WebViewSkin.cs:422`
  - host mount も `Dispatcher.InvokeAsync(...)`
- `WhiteBrowserSkin/Runtime/WhiteBrowserSkinRuntimeBridge.cs:42`
  - `webView.EnsureCoreWebView2Async(environment)`
- `WhiteBrowserSkin/Host/WhiteBrowserSkinHostControl.xaml.cs:187`
  - `SkinWebView.NavigateToString(...)`

したがって、少なくともこのアプリの skin 切り替え制御は「UI スレッドから WebView2 を触る」構造である。

### 10.2 何が別で動くのか

`WebView2` の内部では、Edge Runtime の process model に従って複数プロセスが動く。

- browser process
- renderer process
- GPU process
- audio などの helper process

このため、「HTML の実行エンジンまで全部が MainWindow の 1 スレッドだけで動く」わけではない。

ただし、アプリ側から見える `WebView2` control は STA の UI スレッド前提であり、callback も同じスレッド側へ返ってくる。

### 10.3 今回の重さとの関係

この点は、今回の「skin 切り替えが重い」原因を読むうえで重要である。

重い理由は「WebView2 が別スレッドだから遅い」ではなく、実際には次の組み合わせで UI 体感へ出ている。

- UI スレッド上で refresh が直列化される
- その中で skin catalog 再走査が入る
- 同期 SQLite I/O が入る
- `NavigateToString` 完了待ちまで切り替え経路に含まれる
- 連打時は stale な準備仕事が後段まで進む

つまり、Runtime が別プロセスで動いていても、アプリ側の切り替え制御が UI スレッド前提であれば、十分に「重く見える」。

### 10.4 見立て

今回の改善で狙うべきことは、「WebView2 を別スレッドへ逃がす」ことそのものではない。

本命は次である。

1. UI スレッドに載せている無駄仕事を減らす
2. stale な refresh を前段で止める
3. catalog と DB 保存の直列負荷を下げる
4. 外部 skin 同士は WebView2 実体を使い回し、document 差し替え中心へ寄せる

### 10.5 参考

公式ドキュメントでも、`WebView2` は STA の UI thread 上で作成・操作し、callback も同じ thread で発生すると説明されている。
また Runtime は browser / renderer / GPU を含む別プロセス群で構成される。

- https://learn.microsoft.com/en-us/microsoft-edge/webview2/concepts/threading-model
- https://learn.microsoft.com/en-us/microsoft-edge/webview2/concepts/process-model

## 11. 補足 DBスレッドを分ける方法と効果

結論として、`SQLite` の別スレッド化は可能であり、特に「保存系」には効果がある。

ただし、現行コードを再確認すると、skin 切り替えの重さ全体に対しては「効くが主因の全部ではない」という位置づけである。

本命は引き続き次の組み合わせである。

- 外部 skin refresh の二重 queue
- stale 判定の遅さ
- skin catalog 再走査
- WebView2 host 準備と `NavigateToString`

そのうえで、DB は UI 経路上の同期仕事として体感悪化に寄与している。

### 11.1 現行コードの再確認結果

skin 切り替え本線では、`SQLite` 呼び出しは今も同期で入っている。

- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:103`
  - `ApplySkinByName(...)` から `PersistCurrentSkinState(...)`
- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:134`
  - `UpsertSystemTable(...)`
- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:135`
  - `UpsertProfileTable(...)`
- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:218`
  - `SelectProfileValue(...)`

さらに外部 skin API 側でも、profile 読み書きが UI スレッドへ寄せられている。

- `Views/Main/MainWindow.WebViewSkin.Api.cs:507`
  - `GetExternalSkinProfileValueAsync(...)`
- `Views/Main/MainWindow.WebViewSkin.Api.cs:535`
  - `WriteExternalSkinProfileValueAsync(...)`
- `Views/Main/MainWindow.WebViewSkin.Api.cs:797`
  - `InvokeExternalSkinUiActionAsync(...)`
- `Views/Main/MainWindow.WebViewSkin.Api.cs:819`
  - `InvokeExternalSkinUiTaskAsync(...)`

つまり現状は、

- orchestrator 側の skin 状態保存
- 外部 skin API 経由の profile 読み書き

の両方で DB 呼び出しが UI スレッド寄りに存在している。

### 11.2 分けやすい理由

一方で、`SQLite` 実装自体は別スレッド化しやすい形である。

- `DB/SQLite.cs:444`
  - `CreateReadOnlyConnection(...)`
- `DB/SQLite.cs:450`
  - `CreateReadWriteConnection(...)`

各 API は `SQLiteConnection` を共有せず、毎回接続を開いて閉じる静的メソッドになっている。

したがって、「同じ connection を UI スレッドとバックグラウンドスレッドで共用して壊れる」形ではない。

この点から、呼び出し側を整理すれば次は比較的安全に行える。

- 保存要求を専用ワーカーへ積む
- 読み取りを `Task.Run` などで UI 経路から外す
- 書き込みは 1 本の直列キューに寄せる

### 11.3 一番安全な分け方

最も安全なのは、「保存だけを別スレッドへ逃がす」方式である。

対象は次である。

1. `UpsertSystemTable(...)`
2. `UpsertProfileTable(...)`

この 2 つは「表示成立後の永続化」であり、skin を見せるための即時 UI 判定には使っていない。

そのため、

- UI では skin 切り替えを先に完了させる
- 保存は直後にバックグラウンド直列キューへ積む

という分離がしやすい。

この方式なら、体感改善が見込めるうえ、表示ロジックの意味も崩しにくい。

### 11.4 難しいのは読み取り側

`SelectProfileValue(...)` は保存より扱いが難しい。

- `WhiteBrowserSkin/WhiteBrowserSkinOrchestrator.cs:218`
  - 外部 skin の前回タブ復元に使っている

これは「どのタブを最初に表示するか」を決める前段にあるため、単純に fire-and-forget へ変えると初期表示順が変わる。

読み取りも別スレッド化はできるが、その場合は次のどちらかが必要である。

1. 既定タブで先に表示し、DB 読み取り完了後に必要なら補正する
2. 非同期化しても初期表示確定まで await し、UI スレッドの待機だけ外す

したがって、`SelectProfileValue(...)` は「ただ別スレッドへ投げれば終わり」ではない。

### 11.5 外部 skin API 側の注意点

今回の再確認で、外部 skin API の profile 読み書きも UI スレッドへ寄せられていることが分かった。

- `GetExternalSkinProfileValueAsync(...)`
- `WriteExternalSkinProfileValueAsync(...)`

ここは現在、UI 状態の参照と DB 呼び出しが同じ delegate に入っている。

そのため改善するなら、次の 2 段に分けるのが良い。

1. UI スレッドで必要最小限の値だけ取る
   - `dbFullPath`
   - `skinName`
   - `key`
   - `value`
2. DB 呼び出し自体は UI スレッド外で行う

これで、UI スレッドに残す処理を「状態スナップショット取得」だけに絞れる。

### 11.6 効果の見立て

効果はある。

ただし優先度は「最初にやるべき唯一の施策」ではない。

見立ては次である。

1. 保存系を別スレッド化
   - 効果あり
   - 特に UNC、HDD、ロック競合時に効きやすい
2. 読み取り系を別スレッド化
   - 条件付きで効果あり
   - ただし初期表示の設計変更が必要になりやすい
3. skin 切り替え全体
   - DB 分離だけでは決定打になりにくい
   - catalog と refresh 構造の改善も同時に必要

つまり、DB 分離は「やる価値がある」が、「それだけで skin 切り替えが軽くなる」とまでは言いにくい。

### 11.7 推奨順

現行コード前提での推奨順は次である。

1. `PersistCurrentSkinState(...)` 内の保存だけを非同期直列キュー化する
2. 外部 skin API の profile 書き込みも UI スレッド外へ出す
3. profile 読み取りは、既定タブ fallback を許容できる設計にしてから非同期化する

この順なら、効果と安全性のバランスが良い。
