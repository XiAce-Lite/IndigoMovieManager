# 仕様書_WiXv6インストーラーと自己更新_2026-04-05

最終更新日: 2026-04-05

変更概要:
- `WiX v6 + Burn bundle` を前提に、install / upgrade / uninstall / 自己更新の最終仕様を詳細化
- `GitHub Releases API` と `update manifest` を組み合わせた更新契約を固定
- `UpdateApplyBridge`、保持項目付き uninstall、保存先、ログ、失敗時挙動を実装可能な粒度で明文化
- `VS Code 1.115 + GitHub Pull Requests and Issues` を前提にした開発運用条件を追記

## 1. 目的

- installer 実装の正本仕様を 1 枚に固定する
- `WiX v6` を使った bundle / MSI / BA / app 内更新の責務境界を明確にする
- 実装前に、ユーザー体験と削除境界を曖昧にしない

## 2. 適用範囲

本仕様書が対象とする機能:
- 初回 install
- 既存版からの upgrade
- 明示 uninstall
- uninstall 時の保持項目選択
- アプリ内 `更新ボタン`
- 起動時のバックグラウンド更新確認
- GitHub Releases API を使う最新版検知
- 最新版 bundle の自動 download
- silent update 適用
- 自動再起動

本仕様書が対象外とする機能:
- pre-release 配信
- rollback UI の高度化
- enterprise 向け GPO 配布
- multi-channel 配布
- repair / modify の専用 UI

## 3. 採用前提

- installer engine は `WiX Toolset v6`
- bundle は `Burn`
- build は `dotnet build` 前提の SDK-style project
- 開発環境の標準は `VS Code 1.115`
- GitHub 連携の標準拡張は `GitHub Pull Requests and Issues`
- 更新判定の正本は `GitHub Releases API`
- `WiX` の built-in update feed は使わない
- `Open Source Maintenance Fee` の適用条件が運用上許容できることを採用前提とする

## 4. 用語

| 用語 | 意味 |
| --- | --- |
| App | `IndigoMovieManager.exe` 本体 |
| Bundle | 公開配布する WiX Burn bundle exe |
| MSI | App 本体を install 管理する内部 package |
| BA | Bootstrapper Application |
| Custom BA | uninstall 保持選択を持つ custom managed BA |
| UpdateCheckService | GitHub Releases API を確認する app 側サービス |

| UpdateCoordinator | UI と更新サービスを束ねる app 側 coordinator |
| UpdateApplyBridge | app 終了後に silent apply と再起動を担う別プロセス |
| Update Manifest | 最新版 bundle asset を確定する JSON |
| AppIdentity | `AppIdentityRuntime.ResolveAppIdentityName()` が返す実行 identity |

## 5. 全体構成

### 5.1 公開配布物

GitHub Release に載せる公開 asset は次の 3 つを正本とする。

- `IndigoMovieManager-<version>-win-x64.zip`
- `IndigoMovieManager-Setup-<version>-win-x64.exe`
- `IndigoMovieManager-update-manifest-<version>-win-x64.json`

補足:
- 利用者向けの主入口は `Setup.exe`
- `zip` は退避路として継続
- `MSI` は公開 asset にしない

### 5.2 bundle 内部構成

Bundle は次を含む。

- App MSI
- `.NET Desktop Runtime` prerequisite 判定
- BA
- install / upgrade / uninstall / quiet apply 制御

App MSI は次を含む。

- `IndigoMovieManager.exe`
- 必要 DLL
- `rescue-worker\*`
- `tools\ffmpeg-shared\*`
- `Images\*`
- `skin\*`
- `UpdateApplyBridge.exe`
- `rescue-worker.lock.json`
- `rescue-worker-expected.json`

## 6. 表示名と install 先

### 6.1 表示名

ユーザー向け表示名はすべて `IndigoMovieManager` に統一する。

対象:
- setup 画面タイトル
- ARP 表示名
- GitHub Release の setup asset 名
- install 先フォルダ名

### 6.2 install 先

v1 の install 先は per-user を正本とする。

- `%LOCALAPPDATA%\Programs\IndigoMovieManager`

理由:
- 現状は `Thumb` と `layout.xml` に `AppContext.BaseDirectory` 前提が残る
- `Program Files` 既定は権限衝突の危険が高い

### 6.3 AppIdentity と LocalAppData

App のローカル保存先ルートは既存の `AppIdentity` を使う。

- `%LOCALAPPDATA%\{AppIdentity}\logs`
- `%LOCALAPPDATA%\{AppIdentity}\QueueDb`
- `%LOCALAPPDATA%\{AppIdentity}\FailureDb`
- `%LOCALAPPDATA%\{AppIdentity}\RescueWorkerSessions`
- `%LOCALAPPDATA%\{AppIdentity}\WebView2Cache`
- `%LOCALAPPDATA%\{AppIdentity}\Updates`

補足:
- 表示名と install 先は `IndigoMovieManager`
- ローカル保存先は既存 `AppIdentity` 規約を維持する

## 7. install 仕様

### 7.1 初回 install

Bundle は次の順で処理する。

1. Detect
2. prerequisite 判定
3. install plan
4. Apply
5. 成功時に App を起動可能状態にする

### 7.2 prerequisite

- `.NET 8 Desktop Runtime` の存在を確認する
- 未導入時は bundle 側で prerequisite 導線を出す
- quiet mode では prerequisite も含めて無人適用できることを完了条件とする

### 7.3 shortcut

v1 の既定:
- Start Menu shortcut: 作成する
- Desktop shortcut: 作成しない

### 7.4 起動

install 完了後の `起動する` は Full UI のみ表示対象とする。
quiet apply では自動再起動フローを `UpdateApplyBridge` が担う。

## 8. upgrade 仕様

### 8.1 upgrade 判定

- 同一 product line の旧版が存在する場合、major upgrade として置き換える
- downgrade は不可

### 8.2 upgrade 中の保持挙動

upgrade 時は保持項目 UI を出さない。

理由:
- 旧版 uninstall が内部処理として走るため
- version update で `Thumb` や `layout.xml` を消すのは不正

### 8.3 upgrade 後の期待結果

- App 本体は新しい版へ更新される
- `Thumb`
- `layout.xml`
- `%LOCALAPPDATA%\{AppIdentity}` 配下
- `user.config`
はそのまま残る

## 9. uninstall 仕様

### 9.1 uninstall モード

uninstall は 2 種類に分ける。

- `explicit uninstall`
  - ユーザーが ARP または setup を使って明示実行
- `upgrade uninstall`
  - 新版適用中に内部で実行

### 9.2 保持項目 UI

保持項目 UI は `explicit uninstall` のみ表示する。

表示文言:
- この操作はアプリ本体を削除します
- `.wb`、外部サムネイル、外部 bookmark は削除しません

チェック項目:
- `[x] サムネイルを残す`
- `[x] レイアウトと補助ファイルを残す`
- `[x] ローカルキャッシュとログを残す`
- `[x] ユーザー設定を残す`

既定値:
- すべて ON

### 9.3 カテゴリごとの削除対象

| 保持項目 | OFF の時に削除するもの |
| --- | --- |
| サムネイルを残す | `{InstallDir}\Thumb\**` |
| レイアウトと補助ファイルを残す | `{InstallDir}\layout.xml`, `{InstallDir}\layout.missing-*.xml`, `{InstallDir}\bookmark\**` |
| ローカルキャッシュとログを残す | `%LOCALAPPDATA%\{AppIdentity}\logs\**`, `QueueDb\**`, `FailureDb\**`, `RescueWorkerSessions\**`, `WebView2Cache\**`, `Updates\pending\**`, `Updates\applied\**`, `Updates\logs\**` |
| ユーザー設定を残す | 対象 AppIdentity に属する `user.config` 一式 |

### 9.4 絶対に削除しないもの

- `.wb`
- DB の `system.thum` が指す外部フォルダ
- WhiteBrowser 同居 DB の `thum`
- DB の `system.bookmark` が指す外部フォルダ
- 外部 player
- 外部 tool

### 9.5 実装責務

- MSI は install-managed file を通常どおり削除する
- Custom BA は保持 OFF の runtime-generated file だけを後段 cleanup する
- cleanup は uninstall 成功後に実行する

## 10. GitHub Releases API 仕様

### 10.1 対象 endpoint

最新版確認は次を使う。

- `GET /repos/{owner}/{repo}/releases/latest`

### 10.2 対象 release

- `draft=false`
- `prerelease=false`
- 最新 stable release のみ

### 10.3 request 条件

- timeout: 10 秒
- User-Agent: `IndigoMovieManager/<app-version>`
- `If-None-Match` を使う
- token は使わない

### 10.4 response 利用項目

- `tag_name`
- `published_at`
- `assets[].name`
- `assets[].browser_download_url`
- `assets[].size`
- `assets[].digest` があれば補助検証に使う

### 10.5 レート制限

- `304 Not Modified` は成功扱い
- `403` かつ rate limit header がある場合は reset 時刻まで自動再試行しない
- 起動時チェック失敗は UI を止めず、log のみ
- 手動更新時のみエラー表示する

## 11. Update Manifest 仕様

### 11.1 役割

manifest は `release API だけでは固定しづらい app 固有契約` を持つ正本である。

役割:
- installer asset 名の確定
- sha256 の正本
- 最低対応 version の通知
- schema version 固定

### 11.2 asset 名

- `IndigoMovieManager-update-manifest-<version>-win-x64.json`

### 11.3 JSON schema

```json
{
  "schemaVersion": 1,
  "appId": "IndigoMovieManager",
  "channel": "stable",
  "version": "1.0.3.5",
  "releaseTag": "v1.0.3.5",
  "publishedAtUtc": "2026-04-05T12:34:56Z",
  "minimumSupportedVersion": "1.0.0.0",
  "installer": {
    "assetName": "IndigoMovieManager-Setup-1.0.3.5-win-x64.exe",
    "sha256": "HEX_UPPERCASE",
    "size": 123456789
  },
  "package": {
    "assetName": "IndigoMovieManager-1.0.3.5-win-x64.zip"
  },
  "workerLock": {
    "insideInstaller": true,
    "lockFileName": "rescue-worker.lock.json"
  }
}
```

### 11.4 検証条件

manifest を読む時は次をすべて満たすこと。

- `schemaVersion == 1`
- `appId == IndigoMovieManager`
- `channel == stable`
- `releaseTag` が API の `tag_name` と一致
- `version` が現在 version より新しい
- `installer.assetName` が release assets に存在
- `installer.size` が release asset size と一致

## 12. 更新確認仕様

### 12.1 入口

- App 内 `更新ボタン`
- 起動時バックグラウンド確認

### 12.2 起動時確認

- MainWindow 初期表示完了後に遅延開始する
- 遅延開始時刻: UI ready から 15 秒後
- throttle: 前回確認から 24 時間未満なら skip
- ただし `ReadyToApply` 状態の download 済み更新があれば先にそれを優先する

### 12.3 手動確認

- throttle を無視して即時実行
- 進行状況と失敗理由を UI 表示する

### 12.4 状態

| 状態 | 意味 |
| --- | --- |
| Idle | 待機 |
| Checking | API 確認中 |
| UpToDate | 最新 |
| UpdateAvailable | 新版あり |
| Downloading | bundle download 中 |
| ReadyToApply | 検証済み bundle 準備完了 |
| Applying | silent apply 中 |
| Failed | 失敗 |

## 13. download 仕様

### 13.1 保存先

- `%LOCALAPPDATA%\{AppIdentity}\Updates\pending`
- `%LOCALAPPDATA%\{AppIdentity}\Updates\applied`
- `%LOCALAPPDATA%\{AppIdentity}\Updates\logs`
- `%LOCALAPPDATA%\{AppIdentity}\Updates\state.json`

### 13.2 ファイル名

download 中:
- `IndigoMovieManager-Setup-<version>-win-x64.exe.partial`
- `IndigoMovieManager-update-manifest-<version>-win-x64.json.partial`

検証完了後:
- `.partial` を外して rename

### 13.3 検証

順序:

1. manifest を download
2. manifest schema を検証
3. installer asset を download
4. file size を検証
5. SHA-256 を検証
6. GitHub asset `digest` があれば一致確認
7. 一致後に `ReadyToApply`

### 13.4 再利用

- 同 version の検証済み bundle があれば再 download しない
- hash 不一致なら破棄して取り直す

## 14. UI 仕様

### 14.1 更新ボタン

表示:
- `更新を確認`

押下時:
- `Checking` へ遷移
- 新版無し: `最新です`
- 新版あり未 download: download 開始
- 新版あり download 済み: `更新しますか？` を表示

### 14.2 起動時通知

起動時は silent を基本とする。

通知する条件:
- 新版が download 済みで `ReadyToApply`

通知文言:
- `新しいバージョン {version} をダウンロード済みです。今すぐ更新しますか？`

### 14.3 手動確認失敗

手動確認時の失敗文言:
- `更新確認に失敗しました。ネットワーク状態または GitHub の接続制限を確認してください。`

### 14.4 起動時確認失敗

- モーダルは出さない
- `%LOCALAPPDATA%\{AppIdentity}\logs` へ記録する

## 15. UpdateApplyBridge 仕様

### 15.1 役割

App 終了後に次を行う。

1. 親 App の終了待ち
2. bundle exe の quiet apply 実行
3. 成功時の App 再起動

### 15.2 起動方法

App は bridge を次の形で起動する。

```text
UpdateApplyBridge.exe --request-json "<path>"
```

### 15.3 request JSON schema

```json
{
  "schemaVersion": 1,
  "parentPid": 12345,
  "targetVersion": "1.0.3.5",
  "bundlePath": "C:\\Users\\...\\IndigoMovieManager-Setup-1.0.3.5-win-x64.exe",
  "manifestPath": "C:\\Users\\...\\IndigoMovieManager-update-manifest-1.0.3.5-win-x64.json",
  "mainExePath": "C:\\Users\\...\\IndigoMovieManager.exe",
  "restartArguments": [],
  "logDirectoryPath": "C:\\Users\\...\\Updates\\logs"
}
```

### 15.4 bridge 動作

1. 引数と request JSON を検証
2. 親 PID の終了を最大 120 秒待つ
3. bundle log path を決定
4. bundle exe を quiet mode で起動
5. 完了コードを待つ
6. 成功時に `mainExePath` を起動
7. request JSON を `applied` へ移動

### 15.5 bridge exit code

| code | 意味 |
| --- | --- |
| 0 | 成功 |
| 10 | 引数不正 |
| 11 | request JSON 不正 |
| 12 | 親終了待ち timeout |
| 13 | bundle file 不在 |
| 14 | manifest 検証失敗 |
| 15 | bundle apply 失敗 |
| 16 | 再起動失敗 |

## 16. quiet apply 仕様

- bridge は bundle を quiet mode で起動する
- bundle 側は再起動要求を自前では行わず、まず App 再起動を優先する
- quiet apply の正確な command line は WiX PoC で確定し、本仕様書の appendix に追記する
- ただし要件としては `ユーザーに追加質問を出さず上書き更新できること` を満たさなければならない

## 17. ログ仕様

### 17.1 App 側

既存 log へ `app-update` カテゴリで出す。

出力例:
- update check start
- update check skipped by throttle
- latest release found
- manifest validation failed
- installer download completed
- ready to apply
- bridge launched

### 17.2 bridge 側

保存先:
- `%LOCALAPPDATA%\{AppIdentity}\Updates\logs`

ファイル名:
- `update-apply-<timestamp>.log`

### 17.3 bundle 側

保存先:
- `%LOCALAPPDATA%\{AppIdentity}\Updates\logs`

ファイル名:
- `bundle-<version>-<timestamp>.log`

## 18. セキュリティ仕様

- update 対象は stable release のみ
- manifest は `schemaVersion` を強制
- installer は `sha256` 検証必須
- GitHub asset `digest` は補助検証に使う
- digest / sha256 不一致時は apply 不可
- 起動時 check 失敗では自動 apply へ進まない
- App 本体は download 完了後も bridge 起動前に最終確認ダイアログを必ず出す

## 20. 受け入れ条件

- 初回 install が通る
- upgrade が通る
- explicit uninstall で保持項目 UI が出る
- `サムネイルを残す` OFF で `{InstallDir}\Thumb` だけが削除される

- 失敗時に旧版の起動性が失われない

## 22. この仕様の結論

- installer / update の最終仕様は `WiX v6 + Burn bundle + GitHub Releases API + UpdateApplyBridge` で固定する
- `更新判定` と `更新適用` を分離し、前者は GitHub API、後者は bundle へ責務を寄せる
- uninstall は `保持 ON が既定` で、安全側に倒す
- `Thumb`、`layout.xml`、`LocalAppData`、`user.config` の扱いを category 単位で固定し、外部データへは触らない
