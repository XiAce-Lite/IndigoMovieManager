## 2. 新 repo に置く project 一覧

### 2.1 必須

1. `IndigoMovieManager.Thumbnail.Contracts`
   - worker と host が共有する DTO、artifact contract、path policy、result json の契約を置く。
2. `IndigoMovieManager.Thumbnail.Engine`
   - サムネ生成本体、decoder 群、engine 群を置く。
3. `IndigoMovieManager.Thumbnail.FailureDb`
   - rescue 用の状態保存、lease、heartbeat、failure record を置く。
4. `IndigoMovieManager.Thumbnail.RescueWorker`
   - 実行可能 host、CLI、orchestration、publish entry を置く。
5. `IndigoMovieManager.Thumbnail.Tests`
   - compatibilityVersion、artifact marker、FailureDb の最小回帰を置く。

### 2.2 置かない

- `IndigoMovieManager.Thumbnail.Queue`
- `IndigoMovieManager.Thumbnail.Runtime`
- WPF 本体
- app release package 用の host 固有コード

### 2.3 最小成立の考え方

- worker repo は `Queue` 全体を持たない
- worker repo は `Contracts / Engine / FailureDb / RescueWorker / Tests` だけで build と publish が回る
- `Engine` は repo 外のリンクコンパイルに戻さない

## 3. main repo に残す責務

### 3.1 host と UI

- WPF 本体
- `ThumbnailRescueWorkerLauncher`
- `ThumbnailRescueWorkerLaunchSettingsFactory`
- `MainWindow.ThumbnailRescueWorkerLauncher`
- `Runtime` の host 基盤

### 3.2 通常運用

- 通常 `QueueDb`
- MainDB 連携
- UI 進捗反映
- watcher / tab / list の体感テンポ維持

### 3.3 配布と release

- `scripts/create_github_release_package.ps1`
- `.github/workflows/github-release-package.yml`
- `scripts/invoke_release.ps1`
- app release 手順書

### 3.4 main repo の責務境界

- main repo は worker ソースを持たず、worker artifact の消費側になる
- main repo は worker の `compatibilityVersion` を見て、使う artifact を選ぶ
- worker の publish 実体は external repo 側の責務とする

## 4. 最小 GitHub Actions フロー

### 4.1 worker repo 側

#### PR / push の検証

- `windows-latest` で実行する
- `checkout`
- `setup-dotnet`
- `dotnet restore`
- `dotnet build`
- `dotnet test`
- artifact の zip 作成はしない

#### tag push または手動実行

- `workflow_dispatch` と `push tags v*` を入口にする
- `Publish-RescueWorkerArtifact.ps1` を呼ぶ
- `create_rescue_worker_artifact_package.ps1` で zip と `rescue-worker-artifact.json` を作る
- `actions/upload-artifact` へ成果物を残す
- tag 実行時だけ GitHub Release asset にも添付する

#### 最小 job 構成

1. checkout
2. setup-dotnet
3. build/test
4. publish
5. package
6. upload artifact
7. tag 時のみ release 添付

### 4.2 main repo 側

- app release workflow は app ZIP を主役にする
- worker は external repo の release asset か固定 artifact を参照する
- `github-release-package.yml` は worker 配布の正本に戻さない
- main repo は lock file で `package version`、`worker artifact version`、`compatibilityVersion`、`sha256` を pin する

### 4.3 最小フローの狙い

- worker 単体の健全性確認を外部 repo で閉じる
- main repo は UI と配布導線に集中する
- 配布責務が 2 か所に散らばる事故を避ける

## 5. リスク / main repo 側で固定する運用条件

### 5.1 リスク

- `compatibilityVersion` と package version の対応を間違えると、main repo が古い worker を拾う
- Release asset と Actions artifact のどちらを正本にするか曖昧だと、運用導線が分岐する
- external repo 化後に `Engine` 側のテストが main repo 依存を残すと、CI が壊れやすい
- worker host の名称を変えるなら、artifact 名・doc 名・workflow 名の再同期が必要になる

### 5.2 main repo 側で固定すること

- worker 実行物の正本は GitHub Release asset に固定する
- Actions artifact は CI 確認用だけに使い、main repo の pin 先にしない
- `Contracts / Engine / FailureDb` は package として配り、worker exe は zip artifact として分ける
- `compatibilityVersion` は `CLI / result json / FailureDb schema / worker artifact 配置規約` の破壊的変更で bump する
- main repo と external repo の同時変更は、external repo の preview 発行を先に行い、main repo は lock file を更新して追従する
- launcher の fail-fast 条件と live 確認の合格条件は `TASK-008 main repo残置責務とexternal worker運用 2026-04-03` で正本化する

## 6. 結論

外部 repo の最小セットは、`Contracts / Engine / FailureDb / RescueWorker / Tests` の 5 本で成立する。
main repo は `UI / launcher / Runtime / app release / worker artifact 消費` を残す。
CI は、worker repo 側で build/test/publish/package を閉じ、main repo 側は app release に集中させるのが一番小さい。

### 6.1 いまの実務判断

- 外部 repo に先に出すのは `Contracts / Engine / FailureDb / RescueWorker / Tests` だけでよい
- `Queue` 全体、`Runtime`、WPF、通常 UI は main repo に残す
- worker repo の CI は `PR/push では build + test`、`tag か manual では publish + package + upload` に分ける
- `compatibilityVersion` は artifact 契約が壊れる変更だけで bump する
- main repo は worker の source ではなく、`package + release asset` 消費側として固定する
- worker 実行物の正本は GitHub Release asset、Actions artifact は確認用に限定する
