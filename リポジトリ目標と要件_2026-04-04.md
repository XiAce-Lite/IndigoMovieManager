# IndigoMovieManager リポジトリ目標と要件

最終更新日: 2026-04-04

## 1. この文書の目的

この文書は、IndigoMovieManager リポジトリが何を目指し、何を守り、何を今はやらないかを最初に揃えるための正本である。

- 利用者向けの README だけでは伝わり切らない開発判断の軸を明示する
- AI / 人間の開発者が同じ前提で判断できるようにする
- ブランチ方針、全体計画、配布運用の入口を一本化する

## 2. リポジトリ情報

- リポジトリ名: `IndigoMovieManager`
- 現在の開発本線ブランチ: `workthree`
- ソリューション: `IndigoMovieManager.sln`
- エントリープロジェクト: `IndigoMovieManager.csproj`
- アプリ種別: `WinExe` の WPF デスクトップアプリ
- 対象フレームワーク: `net8.0-windows10.0.19041.0`
- ビルド対象: `x64`
- 正本データ: WhiteBrowser 互換の `*.wb`
- 補助 DB: `*.queue.imm` と `*.failure.imm`
- 主要サブプロジェクト:
  - `src/IndigoMovieManager.Thumbnail.Contracts`
  - `src/IndigoMovieManager.Thumbnail.Engine`
  - `src/IndigoMovieManager.Thumbnail.Queue`
  - `src/IndigoMovieManager.Thumbnail.FailureDb`
  - `src/IndigoMovieManager.Thumbnail.Runtime`
  - `src/IndigoMovieManager.FileIndex.UsnMft`
- release 導線:
  - GitHub Actions の `github-release-package.yml` で package を作成する
  - 配布物には bundled rescue worker を同梱する
  - worker の pin 情報は release 本文と package 内 summary で追跡する
- remote の役割:
  - `origin`: 配布用の開発 remote
  - `upstream`: 参照元 remote
  - `private-probe`: private release 検証用 remote

## 3. このリポジトリの目標

IndigoMovieManager は、WhiteBrowser 互換を重視しながら、動画一覧の管理、監視フォルダ取り込み、検索、サムネイル生成、救済導線を一つにまとめた Windows デスクトップアプリを育てるリポジトリである。

このリポジトリで最優先にすることは次の 4 点である。

1. ユーザーが感じるテンポ感を守り、改善する
2. WhiteBrowser からの移行や併用を壊さない
3. サムネイル生成失敗を観測でき、救済できる運用を維持する
4. 配布物を安定して作れ、検証できる状態を保つ

## 4. 価値として届けるもの

- 既存の `*.wb` を活かしながら動画管理を続けられること
- フォルダ監視やドラッグ＆ドロップから自然にライブラリへ取り込めること
- サムネイルが通常経路で速く作られ、失敗時は rescue worker で救済できること
- NAS / UNC を含む実運用環境で破綻しにくいこと
- release package に必要な worker が同梱され、配布後の診断ができること

## 5. 必須要件

## 5.1 機能要件

- WhiteBrowser 互換 DB `*.wb` を正本として扱う
- 既存の `*.wb` は開けるが、元ファイルを直接壊さない運用を前提にする
- 動画フォルダまたは `*.wb` のドラッグ＆ドロップで開始できる
- 監視フォルダからの取り込みと一覧反映ができる
- 動画の検索、一覧表示、選択、再生導線を提供する
- サムネイル生成が通常経路と rescue 経路の両方で動く
- release package に bundled rescue worker を含める

## 5.2 非機能要件

- ユーザー体感テンポを最優先にする
- UI スレッドへ重い処理を戻さない
- ログだけで遅さや失敗理由を追える観測性を維持する
- 通常動画の初動を、難読動画対策のために悪化させない
- `Factory + Interface + Args` の入口整理を崩さない
- ビルドとテストは `x64` 前提で維持する
- 文字コードと改行は `UTF-8 (BOMなし) + LF` に統一する

## 5.3 配布要件

- 最初の配布は private repository ベースでも成立すること
- GitHub Actions から release package を再現可能に作れること
- release 本文に bundled rescue worker の pin 情報を残せること
- 配布物だけで worker の version / compatibilityVersion / sha256 を追えること

## 6. 守るべき制約

- WhiteBrowser の DB 形式 `*.wb` は変更しない
- WhiteBrowser 互換を壊す変更は、明示判断なしに入れない
- 過去の実験枝の成果は、本線のテンポを壊さない形に一般化できたものだけを入れる
- 正しさだけを理由に UI テンポ悪化を受け入れない
- 観測性を削って高速化したことにしない

## 7. 今はやらないこと

- WhiteBrowser 非互換を前提にした大規模仕様変更
- rescue / worker 都合で公開契約を無秩序に広げること
- 巨大リファクタを一発で入れること
- 実 release 前提の公開 repo 運用に固定すること
- 個別動画名依存の特殊分岐を本線へ持ち込むこと

## 8. 現在の大粒度優先順位

1. サムネイル生成入口整理の維持
2. 救済レーン実動画検証
3. Queue 観測の最小補強
4. UI テンポ改善
5. 難読動画条件の棚卸し
6. release / worker 配布導線の安定化

## 9. 判断に迷った時の基準

次の順に優先して判断する。

1. ユーザー体感テンポを守れているか
2. WhiteBrowser 互換と `*.wb` 正本運用を守れているか
3. ログと診断だけで理由を追えるか
4. 通常経路を重くせずに改善できているか
5. 変更の責務が広がり過ぎていないか
